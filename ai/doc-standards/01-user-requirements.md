# 01 User Requirements Standard（用户需求规范镜像）

> Sync notice: This file is maintained by `ai-project-template` and may be overwritten when a derived project syncs template methodology.
> Do not edit it directly in derived projects; propose reusable changes in `_proposals/` and upstream them to the template repository.

本文件是 `docs/01-user-requirements.md` 的细粒度规范标准，用于 AI 生成、精修、审计和评估用户需求文档。它不是项目事实文档，派生项目的实际用户需求仍写入 `docs/01-user-requirements.md`。

## 1. 定位与边界

`01` 承接 `00` 场景和输入材料，从用户视角描述需求、操作流、可观察验收口径和初步优先级。它回答“用户想完成什么、怎样算对用户有用”，不写系统内部实现、接口字段、数据库设计或最终 Phase 决策。

- 不得新增 `00` 或输入材料没有来源的用户需求；无法确认的内容标为待人工确认。
- 初步优先级和阶段建议不是最终路线图，最终 Phase 以 `03` 为准。
- 用户验收口径必须是用户可观察结果，不写代码级测试步骤。
- 每个进入 `02` 的 U-ID 必须能回到 SC-ID 或其他上游来源。

## 2. 最低结构

| 能力 | 最低字段 / 结构 |
|---|---|
| 文档元信息 | 输入来源、覆盖场景、当前状态、最后更新 |
| 用户需求总览 | U-ID、用户 / 角色、用户需求、来源场景、价值、初步优先级、状态 |
| 用户操作流 | Flow ID、参与角色、前置条件、步骤、成功路径、失败 / 中断路径 |
| 用户验收口径 | AC-ID、U-ID、用户可观察验收口径、不满足示例、备注 |
| 优先级与阶段建议 | U-ID、优先级、建议阶段、理由、待确认项 |
| 非目标与排除需求 | 编号、内容、处理方式、原因 / 来源 |
| 追溯矩阵 | U-ID、来源场景 / 输入、来源锚点、下游 REQ、状态 |
| 待人工确认项 | 结构化确认项表，不得只写问题列表 |

## 3. ID 与追溯规则

- 用户需求使用 `U-001`，用户验收口径使用 `AC-001`，用户操作流可使用 `UFlow-001`。
- 最低链路：`SC-ID / 输入来源 → U-ID → AC-ID → REQ-ID`。
- 每个 U-ID 至少有一个用户可观察验收口径；当前 Phase 的 U-ID 必须能导向 `09` 的 TC 或验收提示。
- 排除需求必须能传播到 `03` 非目标或 Phase 禁止事项。

## 4. 审计检查

| 检查项 | 通过标准 | 常见问题 |
|---|---|---|
| 来源闭合 | 每个 U-ID 有 SC-ID 或输入来源 | 用户需求悬空 |
| 可观察验收 | 每个关键 U-ID 有 AC-ID | 验收口径写成技术实现 |
| 操作流完整 | 成功、失败 / 中断路径可理解 | 只有功能名没有流程 |
| 阶段不越界 | 阶段建议标待确认，最终归属看 `03` | 01 直接决定 Phase |
| 非目标传播 | 排除需求进入 `03` 或待确认项 | 排除需求在后续被重新实现 |

## 5. 下游影响

- 给 `02`：系统需求、NFR 和异常场景来源。
- 给 `03`：功能聚合、优先级取舍和 Phase 路线图输入。
- 给 `09`：用户验收口径和验收测试目标。
- 给审计 / 评估：需求链健康度的 `U-ID` 节点。
