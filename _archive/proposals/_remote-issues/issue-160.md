# TEMPLATE-UPGRADE: 项目演示 SOP 与 AI 触发规则

> 来源：zhiyan-digital-cs-platform（emily8421/zhiyan-digital-cs-platform）派生项目回流
> 类型：模板优化提案草稿
> 状态：待提交到模板仓 issue
> 关联：`ai/commands/README.md`、`ai/document-lifecycle-rules.md`、`ai/project-rules.md`、`docs/README.md`、`template-docs/`

## 1. 背景与问题

派生项目在完成可运行 Demo、MVP 或产品化阶段后，用户经常会问：

```text
我想看演示效果
帮我启动本机 Demo
给我二维码
检查 Demo 是否起来
怎么看当前项目效果
```

这些表达不是开发任务，也不是单纯文档问答，而是"把当前项目交付物跑起来并引导用户查看效果"的场景。不同派生项目的演示内容、技术栈、端口、入口、二维码、公网地址、检查命令都不同，但背后的方法论高度一致：

1. 需要一个固定的项目级演示 SOP，避免每次靠 AI 重新猜启动方式。
2. 需要把自然语言触发词和执行边界写清楚，让 AI 知道何时只输出说明、何时可以启动服务、何时必须等待用户确认。
3. 需要记录访问入口、演示路径、健康检查、失败排查、安全边界和不可做事项。
4. 需要区分"本机 Mock Demo / MVP 试点 / 生产环境"，避免把演示效果误读成真实生产能力。
5. 需要支持项目特有能力，例如二维码、局域网访问、公网预览、桌面端、CLI 输出或移动端扫码。

如果模板没有统一方法，派生项目容易出现以下问题：

- `README.md` 里只有零散启动命令，没有固定"怎么看效果"的用户路径。
- AI 每次都重新搜索端口和脚本，容易遗漏前端、后端、Mock、检查命令或安全边界。
- 用户说"我想看演示效果"时，AI 不知道该走只读说明、启动脚本、健康检查还是验收流程。
- Demo 入口和二维码等运行产物没有忽略规则，容易被误提交。
- 演示成功后没有固定证据回写点，后续 Sprint / Phase 收尾难以复用。

## 2. 建议改动

### 2.1 增加通用场景：查看演示效果

建议在 `ai/commands/README.md` 或场景引导中增加一个通用意图，名称可为：

```text
show-demo / demo-runbook / view-demo
```

自然语言触发示例：

- "我想看演示效果"
- "帮我启动 Demo"
- "怎么看当前项目效果"
- "给我演示入口 / 二维码"
- "检查服务是否起来"
- "手机扫码看一下"

该通用意图不应固定具体脚本或端口，而是路由到当前项目的演示 SOP。若项目尚无演示 SOP，则引导 AI 生成项目专属 SOP 草案并等待用户确认。

### 2.2 约定项目级演示 SOP 位置

建议在模板中约定默认路径：

```text
docs/env/local-demo-runbook.md
```

或更通用地允许：

```text
docs/env/<demo-runbook>.md
```

推荐放入 `docs/env/`，因为演示效果通常涉及本机环境、端口、运行依赖、服务器预案、局域网 / 公网访问、二维码和资源约束。

### 2.3 提供演示 SOP 模板结构

建议在 `template-docs/` 新增演示 SOP 模板，例如：

```text
template-docs/demo-runbook-template.md
```

建议结构：

```markdown
# <项目> 演示效果手册

> 定位：本文件是当前项目的演示 SOP，说明如何启动、检查和查看效果，不替代正式验收记录。

## 1. 当前适用范围
- 适用阶段
- 交付物形态
- 验收依据
- Mock / 降级 / 真实能力边界

## 2. AI 场景：查看演示效果
- 触发词
- AI 执行流程
- AI 输出模板
- 需用户确认的动作
- 禁止事项

## 3. 启动前提
- 运行目录
- 必备工具
- 依赖安装状态
- 本机 / 服务器 / 容器 / 外部服务边界

## 4. 启动方式
- 一键启动
- 手动启动
- 参数说明
- 关闭方式

## 5. 访问入口
- 本机 URL
- 局域网 / 公网 URL
- 二维码 / 深链 / CLI 输出
- API / 管理后台 / 移动端入口

## 6. 检查与验证
- 健康检查命令
- 自动检查脚本
- 期望结果
- 常见失败与排查

## 7. 推荐演示路径
- 用户视角路径
- 管理端 / 运营端路径
- Mock / 降级提示
- 结果观察点

## 8. 安全与边界
- 不处理真实敏感数据
- 不启用未确认外部服务
- 不安装新依赖 / 不产生费用
- 本地运行产物忽略规则
```

### 2.4 增加 AI 执行边界

建议在命令或模板说明中明确：

| 用户意图 | AI 默认行为 |
|---|---|
| "怎么看演示效果" | 只读输出入口和步骤，不启动服务 |
| "帮我启动 Demo" | 可运行已存在的启动脚本；运行前说明会打开服务 / 窗口 |
| "检查 Demo" | 可运行只读健康检查脚本 |
| "给我二维码" | 可生成本地运行产物；运行产物应加入 `.gitignore` |
| "安装依赖 / 接外部服务 / 部署公网" | 必须单独确认，不能隐式执行 |

禁止项：

- 不得为了演示擅自安装依赖、启动付费服务或接真实外部系统。
- 不得把 Mock / Demo 说成生产可用。
- 不得把本地二维码、临时日志、端口状态等运行产物提交进仓库。
- 不得绕过项目 `docs/03`、`docs/05`、`docs/09` 的阶段边界和验证口径。

### 2.5 与正式验收的关系

演示 SOP 只回答"如何看效果"，不替代正式验收。建议规则：

- 演示检查通过后，可作为 Sprint 总结的辅助证据。
- 是否验收通过仍以 `docs/09-verification.md` 或项目约定的验证记录为准。
- 若演示路径与正式文档不一致，应先修正文档或脚本，不得直接把演示事实写成需求事实。

## 3. 版本影响

建议作为模板小版本增强，适用于已有 Demo / MVP / 可运行交付物的派生项目。

可能影响文件：

- `ai/commands/README.md`：新增 `show-demo` / `demo-runbook` 意图路由。
- `ai/commands/show-demo.md`：可选新增命令文件，负责路由到项目演示 SOP。
- `template-docs/demo-runbook-template.md`：新增项目级演示 SOP 模板。
- `ai/document-lifecycle-rules.md`：可补充 `docs/env/*demo-runbook*.md` 的定位说明。
- `docs/README.md` 模板说明：可补充 `docs/env/` 适合放演示 SOP、启动检查和服务器 / 局域网入口说明。

## 4. 影响面

### 4.1 对派生项目的收益

- 用户可用自然语言稳定触发"查看演示效果"。
- AI 不再每次重新搜索启动命令，减少误操作。
- 启动、检查、二维码、失败排查和安全边界集中在项目文档中。
- 支持不同技术栈项目生成各自演示 SOP，不强制统一脚本。

### 4.2 对模板的约束

- 模板只规定结构与方法，不规定具体端口、命令、二维码实现、技术栈或演示内容。
- 没有可运行交付物的项目不强制生成演示 SOP；可在首次完成 Demo / MVP 时生成。
- 对 Lean / CLI / Web / 移动端 / 桌面端项目应允许裁剪。

## 5. 验证方式

模板侧可用以下方式验证：

1. 在一个 Web Demo 派生项目中生成 `docs/env/local-demo-runbook.md`，确认 AI 能根据"我想看演示效果"路由到该文件。
2. 在一个 CLI / 小工具派生项目中生成裁剪版演示 SOP，确认不强制前端、端口或二维码。
3. 检查 AI 输出是否区分：只读说明、启动服务、健康检查、依赖安装、外部服务接入。
4. 检查运行产物是否建议进入 `.gitignore`。
5. 检查演示 SOP 是否明确不替代 `docs/09-verification.md`。

## 6. 待确认项

| ID | 待确认项 | AI 建议 | 建议依据 | 备选方案 | 取舍影响 / 阻塞关系 |
|---|---|---|---|---|---|
| DEMO-C-001 | 通用意图命名采用 `show-demo` 还是 `demo-runbook`？ | 推荐 `show-demo` 作为命令名，`demo-runbook` 作为文档 / 模板名。 | 用户表达通常是"看演示效果"，命令名应贴近动作；runbook 更适合作为文档产物。 | 只用 `demo-runbook`；或并入 `scenario` 元命令不单独建命令。 | 影响命令路由清晰度；不阻塞提案提交。 |
| DEMO-C-002 | 是否新增独立 `ai/commands/show-demo.md`？ | 推荐新增轻量命令文件，路由到项目 SOP；不要复制大段模板。 | `ai/commands` 已用于自然语言意图路由，新增文件便于多工具一致触发。 | 仅在 `ai/commands/README.md` 增加一行；或由 `scenario` 统一路由。 | 新增命令文件增加维护面，但可降低派生项目触发不一致。 |
| DEMO-C-003 | 演示 SOP 默认路径是否固定为 `docs/env/local-demo-runbook.md`？ | 推荐固定默认路径，同时允许项目按入口命名。 | `docs/env/` 已承载本机环境、服务器预案和运行约束；固定默认便于 AI 查找。 | 放入 `README.md`；放入 `docs/research/`；或项目自行决定。 | 固定路径提升可发现性；过度固定可能不适合多入口项目，因此需允许覆盖。 |

## 7. 非目标

- 不要求所有项目都必须有演示 SOP。
- 不规定具体技术栈、端口、脚本语言或二维码实现。
- 不把演示检查替代正式验收记录。
- 不鼓励 AI 自动安装依赖、启动外部服务或部署公网环境。
