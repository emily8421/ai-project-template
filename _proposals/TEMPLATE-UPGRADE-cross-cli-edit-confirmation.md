# TEMPLATE-UPGRADE: 跨 AI CLI 修改前确认机制

> 来源：模板维护者。
> 类型：模板仓库内直接发起的模板优化提案。
> 状态：落地中；当前分支 `change/cross-cli-edit-confirmation` 正在强化项目级写入确认协议、AI CLI 配置建议与 Git 审计兜底。
> 覆盖问题：不同 AI CLI 的文件修改确认行为不一致，需强化项目级确认期望与配置指引。

## 1. 背景与问题

不同 AI CLI / IDE 插件对文件修改的默认确认机制不一致。有的工具在每个文件修改前都会交互确认，有的工具在当前权限配置下可能直接修改工作区文件。

当前模板在 `ai/project-rules.md` §6 已要求：AI 在文件新增、修改、删除、重命名、格式化、批量替换前，必须先说明计划、影响范围与预期变更文件，并等待用户明确确认后再执行。

但项目文档属于“软约束”：它可以约束 AI 行为，但不能从底层强制所有 CLI 的工具调用权限。若用户的 CLI 配置允许直接写文件，仍存在 AI 未逐文件询问就修改的风险。

## 2. 设计目标

- 在模板层明确“本项目期望”的修改前确认协议。
- 给不同 AI CLI 提供配置指引入口，帮助用户开启更严格的确认 / 审批模式。
- 不承诺模板能绕过 CLI 自身权限模型实现硬拦截。
- 用 Git 状态与 diff 审计作为兜底，降低意外修改风险。

## 3. 建议方案

### 3.1 强化项目级行为规则

在入口规则中更明确地要求：

- 每次写入前必须列出：目的、预计文件、预计变更摘要、风险、验证方式。
- 用户确认前不得调用任何会写入文件、安装依赖、生成构建产物或提交代码的命令。
- 如果工具一次 patch 涉及多个文件，必须先列出全部文件与变更摘要。
- 若用户明确授权“直接修改 / 不必确认”，仅对当前任务生效。

### 3.2 增加 CLI 配置指引

在 `template-docs/ai-cli-setup.md` 增加“严格确认模式”小节：

- Claude CLI / Claude Code：提示用户查看官方权限 / approval / allowed tools 配置，建议启用写入前确认或禁用自动写入。
- Codex CLI：提示用户查看本机 Codex 配置，优先使用需要确认的 approval / sandbox 模式。
- Cursor / IDE 插件：提示启用 apply 前预览 diff 或手动确认。

具体命令和配置项容易随工具版本变化，模板不应硬编码过多易过期细节；可给出原则和检查点。

### 3.3 增加审计兜底

建议在开发任务和场景引导中要求：

- 修改前读 `git status --short --branch`。
- 修改后输出 `git status` 与文件清单。
- 必要时让用户审阅 `git diff`。
- 多步任务更新 `.ai/session-handoff.md`，记录已修改文件和待确认项。

## 4. 拟改范围

- 修改：`ai/project-rules.md` §6 模板骨架措辞
- 修改：`ai/global-rules.md` 或新增实现生命周期规则中的写入确认要求
- 修改：`template-docs/ai-cli-setup.md`
- 修改：`template-docs/scenario-guides.md` §3 / §4 / M1 相关确认说明
- 修改：`ai/prompts/dev/02-run-task.md`
- 修改：`ai/prompts/dev/05-fix-bug.md`
- 修改：`ai/prompts/docs/04-edit-single-doc.md`
- 修改：`scripts/check-template.sh`，增加关键确认文案存在性断言（如适用）

## 5. 版本影响

建议作为 patch 或 minor 版本落地。

如果仅强化文案与 setup 指引，可为 patch；若新增统一确认协议并影响多个 prompt / scenario，则建议 minor。

## 6. 验收口径

- 新项目的 `ai/project-rules.md` 明确写入前确认协议。
- `template-docs/ai-cli-setup.md` 有跨 CLI 严格确认配置指引。
- A10 / A11 / 文档编辑等写入场景均提示先列变更计划并等待确认。
- 最终说明不承诺“模板可硬性强制所有 CLI”，而是说明项目规则、工具配置与 Git 审计三层防线。

## 7. 风险与缓解

- **CLI 配置细节过期**：只写原则和入口，避免复制大量官方文档内容。
- **用户误以为绝对强制**：明确这是项目期望与配置建议，硬拦截取决于 CLI 权限设置。
- **确认过多影响效率**：允许用户在单次任务中明确授权批量修改，但授权范围必须清楚且仅当次有效。
