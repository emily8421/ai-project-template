## 1. 背景与问题

`04-architecture` 与 `05-tech-spec` 承接 `03-prd` 的 Phase、REQ、NFR 和约束，是后续 DB / API、详细设计、Sprint 计划和验证计划的设计基线。现有模板已经具备架构目标、组件视图、模块划分、技术栈、依赖配置和风险计划等基础骨架，但从派生项目回流 issue 看，`04-05` 仍存在以下通用缺口：

1. `04` 容易用一张总图同时表达系统上下文、组件关系和部署拓扑，缺少分层视图和边界说明。
2. `04` 对 Phase 约束、运行环境、项目形态裁剪、禁止项和外部信任边界的集中声明不足。
3. `04` 模块、组件和流程 ID 不够稳定，导致 `05-09`、`docs/design/*` 和任务拆分难以引用。
4. `04` 关键流程常只写成功路径，失败、降级、权限拒绝和外部服务不可用路径没有稳定传递到技术方案和验证计划。
5. 架构决策有结论，但缺少 ADR 式备选方案、取舍影响、验证方式和证据引用。
6. `05` 容易列出技术栈和版本，但依赖文件、配置文件、环境变量、密钥 / 凭据管理方式和启用阶段不够结构化。
7. 技术环境评估、资源、Docker、数据库、外部服务、LLM 等风险在 `05`、`09`、README、Phase 升级报告中分散记录，缺少 `Risk-ID → 验证方式 → TC / Sprint → 解锁条件` 闭环。
8. Mock、降级、默认关闭、候选、已验证、已启用等技术状态容易混用，导致后续 Sprint 误把候选能力当成可直接实现能力。
9. 安全、隐私、合规、真实数据、外部 AI、日志和数据留存缺少独立审计结构。
10. Phase 升级前缺少 readiness gate，真实外部服务、数据库、LLM、部署能力可能在验证不足时被误解锁。

这些问题不依赖具体业务项目。凡是包含真实运行依赖、数据库、前端、外部服务、AI / OCR / 模型或容器化部署的派生项目，都需要更细的 `04-05` 模板规范和风险验证口径。

## 2. 设计目标

Batch 3 目标是把 #106 的“04-05 架构与技术方案文档规范细化”和 #113 的“架构与技术方案风险验证规范”合并为一组可实施的模板优化，确保模板能稳定支撑以下链路：

```text
03 Phase / REQ / NFR / 约束
  → 04 架构目标 / 上下文 / 组件 / 模块 / 流程 / 部署 / ADR
  → 05 技术栈 / 依赖配置 / 运行资源 / 安全隐私 / 技术风险
  → 06 DB / 07 API / docs/design/* / 08 Sprint / 09 验证
```

具体目标：

1. 架构目标、约束、上下文、组件、模块、流程和部署拓扑可分层阅读。
2. 每个关键模块、组件和流程能追溯到 REQ、NFR、Phase 和详细设计文档。
3. 架构取舍有 ADR / 决策表记录，且包含状态、备选方案、取舍影响和验证证据。
4. 技术栈有状态、用途、约束来源、启用阶段和版本漂移处理口径。
5. 依赖、配置、环境变量、密钥管理和运行资源有独立矩阵。
6. 技术环境评估结论能回填或引用到 `05`，并影响 `09` 验证计划。
7. 安全、隐私、合规、真实数据、外部服务、日志和数据留存边界可审计。
8. 技术风险、Mock / 降级和资源瓶颈能映射到验证用例、Spike、PoC、Sprint 或 Phase 解锁条件。
9. Phase 升级前可通过 readiness gate 判断真实依赖是否允许进入实现或试点。

## 4. 建议新增或细化的 04-05 规范

### 4.1 `04-architecture` 最低要求

建议在文档标准中明确 `04` 不只是一张总架构图，至少应包含：

| 能力 | 最低字段 / 结构 |
|---|---|
| 文档元信息 | 输入来源、覆盖功能 / REQ、当前状态、最后更新 |
| 架构目标与约束 | 当前 Phase、交付物形态、运行环境、项目形态裁剪、禁止项、权威源、影响 |
| 系统上下文图 | 外部参与者 / 系统、交互、输入、输出、信任边界 / 风险 |
| 容器 / 进程 / 组件视图 | COMP-ID、组件 / 进程、职责、部署位置、通信方式、阶段、状态、关联 REQ / NFR |
| 模块划分 | MOD-ID、模块、职责、输入、输出、边界 / 不负责、关联设计、关联组件 |
| 关键流程与数据流 | Flow-ID、成功路径、异常路径、降级路径、权限拒绝、外部服务不可用、关联 API / 数据 / TC |
| 部署 / 运行拓扑约束 | 环境、拓扑、端口 / 进程、资源要求、持久化、外部依赖、Mock / 降级、备注 |
| 架构决策与取舍 | ADR、决策、状态、适用 Phase、备选方案、取舍影响、验证方式 / 证据 |
| REQ / 功能 → 模块追溯 | REQ / NFR、Phase、COMP-ID、MOD-ID、Flow-ID、详细设计、覆盖状态 |
| 待人工确认项 | 结构化确认项表，不得只写问题列表 |

### 4.2 架构视图检查表

建议在 `docs-system-audit`、`docs-evaluation`、`phase-upgrade` 中增加 `04` 视图检查表：

| 视图 | 必查项 | 通过标准 | 缺失修复 |
|---|---|---|---|
| 系统上下文 | 用户、外部系统、核心服务、边界 | 明确哪些外部系统是真实、Mock、候选或默认关闭 | 补上下文图或边界说明 |
| 组件 / 容器 | 前端、后端、数据、适配层、测试入口 | 每个组件有职责、阶段、状态和覆盖 REQ | 补组件表或模块图 |
| 关键流程 | 主流程、异常流程、降级流程 | 流程能关联 API、服务、数据对象或 TC | 补流程图和映射 |
| 运行拓扑 | 本机、服务器、Docker、外部服务 | 明确端口、资源、可选依赖和降级路径 | 补拓扑和资源约束 |
| ADR / 决策 | 关键技术 / 交付 / 安全决策 | 有状态、理由、备选、取舍和验证方式 | 补 ADR 字段 |

### 4.3 `05-tech-spec` 最低要求

建议在文档标准中明确 `05` 是技术状态、依赖配置和风险验证事实源，至少覆盖：

| 能力 | 最低字段 / 结构 |
|---|---|
| 技术栈与版本 | 类别、技术 / 版本、状态、用途、约束来源、替代品禁令 / 备注 |
| 关键技术决策 | 决策 ID、决策、理由、替代方案、影响范围、权威源、验证状态 |
| 依赖与配置 | 类型、名称、用途、启用阶段、当前状态、配置来源、密钥 / 敏感性、验证方式 |
| 运行环境与资源评估 | 运行项、本机要求、当前本机是否满足、验证证据、降级 / Mock、服务器预案 |
| 技术环境评估结论 | 评估报告路径、Go / Conditional Go / No-Go、阻塞项、需回填章节 |
| Phase 技术约束 | Phase、允许、禁止、Mock / 降级、技术状态说明、权威源 |
| 前端交互设计边界 | 是否触发 `docs/design/frontend-interaction.md`、权限边界、接口依赖、验收路径 |
| 安全、隐私与合规 | 项、要求、阶段、状态、权威源、验证入口 |
| 技术风险与验证计划 | Risk-ID、风险、触发条件、影响、验证方式、对应用例 / 任务、解锁条件 |
| 待人工确认项 | 结构化确认项表，不得把 AI 建议写成确认事实 |

### 4.4 技术状态与依赖配置矩阵

Batch 3 建议在 `05` 标准中引用 Batch 1 的横切状态词典，并将状态绑定到依赖、验证和启用阶段：

| 类型 | 名称 | 用途 | 启用阶段 | 当前状态 | 配置来源 | 密钥 / 敏感性 | 验证方式 |
|---|---|---|---|---|---|---|---|
| Python 包 / Node 包 / Docker 服务 / 外部 API / 数据库 |  |  | Phase1 / Phase2 | 已启用 / 已验证 / 候选 / 默认关闭 / Mock / 降级 / 禁止 | `.env` / 配置文件 / 手工输入 | 无 / secret / token / 隐私 | 命令 / TC / Spike |

状态使用原则：

1. `候选` 不得作为实现事实。
2. `已验证` 不等于 `已启用`；必须说明是否进入当前 Phase。
3. `Mock` / `降级` 不等价真实能力通过，必须有后续验证或解锁条件。
4. `默认关闭` 能力不得被 Sprint 默认实现，除非 Phase 或人工确认解锁。
5. `禁止` 状态必须被 `04`、`05`、`08` 和代码任务共同遵守。

## 5. 技术风险验证闭环

建议在 `05`、`09`、`tech-env-evaluation` 和 `phase-upgrade` 中形成同一条闭环：

```text
05 Risk-ID / 依赖 / 技术状态
  → 技术环境评估 / Spike / PoC / 最小运行验证
  → 09 TC / 资源验证 / 安全验证
  → 08 Sprint / Task 解锁条件
  → Phase 升级结论 / 验收证据
```

建议的技术风险矩阵：

| Risk-ID | 风险 | 触发条件 | 影响 | 当前状态 | 验证方式 | 对应 TC / Sprint | 解锁条件 | 降级 / 预案 |
|---|---|---|---|---|---|---|---|---|
| RISK-001 |  |  |  | 候选 / 待验证 / 已验证 / 降级 | 命令 / TC / Spike / 人工确认 | TC-... / Sprint-... |  |  |

最低规则：

- 当前 Phase 涉及真实运行依赖、外部服务、数据库、LLM、部署、密钥或敏感数据时，必须存在 Risk-ID 或明确豁免理由。
- P0 / P1 技术风险必须映射到 `09` 验证、Spike / PoC 或 Sprint 解锁条件。
- 技术环境评估报告不替代 `05`，其结论确认后应回填或引用到 `05` 和 `09`。
- `collect-env` 只采集事实，不等于技术可行性验证。

## 6. Readiness gate

建议引入条件触发的 readiness gate，不要求所有 Lean 项目填满，但以下能力进入 Sprint / Phase 升级前必须检查：

| 能力类型 | 解锁前必须检查 | 不满足时处理 |
|---|---|---|
| 数据库 / 迁移 / 向量库 | 连接、迁移、持久化、回滚或降级方案 | 降级到文件 / 内存 / Mock，或阻塞相关 Sprint |
| 外部 API / 第三方服务 | 凭据管理、网络、额度、失败兜底、隐私条款 | 默认关闭或 Mock，不得承诺真实能力 |
| LLM / OCR / 模型 | 成本、延迟、隐私、兜底、安全边界、最小运行验证 | 保持候选 / 默认关闭，转人工或规则降级 |
| Docker / 部署 | 镜像、端口、资源、权限、环境变量、日志 | 降级为本机运行或推迟部署能力 |
| 敏感数据 / 权限 | 权限边界、脱敏、审计、日志留存、服务层控制 | 不得用前端隐藏 / 禁用替代后端权限 |

readiness gate 输出应至少包含：能力、当前状态、阻塞关系、验证证据、解锁条件、降级路径、需回填文档。

## 8. 版本影响

建议作为 MINOR 版本处理。

理由：

- 新增 / 恢复 `ai/doc-standards/04-05` 标准文件，影响架构 / 技术方案生成、审计和评估基线。
- 增强架构视图检查、依赖配置矩阵、技术风险验证闭环和 readiness gate。
- 会改变 AI 对真实依赖、外部服务、数据库、LLM、部署和安全隐私风险的默认判断方式。

## 9. 验收口径

1. `ai/doc-standards/04-05` 存在，并分别覆盖架构约束、分层视图、组件 / 模块 / Flow ID、ADR、技术状态、依赖配置、风险验证和 readiness gate。
2. `docs/04-05` 模板骨架与标准文件保持一致，且不引入项目专属事实。
3. `ai/document-lifecycle-rules.md` 明确 04-05 如何承接 NFR、约束、技术环境评估和风险验证。
4. `implementation-lifecycle-rules` 在真实依赖 Sprint 前能引用 `05` 风险矩阵和 readiness gate。
5. `docs-system-audit` 和 `docs-evaluation` 能检查上下文图、组件图、流程图、拓扑、ADR、依赖配置和风险验证闭环。
6. `tech-env-evaluation` 输出可被引用回 `05` Risk-ID、`09` 验证项和 Sprint 解锁条件。
7. `phase-upgrade` 能在升阶段前检查真实依赖是否满足 readiness gate。
8. 自检脚本能防止关键 04-05 标准、风险矩阵、readiness gate 和 `05 ↔ 09` 映射规则被误删。
9. 合并后关闭 #106、#113，并在关闭说明中说明 #106 与 #113 的合并处理方式。

## 10. 非目标

- 不落地 `00-03` 需求链规范；该范围由 Batch 2（#105、#112）处理。
- 不落地 `06-07` DB / API 契约规范；该范围留给 Batch 4（#107、#114）。
- 不落地 `08-09` 开发计划与验证证据规范；该范围留给 Batch 5（#108、#109、#115）。
- 不落地 `docs/design/*` 详细设计标准；该范围留给 Batch 6（#110、#116）。
- 不要求所有项目必须使用 Docker、数据库、LLM 或外部服务。
- 不把候选技术写成已启用技术，不在 Phase 未确认前提前设计远期真实集成细节。
- 不用前端隐藏、按钮禁用或路由守卫替代后端权限和服务层安全控制。
- 不重写旧派生项目文档；旧项目可通过追加矩阵和最小补齐方式兼容。
