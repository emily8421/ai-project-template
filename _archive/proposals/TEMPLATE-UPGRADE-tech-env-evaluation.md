# TEMPLATE-UPGRADE: 技术路线与环境支撑评估机制

> 来源：GitHub issue #87（https://github.com/emily8421/ai-project-template/issues/87）。
> 类型：派生项目回流的模板优化提案。
> 状态：已实施并归档；已从 issue 收件箱转为本地执行稿。
> 建议实施批次：PR-E，建议在前述小步机制落地后独立实施。

## 1. 背景与问题

模板已有环境采集、技术方案、验证计划和资源约束，但缺少系统性的“技术路线与环境支撑评估”机制。现有流程容易出现：

- 环境采集只记录工具版本和硬件资源，没有验证项目锁定依赖能否真实安装 / 运行。
- 技术方案可能直接选择某个运行时版本，却没有说明为什么不用本机默认最新版。
- Demo / MVP / 产品阶段对依赖成熟度、工具链、模型、Docker 镜像和网络权限的要求不同，但缺少阶段化评估报告。
- 编码开始后才发现 Python / Node / Docker / 模型依赖不兼容，导致返工。
- 多项目长期维护时，每个项目锁不同运行时版本，增加本机维护成本。

派生项目暴露出的具体问题是：本机采集到 Python 3.14.3，但技术方案固定后端 Python 3.12.x；按旧依赖在 Python 3.14 安装时 `pydantic-core` 触发源码构建并失败。该问题本应在编码前通过依赖安装验证发现。

## 2. 设计目标

- 明确“默认贴近本机 / 团队当前主版本，除非实测不兼容再降级”的原则。
- 区分“环境采集”和“环境支撑评估”：前者记录事实，后者执行验证并形成决策。
- 将技术路线、运行时版本、关键依赖、Docker 镜像、模型、网络权限、资源约束纳入同一评估。
- 要求评估报告写入 `docs/research/`，并驱动 `docs/05-tech-spec.md`、`docs/09-verification.md` 与依赖文件。
- 对 Demo / MVP / 产品给出不同深度的评估要求。

## 3. 新增报告类型

推荐路径：

```text
docs/research/YYYY-MM-DD-tech-env-evaluation-<phase-or-scope>.md
```

报告定位：

- 属于 `docs/research/` 下的评估留痕。
- 不替代 `docs/05-tech-spec.md`；评估结论确认后再修订 05。
- 不替代 `docs/env/local-env.md`；环境采集是输入，评估报告是判断。

推荐结构：

```markdown
# 技术路线与环境评估报告：<scope>

## 1. 评估摘要
## 2. 评估范围与依据
## 3. 本机 / 团队环境事实
## 4. 技术路线候选
## 5. 依赖与工具支撑矩阵
## 6. 安装 / 导入 / 最小运行验证
## 7. 资源、网络、权限与 Docker 验证
## 8. 风险项与降级策略
## 9. Go / Conditional Go / No-Go 结论
## 10. 对 docs/05、docs/09、依赖文件的修改建议
## 11. 待人工确认项
```

## 4. 评估维度

| 维度 | 必查问题 | 输出 |
|---|---|---|
| 运行时版本 | 是否优先使用本机 / 团队当前主版本；旧版本是否有必要 | 运行时版本决策 |
| 后端依赖 | Web 框架、ORM、迁移、驱动是否支持目标 Python | 安装 / 导入 / 最小运行结果 |
| 前端依赖 | Node / npm / 框架 / 构建工具是否支持目标版本 | 安装 / build / dev server 结果 |
| 数据库 / Docker | 镜像是否可拉取、容器能否启动、扩展是否可用 | Compose 验证结果 |
| AI / 模型 | Embedding、LLM adapter、OCR、解析库是否支持目标环境 | 模型加载 / Mock / 降级结论 |
| 网络与权限 | 是否需要外网、代理、内网中转、管理员权限、端口绑定 | 风险与替代路径 |
| 资源 | CPU / 内存 / 显存 / 磁盘是否满足阶段目标 | 资源瓶颈与服务器预案 |
| 维护成本 | 是否引入多版本运行时、重型工具链或难维护 SDK | 长期维护判断 |

## 5. 阶段深度建议

| 阶段 / 交付物形态 | 评估深度 |
|---|---|
| Demo | 必须验证核心链路依赖安装、导入、最小运行；允许 Mock / 降级，但要显式记录 |
| MVP | 必须验证生产必要依赖、数据库迁移、部署脚本、错误恢复和资源边界 |
| 产品 | 必须验证长期维护、升级策略、安全合规、可观测性、容量和多环境部署 |

## 6. 模板规则建议

- 生成或修订 `docs/05-tech-spec.md` 前，若项目涉及真实运行环境，必须先完成技术环境评估或说明为何可跳过。
- 进入首个编码 Sprint 前，必须至少执行一次依赖安装 / 导入 / 最小运行验证。
- 若本机环境与技术方案基线不一致，必须列为风险项。
- 默认优先贴近本机 / 团队当前主版本；除非依赖实测不支持，否则不应为了模板习惯固定旧版本。
- 评估结论为 Conditional Go / No-Go 时，不得直接进入会触发该风险的 Sprint。
- Python 项目启动说明必须显式区分“创建虚拟环境 / 激活虚拟环境 / 安装依赖 / 启动服务”。

## 7. 编码前门禁

对于保留 `backend/`、`frontend/`、`docker/`、本机模型、外部 API、数据库或其他真实运行依赖的项目，技术路线与环境支撑评估报告应作为进入首个编码 Sprint 的必备输入。

门禁规则：

- 未形成评估报告时，不得开始编码。
- 若用户明确要求跳过，AI 必须记录跳过原因、风险项、影响范围和补做时点，并写入续接文件或项目事实文档。
- 若评估结论为 `No-Go`，不得进入相关 Sprint，必须先修订技术路线或降级方案。
- 若评估结论为 `Conditional Go`，只能进入不触发该风险的 Sprint。
- 评估报告必须可追溯到 `docs/env/*`、`docs/05-tech-spec.md`、`docs/09-verification.md`、依赖文件和运行脚本。

## 9. 与现有机制关系

| 现有机制 | 关系 |
|---|---|
| `collect-env` | 采集事实，作为评估输入 |
| `docs-checklist` | 可检查是否已有技术环境评估报告 |
| `docs-system-audit` | 可审计技术方案与环境评估是否一致 |

## 10. 版本影响

建议作为 minor 版本落地。新增命令、Prompt 与文档检查项，不破坏既有派生项目。已有项目无需立即补评估报告；当进入首个编码 Sprint、修订技术栈或遇到环境问题时触发补做。

## 11. 验收口径

- 用户输入“技术环境评估”时，AI 能路由到专门流程。
- AI 能发现本机环境与 `docs/05-tech-spec.md` 基线不一致，并列为风险。
- AI 不会只因为模板习惯而固定旧运行时版本；必须说明为什么不用当前主版本。
- 报告能给出 Go / Conditional Go / No-Go 结论。
- 评估报告能驱动修订 `docs/05-tech-spec.md`、`docs/09-verification.md` 与依赖文件。
- Demo / MVP / 产品阶段能按不同深度执行环境支撑评估。
- 对存在真实运行依赖的项目，首个编码 Sprint 前必须能检查到技术环境评估报告；缺失时 AI 应阻止开工或记录用户明确跳过决定。
- Python 项目文档能指导用户从“没有 `.venv`”开始完成环境创建、依赖安装、启动与常见失败处理，并能识别非完全隔离环境的风险。

## 12. 风险与缓解

- **一次性强制所有旧项目补齐成本高**：仅对含真实运行依赖且尚未进入编码的项目强制，对已运行项目作为补做建议。
- **联网 / 安装操作有风险**：Prompt 必须区分只读评估、实际安装验证和需要用户确认的操作。
- **与 docs-evaluation 重叠**：明确本机制关注技术路线和可运行性，不替代文档质量评估。
- **环境结论过期**：评估报告记录日期和范围，技术栈修订或环境问题出现时重新评估。
