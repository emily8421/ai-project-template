# 09 Verification Standard（验证计划规范镜像）

> Sync notice: This file is maintained by `ai-project-template` and may be overwritten when a derived project syncs template methodology.
> Do not edit it directly in derived projects; propose reusable changes in `_proposals/` and upstream them to the template repository.

本文件是 `docs/09-verification.md` 的细粒度标准，用于 AI 生成、修订、审计和评估验证计划。它不是项目事实文档，派生项目的实际验证计划、证据和验收记录仍写入 `docs/09-verification.md`。

## 1. 定位与边界

`09` 是验证计划、证据、验收记录、缺陷 / 回归和未验证风险事实源，承接 `02/03` 需求与 Phase、`04-07` 设计 / 契约和 `08` Sprint / Task。

- 当前 Phase 的每个 REQ 至少映射到一个可执行、可复现、可记录证据的 TC。
- 不得把后续 Phase 用例列为当前必过项；可保留骨架但状态必须标为 `后续阶段`。
- 不得把 Mock / 降级 / 条件通过写成真实能力已验证，除非有明确风险接受和补验时点。
- 不得用聊天记录或 handoff 替代正式验收记录；长期验证事实必须回写 `09` 或说明暂不落盘原因。

## 2. 最低结构

| 能力 | 最低字段 / 结构 |
|---|---|
| 文档元信息 | 当前 Phase、交付物形态、覆盖 REQ、验证状态、最后更新、上游依据 |
| 验证策略 | 测试等级矩阵、证据形式、不适用原因 |
| REQ → TC 追溯 | REQ-ID、阶段、交付物形态、TC-ID、验证方式、通过标准、自动化 / 证据位置、状态 |
| TC 用例详情 | 覆盖 REQ、验证类型、前置条件、测试步骤、输入数据、期望结果、异常验证、自动化位置、证据记录 |
| 分阶段验证范围 | Phase、交付物形态、必须通过、不作为当前必过、退出标准 |
| Phase 测试大纲 | Phase、必过 TC、自动化命令、人工验收点、资源 / 环境验证、退出标准 |
| 本机资源与技术环境验证 | 启动、build、端口、Docker / DB / 模型、网络 / 权限、资源指标、后续动作 |
| Mock / 降级 / 外部服务验收口径 | 当前状态、是否等价真实能力、可接受条件、补验时点、风险接受方式 |
| Sprint / Phase 验收记录 | 日期、范围、执行人、结果、命令 / 步骤、验证证据、失败项 / 后续任务 |
| Sprint 验收包 | Sprint / Task、关联 REQ、关联 TC、改动文件、验证命令 / 人工步骤、结果、残留风险 |
| 缺陷与回归记录 | 缺陷 / 变更 ID、来源 / 现象、修复 Task、回归范围、回归命令 / 步骤、回归结果、后续动作 |
| 风险与未验证项 | 编号、未验证项 / 风险、原因、影响、后续计划 |
| 待人工确认项 | 结构化确认项表，不得把 AI 建议写成确认事实 |

## 3. TC 状态枚举

| 状态 | 含义 |
|---|---|
| 待设计 | 仅有 REQ，尚未定义 TC；当前 Phase 不应长期保留 |
| 待执行 | TC 已定义但尚未执行 |
| 自动化已覆盖 | 已有自动化测试覆盖，但尚未完成人工 / 验收确认 |
| 待人工验收 | 自动化或实现已就绪，等待人工 smoke / 验收 |
| 通过 | 已按 `09` 记录证据并通过 |
| 条件通过 | 通过需附带 Mock / 降级 / 风险接受口径 |
| 失败 | 已执行但不满足通过标准 |
| 阻塞 | 环境、依赖、账号、资源或设计缺口导致无法执行 |
| 后续阶段 | 非当前 Phase 必过项 |

## 4. TC 用例详情最低字段

当前 Phase 的每个 TC 至少包含：覆盖 REQ、验证类型、前置条件、测试步骤、输入数据、期望结果、失败 / 异常验证、自动化位置、证据记录。

## 5. 正式回写与 handoff 边界

- `08` 管计划、拆分、进度摘要和完成摘要；`09` 管验证计划、证据、验收记录、缺陷 / 回归和未验证风险。
- `.ai/session-handoff.md` 只用于本地会话续接，不替代正式 `08/09` 进度和验收记录。
- Sprint 完成后，如存在验收事实、验证命令、截图 / 日志、残留风险或状态变化，应回填 `09` 或明确暂不落盘原因。
- Sprint 完成后，如改变 Sprint 状态、下一步、依赖或任务拆分，应回填 `08` 或明确暂不落盘原因。
- Phase 升级前，应检查 `03`、`08`、`09`、`ai/project-rules.md`、README 和 handoff 是否存在状态冲突。
- 未经人工确认，AI 不得自动把 Phase 状态改为已完成、不得自动进入下一 Phase。

## 6. 回写触发清单

| 触发 | 应检查 / 回写 |
|---|---|
| Sprint 完成 | `08` 状态 / 下一步、`09` 验收记录 / Sprint 验收包、残留风险 |
| Bug 修复 | `09` 缺陷与回归记录、相关 TC 状态、`08` 当前进度 |
| Mock / 降级验收 | `09` 条件通过 / 风险接受、`08` 后续补齐任务 |
| Phase 验收 | `03` Phase 状态、`08` Phase 完成情况、`09` Phase 验收、`project-rules` 当前阶段 |
| Phase 升级 | `project-rules`、`03/08/09`、README、handoff 状态一致性 |

## 7. 审计检查

- 当前 Phase REQ 是否都有 TC-ID、通过标准、自动化 / 人工步骤和证据位置。
- TC 状态是否准确；Mock / 降级是否只记为条件通过或风险接受。
- Sprint / Phase 验收记录是否包含日期、范围、执行人、结果、命令 / 步骤、证据和失败项。
- 缺陷、公共 API、数据模型、状态枚举、权限、安全、真实外部服务、数据库、LLM、部署脚本或新依赖变更后，是否有缺陷 / 回归记录。
- 验证事实是否只停留在 handoff、聊天或 PR 描述中，未回写正式 `09`。

## 8. 下游影响

- 给 `08`：Sprint 验证包、完成包、进度状态和下一步依据。
- 给 tests / CI：自动化命令、测试文件、证据路径和回归范围。
- 给 Phase 升级：必过 TC、资源验证、readiness gate、验收记录和残留风险结论。
