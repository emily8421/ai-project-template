## Raw Issue Body

# TEMPLATE-UPGRADE-v1.44.0-ui-prototype-gate

> 来源：LUMEN-DEMO（emily8421/LUMEN-DEMO）派生项目回流

## 1. 动机

在涉及可点击 UI 的项目中，仅按需求 / API / 后端能力推进到功能基本完成后，用户才看到真实界面，容易出现“功能满足，但界面风格、信息密度、交互结构不是用户想要的样子”的返工。

这类返工不是单纯 CSS 问题，通常源于 AI 与用户没有在实现前对齐：

- 操作界面的整体风格：工作台 / 表单 / 看板 / 聊天 / 文档编辑器 / 管理后台等。
- 页面信息架构：导航、上下文栏、主工作区、辅助面板如何分层。
- 信息密度：字号、列表行高、卡片感、首屏可见信息量。
- 阶段优先级：先做交互原型，还是先做后端数据链路 / 技术验证。

建议在模板方法论中增加“UI 原型确认 Gate”，让 AI 在进入前端编码前主动引导用户通过可视化原型对齐界面方向。

同时，UI 设计不应默认等待用户给出专业设计建议。多数用户并不熟悉字体、行高、密度、导航模式或设计系统术语，因此 AI 应默认采用行业常用标准和成熟产品惯例；只有当用户明确提出特殊风格、品牌规范、目标竞品或行业偏好时，才偏离默认标准。

## 2. 拟改目标

为涉及 UI 的 Sprint / Task 增加一个前置判断与确认流程：

1. AI 先判断本任务是否属于 UI 关键任务。
2. 如果是，AI 必须先询问或建议“交互界面优先 / 后端数据优先 / 双轨并行”的实施顺序。
3. 对需要 UI 优先的任务，AI 应先输出低保真可视化原型（可为 HTML / Mermaid / ASCII wireframe / 截图型代码原型），经用户确认后再进入正式实现。
4. 对需要后端优先的任务，也应说明 UI 暂缓原因、后续原型确认时点和风险。

## 3. 建议新增规则

### 3.1 UI 原型确认 Gate

当满足任一条件时，前端实现前必须先进入 UI 原型确认 Gate：

- 用户明确提到“界面、布局、交互、操作体验、看起来、风格、原型、可视化”等诉求。
- Sprint 修改范围包含页面、组件、导航、表单、搜索 / 问答 UI、管理页、桌面端集成或可点击流程。
- 项目交付物是 Demo / MVP，且用户需要对外演示或人工操作走查。
- 页面存在多角色、多视图、多步骤状态流、复杂列表 / 表单 / 编辑器 / 结果阅读区。
- 需求文档只描述功能，没有描述信息架构、视觉密度或关键用户路径。

Gate 输出至少包含：

- 目标用户与首要操作任务。
- 2–3 个布局候选或一个推荐布局，并说明取舍。
- 可视化原型路径或内嵌低保真图。
- 信息密度规范：字号、行高、列表行高、面板 padding、首屏可见信息量。
- 默认 UI 标准来源：采用行业常用设计系统 / 成熟产品惯例，并列出 AI 采用的基线（如桌面后台 / 知识库 / 编辑器 / 数据表格）。
- 需要用户确认的问题与 AI 推荐。

### 3.1.1 默认行业标准原则

AI 在没有用户明确特殊要求时，应主动采用行业常用 UI 标准，而不是把专业设计决策交给用户从零判断。

建议默认：

| UI 类型 | 默认参考风格 | 默认密度 / 字号 | 说明 |
|---|---|---|---|
| 管理后台 / 表单系统 | Ant Design / Fluent / Atlassian 类后台 | 正文 13–14px，label / meta 11–12px | 适合 CRUD、配置、权限、列表筛选 |
| 知识库 / 文档工作台 | Notion / Obsidian / Linear / 飞书文档类生产力工具 | 正文 13–14px，列表行 36–48px | 适合文档编辑、搜索、来源阅读、侧栏导航 |
| 数据密集表格 | Data grid / BI / issue tracker | 正文 12–13px，行高 32–40px | 适合大量记录、日志、指标、告警 |
| 聊天 / 问答界面 | ChatGPT / Slack / Copilot 类问答 | 正文 14–15px，消息气泡或阅读区 | 适合长答案阅读和多轮对话 |
| 营销落地页 / 展示页 | 官网 / Landing page | 正文 16px+，标题更大 | 适合品牌展示，不适合高密度工作台 |

默认原则：

- AI 应先给出推荐风格，例如“本项目是桌面知识库工作台，建议按生产力工具 / 后台系统密度，而不是营销页大卡片风格”。
- 用户只需确认“喜欢 / 不喜欢 / 更接近哪个产品”，而不是必须给出字号、行高、布局规则。
- 如用户没有特殊品牌规范，AI 不应默认使用大卡片、大圆角、大留白的通用 Demo 风格。
- 如果 AI 采用了非默认风格，必须说明原因和风险。

### 3.2 “UI 优先 vs 后端优先”场景引导

AI 应在计划阶段根据风险类型建议顺序：

| 场景 | 建议优先级 | 说明 |
|---|---|---|
| 用户体验、演示效果、操作路径是成败关键 | UI / 原型优先 | 先确认页面长什么样，再实现功能，避免后期大返工 |
| 数据模型、权限、算法、外部服务可行性未知 | 后端 / 技术验证优先 | 先验证核心链路；同时给 UI 低保真占位，不进入精细 UI |
| UI 与后端都高风险 | 双轨并行 | 后端做最小 vertical slice，前端做静态 / mock 原型，二者在接口契约处汇合 |
| 纯内部脚本、CLI、无点击界面 | 后端 / 功能验证优先 | 可豁免 UI 原型 Gate，但需记录豁免理由 |
| 既有成熟设计系统已指定 | 设计系统优先 | 直接按系统组件 / token 出原型，避免自造样式 |

AI 提问方式应避免把专业问题直接抛给用户，例如不要只问“你想要什么 UI 风格？”。更好的方式是：

```text
这个功能更像知识库工作台，我建议默认采用高密度生产力工具风格：左导航 / 上下文列表 / 主工作区，正文 13–14px、列表行 40–48px、少卡片少阴影。
如果你想要更像官网展示页、聊天应用或移动端卡片流，请特别说明。
我先给一个低保真原型，你确认方向后再实现。
```

### 3.3 原型确认后的文档回填

用户确认原型后，AI 应将结论回填到合适文档：

- `docs/design/frontend-interaction.md`：页面流、状态、交互、接口依赖。
- `docs/design/<feature-ui>.md`：复杂工作台或专题 UI 详细设计。
- `docs/08-dev-plan.md`：Sprint 修改范围、验收标准、禁止事项。
- `docs/09-verification.md`：浏览器 smoke、信息密度、关键路径验收项。

原型文件建议放置：

- 需求探索阶段：`docs/research/YYYY-MM-DD-ui-prototype-exploration.md` 或同目录 HTML。
- 正式详细设计阶段：`docs/design/<feature>-prototype.html` 或 `docs/design/<feature>-interaction.md` 附低保真图。

## 4. 建议修改文件

建议在模板仓库中更新：

- `ai/document-lifecycle-rules.md`：在 UI 原型策略处增加“UI 原型确认 Gate”和文档回填要求。
- `ai/implementation-lifecycle-rules.md`：在 Sprint / Task 执行前方案中增加“UI 优先 vs 后端优先”判断。
- `ai/project-rules.md` 模板说明：提醒项目初始化时声明 UI 原型策略与豁免条件。
- `ai/doc-standards/frontend-interaction.md`：增加信息密度、布局候选、可视化原型确认记录字段。
- 可选新增 `ai/prompts/design/ui-prototype-gate.md`：提供可复用提示词。

## 5. 验收标准

模板升级落地后，应满足：

- AI 在 UI 相关任务进入编码前，会主动提出是否需要先确认可视化原型。
- AI 会默认选择行业常用 UI 标准，并说明所选基线；用户无需掌握专业设计术语也能确认方向。
- 复杂 UI Sprint 的计划中包含原型确认、信息密度和浏览器 smoke 验收项。
- 用户可以在实现前看到低保真原型，而不是功能完成后才发现界面方向不符。
- 后端优先场景不会被强行拖入 UI 打磨，但会记录 UI 暂缓原因和后续确认点。

## 6. 影响面

- 正向影响：减少 UI 型项目后期返工，提高用户与 AI 对界面预期的一致性。
- 成本：UI 型任务前置增加一次原型确认步骤；对纯后端 / CLI 项目可豁免。
- 风险：如果规则写得过重，可能拖慢小任务；因此需保留“豁免理由”和“后端优先”分支。

## 7. 版本建议

- 建议作为模板 `v1.44.0` 或后续 minor 版本增强。
- 该改动属于流程 / 文档标准增强，不改变具体项目技术栈。
