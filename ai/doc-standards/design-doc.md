# Design Doc Standard（docs/design/* 通用详细设计标准）

> Sync notice: This file is maintained by `ai-project-template` and may be overwritten when a derived project syncs template methodology.
> Do not edit it directly in derived projects; propose reusable changes in `_proposals/` and upstream them to the template repository.

本文件定义 `docs/design/*` 的通用详细设计标准，供 AI 生成、精修、审计和评估非平凡子系统设计时使用。它是 `04-07` 总体设计 / 契约与 `08-09` 实现计划 / 验证之间的承接层，不替代 `docs/04-architecture.md`、`docs/05-tech-spec.md`、`docs/06-db-design.md` 或 `docs/07-api-spec.md`。

## 1. 适用范围与触发条件

满足以下任一条件时，应新增或补齐 `docs/design/<subsystem>.md`；若豁免，必须在 `ai/project-rules.md`、`docs/05-tech-spec.md`、评估报告或待确认项中说明理由：

- 非平凡子系统、跨模块流程、复杂状态机或异步任务。
- 多页面、多角色、复杂表单、状态流或验收依赖点击路径的 UI / 交互设计。
- 权限 / 安全边界、租户 / 空间隔离、敏感字段、审计或越权失败路径。
- AI / RAG / 外部模型、OCR、第三方服务、导入任务、回调、队列、限流、重试或真实凭据。
- Mock / 降级 / Demo 与真实能力存在差异，或当前 Phase 只允许候选 / 默认关闭 / 禁止实现。
- 高风险愿景能力需要 readiness gate、Spike / PoC 或人工确认后才能进入 Sprint / Phase。

简单项目、单一函数、小脚本或 `04/05/06/07` 已足够支撑实现和验证的能力可豁免；豁免不等于跳过权限、安全、验证和回写要求。

## 2. 设计边界

- `docs/design/*` 只能细化上游已批准需求、架构、技术方案、DB / API 契约和 Phase 范围。
- 不得新增 `03` 未批准需求、`06` 未同步表 / 字段、`07` 未同步接口、未批准验收目标或 Phase 外能力。
- 若详细设计发现缺表、缺字段、缺 API、缺 TC 或 Phase 范围冲突，应触发上游文档修订，而不是只在 design 中孤立补写。
- 前端隐藏入口、按钮禁用、文案提示或路由守卫只能作为可见性控制，不得替代后端接口、服务层、DB 或鉴权中间件的权限边界。
- `docs/design/frontend-interaction.md` 是 `docs/design/*` 的页面 / 交互型子类型，必须满足本标准的元信息、追溯、验收和待确认项要求。

## 3. 推荐文件结构

```markdown
# 详细设计：<子系统名>（<slug>）

## 0. 文档元信息
## 1. 职责与边界
## 2. 上游依据与追溯
## 3. 核心流程 / 状态机
## 4. 数据、接口与权限契约
## 5. 失败、异常与降级路径
## 6. 阶段增量、readiness gate 与实现状态
## 7. 验证与验收追溯
## 8. 与其他子系统交互
## 9. 实现偏差 / 设计回写
## 10. 待人工确认项
```

旧项目可先增量补 `0. 文档元信息`、追溯矩阵、readiness gate、实现偏差 / 设计回写和待人工确认项，不要求全文重写。

## 4. 必填结构

### 4.1 文档元信息

| 项 | 内容 |
|---|---|
| 设计对象 | 子系统 / 页面 / 模块 / 流程 / 策略 / 集成 |
| 文档路径 | `docs/design/<subsystem>.md` |
| 输入来源 | `docs/02-srs.md` / `03` / `04` / `05` / `06` / `07` / `docs/env/*` / `docs/research/*` |
| 覆盖 REQ / NFR | REQ-ID / NFR-ID / 边界场景 |
| 所属 Phase | `[P1]` / `[P2]` / `[愿景]` |
| 交付物形态 | Demo / MVP / 产品 / 不适用 |
| 当前状态 | 骨架 / Pn-已设计 / Pn-已实现 / 待验证 / 条件实现 / 降级实现 / 后续阶段 / 禁止当前阶段实现 |
| 最后更新 | YYYY-MM-DD |
| 下游影响 | `docs/08-dev-plan.md` / `docs/09-verification.md` / tasks / code / tests |

### 4.2 职责与边界

| 项 | 内容 |
|---|---|
| 本设计负责 | 具体流程、状态、权限、数据 / API 依赖、验收追溯 |
| 本设计不负责 | 明确排除的模块、接口、表、Phase 外能力 |
| 不新增内容 | 未授权需求 / 接口 / 表 / 验收目标 / Phase 外能力 |
| 权限边界 | 前端可见性、API 鉴权、服务层校验、DB 约束各自职责 |
| 依赖前置 | 真实服务、SDK、凭据、环境、迁移、评估或人工确认 |

### 4.3 上游依据与追溯

| 来源 | 章节 / ID | 本设计承接内容 | 下游影响 |
|---|---|---|---|
| `docs/02-srs.md` | REQ-... / NFR-... |  |  |
| `docs/03-prd.md` | Phase / 非目标 / AC |  |  |
| `docs/04-architecture.md` | COMP-ID / MOD-ID / Flow-ID |  |  |
| `docs/05-tech-spec.md` | Risk-ID / readiness gate / 技术状态 |  |  |
| `docs/06-db-design.md` | Table / Field / 数据契约 |  |  |
| `docs/07-api-spec.md` | API-ID / Error / Permission |  |  |
| `docs/08-dev-plan.md` | Sprint / Task / 输入设计 |  |  |
| `docs/09-verification.md` | TC-ID / 验收 / 缺陷 |  |  |

最低追溯链为：`REQ / NFR → Phase → COMP-ID / MOD-ID / Flow-ID → Table / Field → API-ID / Permission / Error → Design Point → Sprint / Task → TC-ID / 验收证据`。无 DB / API 的项目可裁剪 `Table / Field` 或 `API-ID`，但必须保留裁剪原因。

### 4.4 核心流程 / 状态机

每个关键流程建议使用稳定 ID，例如 `Flow-D-001`：

```markdown
### Flow-D-001：<流程名称>

- 触发：
- 参与者 / 组件：
- 输入：
- 步骤：
- 输出：
- 成功条件：
- 失败 / 降级路径：
- 关联 REQ / API-ID / TC-ID：
```

存在状态流、导入任务、异步任务、审批、队列或外部回调时必须补状态机：

| 状态 | 含义 | 进入条件 | 退出条件 | 用户 / 系统可见表现 | 终态 |
|---|---|---|---|---|---|

### 4.5 数据、接口与权限契约

| 能力 / 流程 | 数据表 / 字段 | API-ID / 命令 | 权限规则 | 错误码 | 契约状态 | 备注 |
|---|---|---|---|---|---|---|

- 字段、接口、错误码和权限以 `06/07` 为权威；design 只引用、解释和组合，不孤立定义稳定契约。
- 若设计需要新增字段、接口、错误码或权限规则，先标为待确认项并回写 `06/07`。
- 敏感字段、租户隔离、空间边界、越权失败和审计记录必须有后端 / 服务层 / DB 边界说明。

### 4.6 失败、异常与降级路径

| 场景 | 触发条件 | 系统行为 | 用户可见信息 | 记录 / 日志 | 是否阻塞验收 | 关联 TC |
|---|---|---|---|---|---|---|

Mock / 降级 / Demo 与真实能力差异必须单独说明：

| 能力 | 目标设计 | 当前实现 / Demo | Mock / 降级原因 | 是否等价真实能力 | 补齐时点 | 对验收影响 |
|---|---|---|---|---|---|---|

### 4.7 阶段增量、readiness gate 与实现状态

阶段增量：

| 阶段 | 功能范围 | 交付物形态 | 设计状态 | 实现状态 | 备注 |
|---|---|---|---|---|---|

readiness gate：

| 能力 | 当前状态 | 解锁条件 | 验证证据 | 降级路径 | 是否阻塞 Sprint / Phase |
|---|---|---|---|---|---|

状态枚举建议：`骨架`、`Pn-已设计`、`Pn-已实现`、`待验证`、`条件实现`、`降级实现`、`后续阶段`、`禁止当前阶段实现`。候选、高风险或默认关闭能力不得写成当前已实现或当前必过验收。

### 4.8 验证与验收追溯

| 设计点 | 关联 REQ | 关联 Sprint / Task | 关联 TC | 验证方式 | 状态 |
|---|---|---|---|---|---|

- 当前 Phase 的关键设计点必须能映射到 `08/09`。
- 高风险愿景设计可标 `待 Spike / PoC`，不得列为当前必过验收。
- 验证证据和验收结论仍以 `docs/09-verification.md` 为正式记录，design 只保留设计点到 TC 的入口。

| 方向 | 子系统 / 文档 | 交互内容 | 契约来源 | 风险 |
|---|---|---|---|---|
| 依赖 |  |  |  |  |
| 影响 |  |  |  |  |
| 被调用 |  |  |  |  |

### 4.10 实现偏差 / 设计回写

| 偏差 ID | 代码 / 配置事实 | 原设计 | 偏差类型 | 处理结论 | 回写目标 | 验证 / 证据 |
|---|---|---|---|---|---|---|

偏差类型可包括：`实现偏离`、`设计滞后`、`Mock / 降级`、`默认关闭`、`后续阶段`、`验证缺口`。长期事实不得只留在聊天、PR 或 `.ai/session-handoff.md`；需要回写 `docs/design/*`、`06/07`、`08/09` 或明确暂不落盘原因。

### 4.11 待人工确认项

| ID | 待确认项 | AI 建议 | 建议依据 | 备选方案 | 取舍影响 / 阻塞关系 |
|---|---|---|---|---|---|

AI 建议不等于用户已确认事实；用户确认后才可回填权威文档、任务单或实现计划。

## 5. 分类 checklist

| 类型 | 必需重点 | 可裁剪项 |
|---|---|---|
| 服务型 | 服务职责、输入输出、错误处理、数据 / API 依赖、日志、安全、幂等 | 页面结构、响应式 |
| 页面 / 交互型 | 页面清单、用户路径、状态模型、表单校验、文案、接口依赖、验收路径 | 后端内部服务流程 |
| 权限 / 安全型 | 身份、授权、租户 / 空间边界、敏感字段、审计、越权失败 | 页面布局 |
| AI / RAG / 外部模型型 | 证据来源、召回 / 排序、提示词边界、成本、隐私、兜底、不可自动承诺 | 常规 CRUD 细节 |
| 导入 / 异步任务 / 外部集成型 | 状态机、幂等、重试、超时、限流、凭据、Mock / 真实边界 | 纯展示细节 |
| 策略 / 规则型 | 规则优先级、冲突处理、高风险边界、证据来源、不得提前实现声明 | 页面布局 |
| 配置型 | 配置字段、版本、校验规则、加载 / 发布状态、失败处理 | 复杂运行拓扑 |
| 高风险愿景型 | 当前 Phase 禁止项、候选状态、readiness gate、人工确认、降级路径 | 已确认实现细节 |

## 6. 审计与评估口径

审计 / 评估 `docs/design/*` 时至少检查：

1. 是否触发详细设计；未创建时是否有明确豁免理由。
2. 是否具备元信息、职责边界、上游依据、追溯矩阵、流程 / 状态机、失败 / 降级路径。
3. 是否正确引用 `06/07` 契约，未孤立新增字段、接口、错误码或权限。
4. 是否存在 readiness gate，候选 / 高风险 / 默认关闭能力是否有解锁条件、验证证据和降级路径。
5. 是否区分 Mock / 降级 / Demo 与真实能力，并说明对验收的影响。
6. 是否能输出到 `08/09` 的 Sprint、Task、TC 和验收路径。
7. 是否保留实现偏差 / 设计回写区，且长期事实已回写正式文档或说明暂不落盘原因。
8. 待人工确认项是否包含 AI 建议、依据、备选方案和阻塞关系。

## 7. 禁止项

- 禁止把 `docs/design/*` 写成新的 PRD、API 规范、DB 设计或验收报告。
- 禁止用 design 文档绕过 `03` Phase、`06/07` 契约或 `08/09` 验证门槛。
- 禁止把后续阶段、愿景能力、候选能力或 Mock 能力描述为当前可交付真实能力。
- 禁止只有 happy path，没有失败、异常、权限拒绝、降级或不可用路径。
- 禁止待确认项只列问题，不给 AI 建议、依据、备选方案和影响。
