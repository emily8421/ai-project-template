## 0. 为什么是瘦身版

维护者指出：PR #206 已把 Checkpoint Mode 从「每步人工确认」改为「风险分级、只高风险确认」，效率已回升且基本满足需求。本 patch **不动 PR #206 的确认规则**，也不加紧汇报频率；只补四类**零降效**缺口：

1. 规则在文件里、不在执行流里 → codex 长任务中遗忘 → **传递加固（A2）**。
2. 大上下文 / 长任务没有自动触发信号 → **触发条件补充（A3）**。
3. 大范围搜索后不回任务原点 → 搜索失忆 → **搜索回锚点**。
4. 必读上下文过重 → 跑飞 → **每层最小必读**（从主线 B 并入；唯一直接缓解跑飞根因的结构性手段）。

## 1. A2：codex 传递加固

### 1.1 `AGENTS.md` 头部前置（建议新增段落）

位置：`AGENTS.md` 紧接项目入口说明之后、`ai/index.md` 路由之前。

```text
## Checkpoint 节拍（codex 长任务防跑飞，每次进入都先读这段）

- 搜索批次（≥3 文件 / 全仓 grep）后，先回任务原点汇报"发现什么 / 还差什么 / 是否仍相关"，再继续。
- 命令失败、超时、sandbox / network / 权限错误、CI pending → 立即停并汇报，不连续重试、不继续后续步骤。
- push / 创建或合并 PR / 关闭 issue / 删除分支 / 发布 / 装依赖 / 破坏性命令 → 单步等确认。
- 低风险只读与已授权范围内编辑可合并小批次，批次末 1–3 行汇报即可，不必每步停下。
- 新增规则前先问：能否合并或删除既有规则？防跑飞靠更少更硬更自动触发，不靠更多规则。
```

依据：codex 原生读 `AGENTS.md`；长任务中全局规则最易被遗忘，入口前置 + 命令头部前置最有效（见 `ai/global-rules.md` §7）。

### 1.2 每个 `ai/commands/*.md` 头部前置（建议）

在每个命令文件标题下方加一行引用 + 节拍要点缩略：

```text
> ⏸️ Checkpoint 节拍：搜索批次后回锚点汇报；失败即停；高风险单步确认；低风险合并批次末汇报。详见 `AGENTS.md` Checkpoint 节拍段。
```

依据：命令即触发——用户通过 `/run` 等命令启动任务时，规则随命令加载，不依赖「全局规则被记住」。

## 2. A3：大上下文 / 长任务自动触发

### `ai/rules-core.md` §2 不可跳过的硬规则（建议追加一条）

old（现状，末条附近）：

```text
- 不得把 AI 建议、候选方案、研究记录、原型探索或未确认事项写成已确认项目事实。
```

new（在其后追加）：

```text
- 不得把 AI 建议、候选方案、研究记录、原型探索或未确认事项写成已确认项目事实。
- 长任务或已读上下文明显增大（如连续读取多个大规则 / 文档 / 长日志）时，默认进入 Checkpoint Mode：上下文越重，跑飞与失忆风险越高。
```

依据：维护者补充现象「模板体量越大、需读上下文越多 → 不确定性事件」——把这一根因显式写成触发信号。

## 3. 搜索回锚点

### `ai/rules-core.md` §4 输出与上下文卫生（建议追加一条）

new（追加，**不动既有「灵活小批次」措辞**）：

```text
- 大范围搜索（≥3 文件 / 全仓 grep）完成后，必须先回到任务原点汇报：发现了什么、还差什么、是否仍与当前任务相关，再决定下一步；不得搜索完直接继续而不回锚点。
```

依据：维护者现象「执行大范围搜索忘了原先需要的汇报」——搜索后强制回锚点，防失忆。注意：这是「搜索后汇报」，不是「每写入动作汇报」，不影响低风险批量任务的连贯性。

## 3.5 每层最小必读（从主线 B 并入）

> 2026-07-15 主线 B 降级时并入：这是主线 B 三个解耦机制里唯一直接缓解跑飞根因的部分，归入主线 A 更合适；契约分层 / 影响域清单另作处置（见提案 §5.3）。

### `ai/rules-core.md` §4 输出与上下文卫生（建议追加一条）

new（追加）：

```text
- 按任务所属层只读 Core + 本层契约的「必读文件」，不预读全模板；跨层信息通过层间契约获取，不靠全量加载——上下文越重跑飞风险越高。
```

依据：维护者根因「模板体量越大、需读上下文越多 → 不确定性事件」。最小必读把"全模板必读"收敛为"层内必读"，直接减重。落地依赖各 Profile 契约的「必读文件」字段（提案 §6 契约模板已有）。
