## Raw Issue Body

﻿# TEMPLATE-UPGRADE: 细化 09 Verification 文档规范

> 来源：LUMEN_demo_T2.1（emily8421/LUMEN-DEMO）派生项目回流

## 1. 背景与问题

派生项目在进入桌面端集成与 Phase 验收前，对 `docs/09-verification.md` 做单文档评估时发现：当前模板已经要求 09 文档包含验证策略、REQ → 用例追溯、资源验证、验收记录和风险项，但规范粒度仍偏骨架，容易让派生项目生成“看似有验证计划、实际不能直接验收”的文档。

典型问题包括：

1. REQ → 用例矩阵只写自然语言验收句，没有稳定 `TC-ID`、验证方式、通过标准、自动化位置和状态枚举。
2. 缺少“用例详情”约束，导致浏览器 smoke、API 集成、资源验证和人工验收步骤只能留在会话上下文或本地续接文件中。
3. 对 Mock / 降级能力的验收口径不够明确，容易把“Demo 可降级演示”误写成“真实能力已通过”。
4. 资源 / 环境验证章节只列约束，未强制记录启动、build、端口、Docker / DB / 模型、内存 / 显存 / 磁盘、外部服务可用性等证据字段。
5. Sprint / Phase 验收记录字段过少，不能稳定承载执行人、命令、日志、截图、失败项、后续任务和回归结果。
6. UI 型项目虽然已有前端交互设计触发规则，但 09 文档未强制把浏览器 smoke、桌面端兼容、页面状态、权限可见性和导入 / 搜索 / 问答等点击路径转成可验收 TC。
7. 已进入实现期后，文档仍可能停留在“待实现”状态，缺少“自动化已覆盖 / 待人工验收 / 条件通过 / 失败 / 阻塞 / 后续阶段”等细分状态。

这些问题不依赖具体业务项目；凡是采用 `docs/00-09` 文档驱动开发、并需要从实现计划进入验证 / 验收阶段的派生项目，都可能遇到相同的验收粒度不足问题。

## 2. 目标

细化模板中的 `09-verification` 规范，使派生项目生成或修订 09 文档时能够直接支撑：

1. 从 `REQ → Phase → Sprint / Task → TC → 自动化测试 / 人工验收 → 验收记录` 的完整追溯。
2. 对每个当前 Phase REQ 形成可执行、可复现、可记录证据的 TC。
3. 清晰区分自动化测试、浏览器 / E2E smoke、人工验收、资源验证、回归验证与未验证风险。
4. 对 Mock / 降级 / 外部服务不可用 / 尚未生产化能力给出明确验收判定口径。
5. 在 Sprint 或 Phase 完成时沉淀最小验收包，而不是只写“测试通过”。
6. 支持 UI 型项目把前端交互设计中的页面流、状态、权限和浏览器范围映射到 09 用例。

## 4. 09 文档建议结构

建议将 `ai/doc-standards/09-verification.md` 的模板骨架细化为以下结构。

```markdown
# 09 验证计划（Verification Plan）

## 0. 文档元信息
## 1. 验证策略
## 1.1 测试等级矩阵
## 2. REQ → TC 追溯矩阵
## 3. TC 用例详情
## 4. 分阶段验证范围
## 4.1 Phase 测试大纲
## 5. 本机资源与技术环境验证
## 5.1 Mock / 降级 / 外部服务验收口径
## 6. Sprint / Phase 验收记录
## 6.1 Sprint 验收包
## 6.2 缺陷与回归记录
## 7. 风险与未验证项
## 8. 待人工确认项
```

## 5. 字段级规范建议

### 5.1 文档元信息

`09` 至少应声明：

| 字段 | 要求 |
|---|---|
| 当前 Phase | 与 `docs/03-prd.md` §3 和 `ai/project-rules.md` 当前阶段一致 |
| 交付物形态 | `Demo / MVP / 产品`，不得把 Demo 写成 MVP |
| 覆盖 REQ | 列当前 Phase 必须覆盖的 REQ；后续阶段只保留骨架 |
| 验证状态 | `草稿 / 待人工确认 / 已确认 / 验收中 / 已完成 / 部分完成` |
| 最后更新 | 使用绝对日期 |
| 上游依据 | 至少引用 `02/03/04/05/08` 与 `docs/env/*`，涉及接口 / DB / UI 时引用 `06/07/docs/design/*` |

### 5.2 测试等级矩阵

每个测试等级必须写“是否适用”；不适用必须说明原因。

| 测试等级 | 是否适用 | 覆盖范围 | 证据形式 | 不适用原因 / 备注 |
|---|---|---|---|---|
| 单元测试 | 是 / 否 | service / helper / component | 命令 + 日志 |  |
| 集成测试 | 是 / 否 | API / DB / 外部服务 / 模块协作 | 命令 + 测试数据 |  |
| 系统 / E2E | 是 / 否 | 主流程、浏览器 smoke、CLI smoke | 步骤 + 截图 / 日志 |  |
| 验收测试 | 是 / 否 | REQ / Sprint / Phase | TC-ID + 结论 |  |
| 回归测试 | 是 / 否 | 已有能力和修复项 | 历史用例 + 结果 |  |
| 资源 / 环境验证 | 是 / 否 | 启动、build、端口、依赖、模型、资源 | 指标 + 命令 |  |

### 5.3 REQ → TC 追溯矩阵

当前 Phase 的每个 REQ 必须至少映射到一个 TC；一个 TC 可覆盖多个强相关 REQ，但不得让当前 Phase REQ 悬空。

| REQ-ID | 阶段 | 交付物形态 | TC-ID | 验证方式 | 通过标准 | 自动化 / 证据位置 | 状态 |
|---|---|---|---|---|---|---|---|
| REQ-001 | [P1] | Demo | TC-001 | 自动 + 手动 |  | `tests/...` / smoke 记录 | 待执行 |

建议状态枚举：

| 状态 | 含义 |
|---|---|
| `待设计` | 仅有 REQ，尚未定义 TC；当前 Phase 不应长期保留 |
| `待执行` | TC 已定义但尚未执行 |
| `自动化已覆盖` | 已有自动化测试覆盖，但尚未完成人工 / 验收确认 |
| `待人工验收` | 自动化或实现已就绪，等待人工 smoke / 验收 |
| `通过` | 已按 09 记录证据并通过 |
| `条件通过` | 通过需附带 Mock / 降级 / 风险接受口径 |
| `失败` | 已执行但不满足通过标准 |
| `阻塞` | 环境、依赖、账号、资源或设计缺口导致无法执行 |
| `后续阶段` | 非当前 Phase 必过项 |

### 5.4 TC 用例详情

每个当前 Phase TC 应至少包含：

```markdown
### TC-001：<用例名称>

- 覆盖 REQ：REQ-...
- 验证类型：单元 / 集成 / API / 浏览器 smoke / 人工验收 / 资源验证 / 回归
- 前置条件：账号、权限、数据夹具、服务、浏览器、外部服务、Mock 开关
- 测试步骤：可复现步骤，避免只写一句验收目标
- 输入数据：文件、请求体、关键词、问题、示例账号、测试文档
- 期望结果：页面 / API / 日志 / 数据库 / 资源指标的可观察结果
- 失败 / 异常验证：权限拒绝、空结果、外部服务不可用、降级提示等
- 自动化位置：测试文件、命令、CI job；无自动化则写“人工验收”并说明原因
- 证据记录：日志、截图、报告路径或验收记录行
```

### 5.5 Mock / 降级 / 外部服务验收口径

凡涉及 Mock、降级、外部服务或重资源能力，`09` 必须明确：

1. 当前交付物形态是否允许降级通过。
2. 降级通过是否等价于真实能力通过；通常不得等价。
3. 用户可见提示或返回口径是什么。
4. 需记录哪些证据证明降级路径被触发且未越界。
5. 升级到 MVP / 产品前需要补哪些真实验证。

建议表格：

| 能力 | 当前 Phase 口径 | 是否允许 Mock / 降级 | 通过标准 | 不得宣称 | 升阶段补验 |
|---|---|---|---|---|---|
| OCR | Demo | 是 | 已提取文本可入库并可检索，页面 / API 明确降级 | 不宣称真实 OCR 已通过 | 接真实 OCR 引擎、质量样本、资源占用 |

### 5.6 UI / 浏览器 smoke 要求

当项目触发前端交互设计时，`09` 必须包含或引用浏览器 smoke TC，至少覆盖：

1. 支持浏览器与分辨率范围。
2. 登录 / 权限 / 空间或租户切换。
3. 当前 Sprint 涉及的主流程点击路径。
4. 加载态、空态、错误态、禁用态和成功反馈。
5. 权限可见性只作为 UI 表现，不替代后端权限验证。
6. 与 `docs/design/frontend-interaction.md` 页面清单、API 依赖和验收路径一致。

### 5.7 Sprint / Phase 验收记录

验收记录表建议最少包含：

| 日期 | 范围 | 执行人 | 关联 TC | 结果 | 验证证据 | 失败项 / 后续任务 |
|---|---|---|---|---|---|---|
| YYYY-MM-DD | Sprint-X / Phase-Y |  | TC-... | 通过 / 条件通过 / 未通过 | 命令 / 日志 / 截图 / 报告 |  |

Sprint 验收包建议最少包含：

| Sprint / Task | 关联 REQ | 关联 TC | 改动文件 | 验证命令 / 人工步骤 | 结果 | 残留风险 |
|---|---|---|---|---|---|---|

缺陷与回归记录建议最少包含：

| 缺陷 ID | 来源 / 现象 | 修复 Task | 回归范围 | 回归结果 | 后续动作 |
|---|---|---|---|---|---|

### 5.8 风险与待确认项

风险项应编号，避免只写散点说明。

| 编号 | 未验证项 / 风险 | 原因 | 影响 | 后续计划 | 是否阻塞 |
|---|---|---|---|---|---|
| VR-001 |  |  |  |  | 是 / 否 |

待人工确认项应继续使用模板统一结构：

| ID | 待确认项 | AI 建议 | 建议依据 | 备选方案 | 取舍影响 / 阻塞关系 |
|---|---|---|---|---|---|

## 6. Prompt 增强建议

### 6.1 `generate-docs`

生成 09 时应额外要求：

- 为当前 Phase 每个 REQ 生成稳定 `TC-ID`。
- `TC-ID` 应能回到 REQ、Sprint / Task 和测试等级。
- 当前 Phase 至少生成矩阵 + 用例详情骨架；后续 Phase 只保留骨架且标 `后续阶段`。
- 对 Mock / 降级 / 外部服务能力生成单独口径表。
- UI 型项目生成浏览器 smoke TC 或明确引用 `docs/design/frontend-interaction.md`。

### 6.2 `edit-single-doc`

修订 09 时应提示 AI 做以下检查：

- 是否存在当前 Phase REQ 无 TC。
- TC 是否只有自然语言目标、缺少步骤和期望结果。
- 状态是否与实现事实、自动化测试和验收记录一致。
- Mock / 降级能力是否被写成真实能力通过。
- 验收记录是否有证据而不是一句“通过”。

### 6.3 `docs-evaluation`

评估 09 时建议增加一个硬性判定问题：

> 这份 09 文档是否能让另一个 AI / 人类在无聊天上下文情况下复现当前 Sprint 或 Phase 验收？

若答案是否，通常至少为 `Conditional Go`；若当前 Phase REQ 缺少 TC 或 Mock / 降级口径不清且影响验收，应为 `No Go` 或带 P0 条件的 `Conditional Go`。

### 6.4 `sprint-summary`

Sprint 总结后应提示：

- 将通过 / 失败 / 条件通过的 TC 回填到 09 验收记录。
- 若用户暂不允许落盘，写入续接文件并明确“尚未成为正式验收记录”。
- 失败项必须回到 08 / task / bug 流程，不得在总结中宣称完成。

## 7. 兼容与迁移建议

对已存在的派生项目，不建议强制一次性重写整个 09。建议采用渐进迁移：

1. 保留现有 REQ → 用例矩阵内容。
2. 为当前 Phase REQ 补 `TC-ID`、验证方式、通过标准和状态。
3. 先补当前正在执行 Sprint 的 TC 详情和验收记录结构。
4. 再补测试等级矩阵、资源验证表、Mock / 降级口径表。
5. 历史 Sprint 若没有完整证据，可标“历史完成，证据见 Git / PR / 续接记录；未补录截图”，不得伪造验收证据。

## 8. 验收标准

本提案落地后，模板应满足：

1. 新项目生成的 `docs/09-verification.md` 默认包含 TC-ID、测试等级矩阵、用例详情、资源验证、验收记录、风险和待确认项结构。
2. `docs-evaluation` 对只有自然语言验收句、缺少 TC 详情的 09 文档能稳定给出 `Conditional Go` 或 `No Go`。
3. `run-dev-task` / `sprint-summary` 能引用 09 中的 TC-ID，而不是只引用 REQ。
4. UI 型项目能从 09 中找到浏览器 smoke 验收路径，且不把前端隐藏入口当成权限边界。
5. Mock / 降级能力不会被模板引导写成真实能力已通过。
6. 已有派生项目可按最小迁移步骤补齐当前 Phase，不需要重写全部历史文档。

## 9. 版本影响

建议版本：模板方法论 minor 版本升级。

影响面：

- 对新项目：生成的 09 文档更厚，但验收可执行性更强。
- 对旧项目：同步后不会自动修改项目事实文档；需要通过 `docs-system-audit`、`docs-evaluation` 或 `edit-single-doc` 按需迁移。
- 对 Prompt：会增加 09 生成 / 修订 / 评估时的检查项，但不改变文档驱动开发主链路。
- 对脚本：原则上无需脚本变更；如模板检查未来加入 09 结构校验，可另开提案。

## 10. 非目标

- 不要求所有项目引入完整 E2E 自动化框架。
- 不要求 Lean / 小工具项目编写冗长测试计划；Lean 项目仍可使用最小 TC，但必须可复现。
- 不替代 `docs/05-tech-spec.md` 的技术选型和资源评估，也不替代 `docs/env/local-env.md` 的环境事实采集。
- 不把派生项目的具体业务 REQ、账号、文件名或测试数据写入模板。

## 11. 待模板维护者确认

| ID | 待确认项 | AI 建议 | 建议依据 | 备选方案 | 取舍影响 / 阻塞关系 |
|---|---|---|---|---|---|
| C-001 | 是否将 `TC-ID` 作为 09 当前 Phase 必填字段 | 建议必填 | 无 TC-ID 时 Sprint、测试、验收记录无法稳定追溯 | 仅推荐不强制 | 不强制会继续出现“REQ 有验收句但无法执行”的问题 |
| C-002 | 是否把 Mock / 降级口径表设为条件必填 | 建议涉及 Mock / 降级 / 外部服务时必填 | 可防止 Demo 降级被误写成真实能力通过 | 放入风险项即可 | 风险项容易遗漏“不等价真实通过”的约束 |
| C-003 | 是否要求 UI 型项目在 09 中包含浏览器 smoke TC | 建议触发前端交互设计时必填或引用专门交互设计验收路径 | UI 验收依赖点击路径和页面状态，仅靠 API 测试不足 | 只在 `frontend-interaction` 中写 | 09 无入口会导致 Sprint / Phase 验收无法集中追溯 |
| C-004 | 是否将 09 状态枚举统一到模板规则 | 建议统一最小枚举，项目可扩展 | 多项目状态口径不同会影响 AI 续接和审计判断 | 保持自由文本 | 自由文本灵活但不利于自动评估 |
