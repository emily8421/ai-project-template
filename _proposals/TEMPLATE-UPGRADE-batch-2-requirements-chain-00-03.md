# TEMPLATE-UPGRADE: 00-03 需求链文档规范、健康度与状态传播

> Proposal status: Draft for review
> Proposal batch: Batch 2
> Related issues: #105, #112
> Source mirrors: `_proposals/_remote-issues/issue-105.md`, `_proposals/_remote-issues/issue-112.md`

## 1. 背景与问题

`00-scenario`、`01-user-requirements`、`02-srs`、`03-prd` 是文档驱动开发的需求链入口。现有模板已经具备基础骨架，但从派生项目回流 issue 看，需求链仍存在以下通用缺口：

1. `00` 容易停留在背景叙述，场景边界、非目标、来源锚点和下游影响不够稳定。
2. `01` 容易列出 U-ID，但用户操作流、用户可观察验收口径、优先级依据和 U-ID → REQ 追溯不足。
3. `02` 容易只列功能需求，NFR、约束假设、异常 / 边界场景和验证入口不足。
4. `03` 容易有 Phase 路线图，但进入 / 退出标准、功能范围 + 交付物形态、取舍依据和非目标追溯不够清晰。
5. 需求链状态、阶段标签、来源锚点和变更传播规则不统一，后续 `04-09`、`tasks/*` 和实现前门禁难以判断“当前 Phase 已批准做什么”。
6. 结构化待人工确认项虽已有基线，但在需求链中仍可能退化为一句“无新增确认项”，缺少关闭、回填和影响范围说明。

这些问题不依赖特定业务项目。凡是使用 `Scenario → 用户需求 → SRS → PRD → 设计 → 计划 → 验证 → 代码` 链路的派生项目，都需要更明确的 `00-03` 需求链规范和审计口径。

## 2. 设计目标

Batch 2 目标是把 #105 的“00-03 文档规范细化”和 #112 的“需求链健康度、状态传播与兼容补齐”合并为一组可实施的模板优化，确保模板能稳定支撑以下链路：

```text
上游输入 / 愿景 / 人工确认
  → 00 场景 / 边界 / 来源锚点
  → 01 U-ID / 用户操作流 / 用户验收口径
  → 02 REQ-ID / NFR / 约束 / 异常场景
  → 03 Phase / 优先级 / 非目标 / 进入退出标准
  → 04-09 / docs/design / tasks / tests
```

具体目标：

1. 每个场景能追溯到输入材料、人工确认或历史系统事实。
2. 每个 U-ID 能追溯到场景，并能映射到一个或多个 REQ-ID。
3. 每个当前 Phase REQ 都有可验证口径；远期 REQ 只保留骨架并标明待细化。
4. 每个 Phase 同时声明功能范围、交付物形态、进入标准、退出标准和状态。
5. 非目标、约束、假设、异常场景能传递到架构、技术方案、验证计划和任务拆分。
6. 待人工确认项以结构化表格保留 AI 建议、依据、备选方案、影响和阻塞关系，并在确认后回填权威文档。
7. 文档评估能输出 `SC → U-ID → REQ-ID → Phase → AC / TC` 需求链健康度矩阵。

## 4. 建议新增或细化的 00-03 规范

### 4.1 `00-scenario` 最低要求

建议在文档标准中明确 `00` 不只是背景摘要，至少应包含：

| 能力 | 最低字段 / 结构 |
|---|---|
| 文档元信息 | 输入来源、入口模式、文档剖面、当前状态、最后更新 |
| 目标用户与角色 | 角色 ID、角色、目标、痛点、成功标准、来源锚点 |
| 典型场景 | 场景 ID、参与角色、触发条件、场景叙事、期望结果、来源锚点 |
| 场景边界与非目标 | 做什么、不做什么、适用阶段、原因 / 来源 |
| 上游来源映射 | 场景 / 角色、来源文件或人工输入、来源锚点、可信度、备注 |
| 下游影响 | 预计影响的 U-ID、REQ、设计、验证或任务入口 |
| 待人工确认项 | 结构化确认项表，不得只写问题列表 |

### 4.2 `01-user-requirements` 最低要求

建议在文档标准中明确 `01` 承接 `00`，并输出用户视角的可观察结果：

| 能力 | 最低字段 / 结构 |
|---|---|
| 用户需求总览 | U-ID、用户 / 角色、用户需求、来源场景、价值、初步优先级、状态 |
| 用户操作流 | Flow ID、参与角色、前置条件、步骤、成功路径、失败 / 中断路径 |
| 用户验收口径 | U-ID、用户可观察验收口径、不满足示例、备注 |
| 优先级与阶段建议 | U-ID、优先级、建议阶段、理由、待确认项 |
| 非目标与排除需求 | 编号、内容、处理方式、原因 / 来源 |
| 追溯矩阵 | U-ID、来源场景 / 输入、来源锚点、下游 REQ、状态 |

### 4.3 `02-srs` 最低要求

建议在文档标准中明确 `02` 是系统需求事实源，至少覆盖：

| 能力 | 最低字段 / 结构 |
|---|---|
| 功能需求 | REQ-ID、系统需求、来源 U-ID、可验证口径、初步阶段、状态 |
| 非功能需求 | NFR-ID、类型、需求描述、来源、验证方式、状态 |
| 约束与假设 | 编号、类型、内容、状态、影响范围 |
| 边界条件与异常场景 | 编号、关联 REQ、条件、期望行为、验证提示 |
| U-ID → REQ-ID 追溯 | U-ID、REQ-ID、覆盖说明、是否完整覆盖、备注 |
| 验证入口 | 当前 Phase REQ 应能导向 `09` AC / TC 或至少保留验证提示 |

### 4.4 `03-prd` 最低要求

建议在文档标准中明确 `03` 是 Phase 范围和产品取舍事实源：

| 能力 | 最低字段 / 结构 |
|---|---|
| 产品目标与范围 | 当前 Phase 目标、交付物形态、范围边界 |
| 功能清单 | 功能 ID、功能描述、覆盖 REQ、阶段、交付物形态、状态 |
| Phase 路线图 | Phase、功能范围、交付物形态、进入标准、退出标准、状态 |
| 优先级与取舍 | 功能 / REQ、优先级、是否进入当前 Phase、理由、风险 |
| 非目标 | 编号、非目标、适用范围、来源、备注 |
| REQ 覆盖矩阵 | REQ-ID、对应功能、阶段、交付物形态、覆盖状态 |
| 证据 / 验收引用 | 推荐列，用于链接 `09` 验收记录、Sprint 总结或人工确认证据 |

## 5. 需求链健康度矩阵

建议在 `docs-evaluation`、`docs-system-audit`、`phase-upgrade` 中增加可选输出：

| 链路节点 | 检查项 | 常见断点 | 建议处理 |
|---|---|---|---|
| 输入 → SC | 场景是否有来源锚点 | 场景来自 AI 推断但未标待确认 | 补来源或列入待确认 |
| SC → U-ID | 每个关键场景是否有用户需求 | 场景存在但没有 U-ID | 补 `01` 或标为非目标 |
| U-ID → REQ-ID | 用户需求是否被系统需求覆盖 | U-ID 悬空 | 补 `02` 或调整 `01` |
| REQ-ID → Phase | 当前 Phase 是否声明范围 | REQ 有但未进路线图 | 补 `03` 阶段归属 |
| REQ / Phase → AC / TC | 当前 Phase 是否可验证 | Phase 已确认但无验收口径 | 补 `09` 或验收提示 |
| 非目标 → 下游 | 非目标是否被设计 / 计划尊重 | 后续设计重新引入非目标 | 回写 `03` 并调整下游 |
| 待确认 → 状态 | 确认项是否有建议、阻塞关系和关闭状态 | 只列问题，无影响说明 | 结构化补齐并回填权威文档 |

默认策略：

- 正式 `docs/00-03` 保持轻量骨架。
- 健康度矩阵优先写入评估 / 审计报告，必要时再回写到 `03`、`09` 或 `docs/research/`。
- Lean 项目可只输出 P0 / P1 断点，不强制填满完整矩阵。

## 6. 状态传播与兼容补齐规则

建议补充以下通用规则：

1. `03-prd` 当前 Phase 状态不得由 AI 自动从“建议”改成“已确认”；必须有人工确认、验收记录或项目事实依据。
2. 当 `00-03` 任一上游事实发生变化时，AI 应检查下游 `04-09`、`docs/design/*`、`tasks/*` 和 `tests` 的影响，并列出是否阻塞当前 Sprint / Phase。
3. 当现有项目缺少 00-03 某些标准章节时，AI 不应一次性重写全部文档；应先输出兼容补齐计划，按最小变更补齐影响当前阶段的断点。
4. 旧项目若只有历史自由文本需求，可先建立来源锚点与 ID 映射，再逐步补 U-ID / REQ-ID / Phase 矩阵。
5. 对当前 Phase 有影响的待确认项必须标明阻塞关系；不阻塞项可进入后续优化清单。
6. 如果 `project-rules.md` 当前阶段与 `03-prd` 冲突，应先报告冲突并请人工确认权威来源，不得擅自选择。

## 8. 版本影响

建议作为 MINOR 版本处理。

理由：

- 新增 / 恢复 `ai/doc-standards/00-03` 标准文件，影响文档生成、审计和评估基线。
- 增强需求链追溯、健康度矩阵、状态传播和兼容补齐规则。
- 会改变 AI 对 `00-03`、阶段转换和编码前门禁的默认判断方式。

## 9. 验收口径

1. `ai/doc-standards/00-03` 存在，并分别覆盖来源锚点、U-ID、REQ-ID、Phase、非目标、待确认项和下游影响。
2. `docs/00-03` 模板骨架与标准文件保持一致，且不引入项目专属事实。
3. `ai/document-lifecycle-rules.md` 明确 00-03 最小追溯、状态传播和兼容补齐规则。
4. `generate-docs` 生成前能要求建立来源锚点、U-ID、REQ-ID、Phase、非目标和待确认项结构。
5. `docs-system-audit` 和 `docs-evaluation` 能输出或检查 `SC → U-ID → REQ-ID → Phase → AC / TC` 健康度矩阵。
6. `phase-upgrade` 能检查 Phase 状态、证据引用和当前阶段权威来源。
7. 自检脚本能防止关键 00-03 标准和追溯 / 状态传播规则被误删。
8. 合并后关闭 #105、#112，并在关闭说明中说明 #105 与 #112 的合并处理方式。

## 10. 非目标

- 不落地 `04-05` 架构与技术方案文档规范；该范围留给 Batch 3（#106、#113）。
- 不落地 `06-07` DB / API 契约规范；该范围留给 Batch 4（#107、#114）。
- 不落地 `08-09` 开发计划与验证证据规范；该范围留给 Batch 5（#108、#109、#115）。
- 不落地 `docs/design/*` 详细设计标准；该范围留给 Batch 6（#110、#116）。
- 不重复定义 Batch 1 的横切状态词典，只引用其状态和待确认项基线。
- 不重写旧派生项目文档；只提供兼容补齐策略。
