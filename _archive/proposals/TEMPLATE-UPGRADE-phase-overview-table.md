# TEMPLATE-UPGRADE — 阶段路线图推荐「双维度总览表」

> 去项目化提案，目标回流到 `ai-project-template` 模板仓库。不写入任何具体项目业务、账号或路径。

## 动机

`global-rules §8.1` 要求阶段双维度（功能范围 `[P1]/[P2]/[愿景]` + 交付物形态 `Demo/MVP/产品`）在 `docs/03-prd.md §3` 路线图声明。但实际派生项目中两个维度的「可见性」严重不对等：

- **功能范围标签**是「要素级」标签——每个架构模块 / 数据表 / 接口 / Sprint 都标 `[P1]/[P2]/[愿景]`，遍布 `04-09` + `docs/design/*`，数量级可达上百次。
- **交付物形态**是「阶段级」属性——只在 `03 §3` 等少数阶段声明点出现，十几次。

结果：交付物形态（Demo/MVP/产品）这条演进线**被海量的功能范围标签淹没**，读者翻文档时 `[P1]/[P2]/[愿景]` 扑面而来，Demo/MVP/产品 不易一眼把握；甚至误以为项目阶段是单维度（只有 P1/P2/愿景），质疑「交付物形态是否明确」。实测：某派生项目 `03 §3` 已完整声明三阶段双维度，仍被读者认为「没有明确 Demo/MVP/产品 全景」。

## 拟改

在 `doc-standards/03-prd.md`（PRD 撰写规范镜像）与/或 `global-rules §8.1` 增加「阶段双维度总览表」推荐：

- `03 §3` 路线图**顶部**用一张表集中呈现双维度，列至少含：阶段 / 功能范围 / 交付物形态 / 状态 / 进入标准 / 退出标准。
- 表下方保留各阶段 `###` 子节展开详情（目标 / 功能范围 / 交付物形态释义 / 演示场景 / 进入·退出标准）。
- 总览表让 Demo→MVP→产品 一眼可见，与遍布全文的要素级 `[P1]` 标签形成「全景 ↔ 要素」对照，避免交付物形态被淹没。
- 可选协同：根 `README.md`「当前状态」给一行 Demo→MVP→产品 演进全景，指向 `03 §3`。

## 版本

建议纳入下一个模板小版本（具体版本号由模板维护者定）。仅新增一项撰写推荐，不改变双维度定义与既有要求。

## 影响

- 受影响：所有派生项目的 `docs/03-prd.md`（`doc-standards/03` 随 `sync-template` 刷新）。
- 兼容性：非破坏性。新增推荐结构；既有「`03 §3` 每 Phase 声明双维度」要求不变，总览表是其可视化增强。
- 实现成本：低（规范镜像补一张表模板）。

## 待模板维护者确认

- 总览表是「推荐」还是「强制」：本提案建议**推荐**（撰写规范级），避免对极简 Lean 剖面项目造成负担。
- 最小列集：本提案给 6 列（阶段 / 功能范围 / 交付物形态 / 状态 / 进入 / 退出），Lean 剖面可裁剪。
