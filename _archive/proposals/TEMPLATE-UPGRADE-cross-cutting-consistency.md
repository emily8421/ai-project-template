# TEMPLATE-UPGRADE: 文档横切一致性规则（Document Lifecycle 规则补充）

> 类型：派生项目起草的模板优化提案（去项目化）。
> 状态：待回流 `ai-project-template`（按 `CONTRIBUTING.md` §4 上行流程）。
> 关联：补充《TEMPLATE-UPGRADE: Document Lifecycle Generation Rules》（另一派生项目提案，下称 **lifecycle 提案**）。

## 0. 与 lifecycle 提案的关系

本提案**认同并依附** lifecycle 提案，不重复其内容（阶段模型 / 多入口 / 文档剖面 / 生成矩阵 / 纵向追溯链 / 变更传播 / 生成前后声明 / 禁止项）。本提案是其**补充**，聚焦三个该提案未充分覆盖、且与纵向模型**正交**的缺口，均来自派生项目实证。

若两份提案合并落地，本提案内容宜并入 lifecycle 提案拟新增的 `ai/document-lifecycle-rules.md` 的「变更传播」与「文档接入」章节，并对 `INIT-PROMPT.md §4` 做一处措辞修正。

## 1. 动机

lifecycle 提案建立了「文档上下游」的**纵向**追溯与传播，但文档体系还有两类纵向模型覆盖不了的一致性需求：

- **横向事实一致性**：同一事实/裁决（如某平台能力的合规结论、某技术的合限定性）会被多个文档引用。纵向传播管不了「同一事实多处声明不同步」。
- **约束的动态事件性**：lifecycle 提案把约束当作「上游输入包」（静态前置）。但合规结论、平台能力、合规裁决常在**项目运行中**才出现，是**事件**，不是初始输入。

派生项目（数字客服 Demo）实证三症状（均非人为疏忽，而是规则缺环节）：

1. **横切变更被 §4 抑制**：技术核实「企微客户群内自动回复不成立」后，仅 `02-srs` 的 REQ-15 更新，`01`(F15b 标签)、`03`(§2 优先级)、`vision`(文首 ⚠️) 漏改。根因是 `INIT-PROMPT §4` 第 3 条明确「不要自行连带修改」——局部变更时对，横切事实变更时**主动制造遗漏**。
2. **事实无权威源导致分裂**：某技术路径（个人微信灰路径）的合限定性，在 `project-rules §2`、`03 §4`、`04`、`05`、`DEC-7`、外部策略文档等 7+ 处各自声明，分裂成「永久禁止」与「非商用验证 Spike」两阵营，**同一文件内**（`08-dev-plan` 33 行 vs 407 行）就打架。
3. **外部文档接入无规则**：咨询方撰写的外部策略文档（用外部命名体系、未锚定到本仓库 docs）变成**无引用孤岛**，其结论（如「愿景 ⚠️ 该更新」）也无人落实。

## 2. 拟改（增量）

### 2.1 变更传播：区分「局部变更」与「横切事实变更」，并修正 §4 反模式

lifecycle 提案 §2.7 已建变更传播规则，但未区分变更性质，且未处理与现有 `INIT-PROMPT §4` 的冲突。建议：

**规则**：变更分两类，传播义务不同——

| 变更类型 | 定义 | 传播义务 |
|---|---|---|
| 局部变更 | 单一需求/契约/字段的增删改，影响有限 | 沿用 §4：声明影响、待确认再连带，防范围蔓延 |
| 横切事实变更 | 一个**事实/结论/裁决**（合规、平台能力、可行性核实、合限定性）变化，潜在影响多个不相邻文档 | **强制连带回梳**：AI 必须主动列出受影响文档清单并逐项更新，不得「等确认」 |

**修正 §4**：`INIT-PROMPT §4` 第 3 条「不要自行连带修改」追加例外——「**横切事实/约束变更除外：此时必须主动列出并更新所有受影响文档**」。否则 lifecycle 提案的变更传播规则与 §4 落地冲突（AI 不知该听「传播」还是「不连带」）。

### 2.2 事实的单一权威源（SSOT）+ 引用同步（第二种传播模式）

lifecycle 提案的追溯/传播是**纵向**（vision→00→…→code）。本提案补充**横向**事实同步，作为并列的第二种传播模式。

**规则**：
- 事实/裁决/合限定性（平台能力结论、合规裁决、技术选型禁令、可行性核实结果）必须有**唯一权威定义点**——决策记录（`docs/decisions/`）或 `project-rules` 禁止/允许清单。
- 其他文档**引用**权威源（「见 DEC-X / project-rules §Y」），**不得各自重新声明**事实本身。
- **第二种传播模式**：权威源变更 → 必须检索（grep）所有引用处并同步，作为与「纵向文档传播」并列的强制动作。

> 实证：症状 2 若有 DEC 权威源 + 他处引用，分裂不会发生；裁决变更只需改 DEC + 同步引用。

### 2.3 外部文档接入锚定规则

lifecycle 提案 §2.3「上游输入包」与 Existing-system 入口均未覆盖「项目运行中从外部带入的策略/调研/决策/会议文档」。

**规则**：运行中接入的外部文档（非从本仓库 docs 生成的输入）必须：

1. **顶部锚定说明**：外部文档命名 → 本仓库 docs 的映射表（如外部《用户需求说明书 V1.1》→ `docs/01-user-requirements.md》）。
2. **定位声明**：文档类型（策略/调研/决策/会议）+ 它驱动/约束哪些 docs（如「驱动 DEC-1 通道选型 + 05 技术方案」）。
3. **纳入追溯链**：其结论若影响 00-09 或 project-rules，必须触发回梳（接 2.1 横切变更）；不得成为无引用孤岛。
4. **分区归属**：按 `docs/README` 归类——策略决策类进 `docs/decisions/`、调研进 `docs/research/`，**不放 `docs/vision/`**（vision 区专属叙事源文档）。

### 2.4 横切变更后的强制一致性验证（变更传播的闭环）

`INIT-PROMPT §10` checklist 仅在「生成 03-09 后」触发。症状 1 中技术核实后未触发任何跨文档检查，致遗漏存活。

**规则**：横切事实/约束变更完成回梳后，**强制运行一次聚焦的跨文档一致性检查**（针对变更影响范围，复用 §10 C 段「交叉一致性」口径），作为变更传播的闭环。输出：受影响文档清单 + 各项一致性核对结果 + 残留待确认项。

## 5. 迁移建议（派生项目实证）

派生项目（数字客服 Demo）可作为本规则的**首个验证案例**，用三症状回梳：

- 症状 1（sprint-0 核实遗留）：按 2.1 横切回梳，补齐 `01`(F15b 标签) / `03`(§2 优先级) / `vision`(文首 ⚠️)；
- 症状 2（合限定性分裂）：按 2.2 立 DEC 权威源 + 同步 7+ 引用处；
- 症状 3（外部策略文档孤岛）：按 2.3 加锚定说明 + 移出 `vision/`。

## 6. 验收口径

- 横切事实变更后，AI 主动列出受影响文档清单并逐项更新（不再因 §4 而漏改）。
- 任一事实/裁决有唯一权威源，他处引用而非声明；权威源变更触发引用同步。
- 外部接入文档带锚定说明 + 定位声明 + 纳入追溯链，无孤岛。
- 横切变更后输出聚焦一致性检查结果（变更传播闭环）。
- §4 与变更传播规则不再冲突。
