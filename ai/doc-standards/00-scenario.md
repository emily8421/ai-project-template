# 00 Scenario Standard（场景文档规范镜像）

> Sync notice: This file is maintained by `ai-project-template` and may be overwritten when a derived project syncs template methodology.
> Do not edit it directly in derived projects; propose reusable changes in `_proposals/` and upstream them to the template repository.

本文件是 `docs/00-scenario.md` 的细粒度规范标准，用于 AI 生成、精修、审计和评估场景文档。它不是项目事实文档，派生项目的实际场景事实仍写入 `docs/00-scenario.md`。

## 1. 定位与边界

`00` 是需求链入口，负责把输入材料、产品愿景或人工背景转成可追溯的场景、角色、边界和下游影响。它回答“为什么做、谁使用、在哪些场景发生”，不写系统需求、功能实现、技术方案或阶段计划。

- 不得凭空新增无来源场景；无法确认的内容必须标为“待人工确认”。
- 不得把愿景能力直接写成当前 Phase 功能；阶段归属由 `03` 决定。
- 不得把非目标遗漏给下游，避免后续设计 / 计划重新引入被排除内容。
- 必须能导向 `01` 的 U-ID 或明确标为非目标 / 待确认。

## 2. 最低结构

| 能力 | 最低字段 / 结构 |
|---|---|
| 文档元信息 | 输入来源、入口模式、文档剖面、当前状态、最后更新 |
| 背景与问题 | 背景、当前痛点、业务机会、现有流程问题、来源锚点 |
| 目标用户与角色 | 角色 ID、角色、目标、痛点、成功标准、来源锚点 |
| 典型场景 | 场景 ID、参与角色、触发条件、场景叙事、期望结果、来源锚点 |
| 场景边界与非目标 | 做什么、不做什么、适用阶段、原因 / 来源 |
| 上游来源映射 | 场景 / 角色、来源文件或人工输入、来源锚点、可信度、备注 |
| 下游影响 | 预计影响的 U-ID、REQ、设计、验证或任务入口 |
| 待人工确认项 | 结构化确认项表，不得只写问题列表 |

## 3. ID 与追溯规则

- 场景使用 `SC-001`，角色使用 `R-001`，来源锚点可引用 `docs/inputs/*`、`docs/vision/*`、会议纪要或当前用户确认。
- 最低链路：`输入来源 → SC-ID / R-ID → U-ID`。
- 每个进入 `01` 的关键场景必须有来源锚点；没有来源但有价值的内容只能作为待确认项。
- 非目标必须能被 `01-03`、`04-09` 和 Sprint 禁止事项识别。

## 4. 审计检查

| 检查项 | 通过标准 | 常见问题 |
|---|---|---|
| 来源可信 | 每个关键场景有来源或待确认状态 | AI 自行补场景但未标待确认 |
| 角色清晰 | 角色目标、痛点和成功标准可观察 | 角色只是泛称“用户” |
| 边界明确 | 非目标和适用阶段写清楚 | 后续文档重新引入非目标 |
| 下游可追溯 | 场景能导向 U-ID 或说明不进入需求 | 场景存在但 01 无承接 |
| 状态准确 | 草案、待确认、已确认等状态不混用 | 待确认内容写成已确认事实 |

## 5. 下游影响

- 给 `01`：用户需求、操作流和用户验收口径来源。
- 给 `02`：系统需求的背景和边界依据。
- 给 `03`：产品范围、非目标和 Phase 取舍依据。
- 给审计 / 评估：需求链健康度的 `SC-ID` 起点。
