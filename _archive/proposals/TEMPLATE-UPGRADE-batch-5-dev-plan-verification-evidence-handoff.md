## 1. 背景与问题

`08-dev-plan`、`09-verification` 与 `tasks/*` 承接 `03` Phase、`04-07` 设计 / 契约和实现生命周期，是“能否开始编码、如何验收、如何回写事实”的执行基线。现有模板已经要求 Sprint 写目标、输入文档、修改范围、验收标准和禁止事项，并要求 `09` 包含验证策略、REQ → 用例追溯、资源验证、验收记录和风险项，但从派生项目回流 issue 看，`08-09` 仍存在以下通用缺口：

1. `08` 有 Sprint 详情，但缺少当前 Phase 目标、退出标准和禁止越界的集中声明。
2. `08` 缺少 Sprint 总览表，无法快速查看每个 Sprint 的 REQ、修改范围、验证包、状态和依赖。
3. Sprint 输入文档常只列文件名，未精确到 REQ、设计章节、API-ID、表名、TC-ID。
4. Sprint 修改范围常写模块名，缺少“是否超过 3 个文件 / 模块、是否应拆 task、是否需要 worktree”的判断。
5. Sprint 验收标准常是自然语言，未强制连接 `09` TC-ID、测试等级、验证命令、人工步骤和资源验证。
6. `tasks/*` 任务单模板缺少 Task-ID、所属 Sprint、关联 REQ / TC、任务状态、验证命令、完成记录和残留风险。
7. `09` 的 REQ → 用例矩阵容易只写自然语言验收句，缺少稳定 TC-ID、验证方式、通过标准、自动化位置和状态枚举。
8. `09` 缺少用例详情约束，浏览器 smoke、API 集成、资源验证和人工验收步骤容易留在会话上下文或本地 handoff。
9. Mock / 降级能力的验收口径不清，容易把“Demo 可降级演示”误写成“真实能力已通过”。
10. Sprint 完成事实容易散落在 Git、PR、续接文件和聊天中，正式 `08` / `09` 不易审计。
11. Phase 升级后，`03`、`08`、`09`、`ai/project-rules.md`、README 和 handoff 之间的状态可能不一致。

这些问题不依赖具体业务项目。凡是采用文档驱动开发并按 Sprint / Task 小步实现、验证和验收的派生项目，都需要更细的 `08-09` 计划 / 证据 / 回写规范。

## 2. 设计目标

Batch 5 目标是把 #108 的“08 开发计划、任务拆分与进度留痕”、#109 的“09 Verification 文档规范细化”和 #115 的“08-09 进度回写与验证证据规范”合并为一组可实施的模板优化，确保模板能稳定支撑以下链路：

```text
03 Phase / REQ / 退出标准
  → 04-07 设计 / 技术 / DB / API 契约
  → 09 TC / 验证计划
  → 08 Sprint 总览 / Sprint 详情 / Task 拆分
  → code / tests / Git commit / PR
  → 09 验收记录 / Sprint 完成包 / 残留风险
  → 03 / 08 / 09 / project-rules / README 状态回写
```

具体目标：

1. 当前 Phase 的功能范围、交付物形态、退出标准和禁止事项在 `08` 一眼可见。
2. 每个 Sprint 能追溯到 REQ、设计章节、API-ID / 表名、TC-ID 和验证方式。
3. 复杂 Sprint 能按规则拆成 `tasks/task-*.md`，并记录拆分原因和任务状态。
4. 每个 Sprint 有验证包，明确测试等级、命令 / 人工步骤、资源验证和不适用原因。
5. 当前 Phase 的每个 REQ 至少映射到一个可执行、可复现、可记录证据的 TC。
6. 每个 Sprint / Task 完成时形成最小完成包，包含改动文件、验证结果、验收记录、残留风险和下一步。
7. Mock / 降级任务能明确“不等价真实能力完成”的边界和后续补验时点。
8. handoff 只记录临时状态，长期进度和验收事实必须回填正式 `08` / `09` 或说明暂不落盘原因。
9. Phase 升级前能检查 `03`、`08`、`09`、`ai/project-rules.md`、README 和 handoff 的状态一致性。

## 4. 建议新增或细化的 08 / tasks 规范

### 4.1 `08-dev-plan` 最低要求

建议在文档标准中明确 `08` 是计划、拆分、进度和完成摘要事实源，至少应包含：

| 能力 | 最低字段 / 结构 |
|---|---|
| 文档元信息 | 当前 Phase、交付物形态、输入基线、当前状态、最后更新 |
| 当前 Phase 目标 | 当前 Phase、功能范围、交付物形态、退出标准、禁止越界、权威源 |
| Phase / Sprint / Task 定义 | Phase、Sprint、Task、完成包的定义和输出位置 |
| Sprint 总览 | Sprint-ID、目标、覆盖 REQ / NFR、输入设计 / 契约、修改范围、验证包、状态、任务单 |
| Sprint 详情 | 目标、输入文档、修改范围、验证包、验收标准、禁止事项、完成包 |
| 依赖关系与里程碑 | 项、前置依赖、阻塞风险、是否可并行、处理方式、里程碑 |
| 任务拆分规则 | 复杂度阈值、拆分触发条件、Task 文件命名、worktree / 分支建议 |
| 当前进度记录 | 日期、Sprint / Task、进度、验证结果、关联提交 / PR、下一步、是否已回填 09 |
| 待人工确认项 | 结构化确认项表，不得只写问题列表 |

### 4.2 Sprint 验证包

建议 `08` 中每个 Sprint 至少引用一个验证包：

| 字段 | 要求 |
|---|---|
| 关联 TC | 当前 Sprint 必过 TC-ID 或待新增 TC-ID |
| 测试等级 | 单元 / 集成 / API / E2E / 验收 / 回归 / 资源环境 |
| 自动化命令 | 具体命令；暂无自动化则写明人工步骤和原因 |
| 人工步骤 | 浏览器 smoke、CLI smoke、桌面端、权限、导入 / 搜索 / 问答等可复现步骤 |
| 资源 / 环境验证 | 端口、内存 / 显存、磁盘、Docker / DB / 模型、网络 / 权限 |
| Mock / 降级口径 | 是否等价真实能力、用户可见提示、补验时点 |
| 不适用说明 | 不适用的测试等级必须说明原因 |

### 4.3 Sprint 完成包

建议 `08` 记录进度摘要，`09` 记录验证证据。最小完成包字段：

| 字段 | `08` 职责 | `09` 职责 |
|---|---|---|
| 改动文件 | 摘要记录 | 可引用详情或 PR |
| 验证命令 / 人工步骤 | 引用验证包 | 记录实际执行命令 / 步骤 |
| 验证结果 | 摘要状态 | 记录证据、日志、截图、失败项 |
| 关联提交 / PR | 记录引用 | 记录引用 |
| 验收记录 | 指向 `09` | 权威记录 |
| 残留风险 | 摘要 | 风险与未验证项权威记录 |
| 下一步 | 计划入口 | 后续验证 / 回归入口 |

### 4.4 Task 模板最低要求

建议增强 `tasks/README.md` 或任务模板来源。当 Sprint 复杂到需要拆分时，`tasks/task-*.md` 至少包含：

```markdown
# task-00X-功能名

## 0. 任务元信息
## 1. 目标
## 2. 输入文档
## 3. 修改范围
## 4. 验证包
## 5. 验收标准
## 6. 禁止事项
## 7. 降级 / Mock 边界（如适用）
## 8. 完成记录
## 9. 待人工确认项
```

任务元信息建议字段：

| 项 | 内容 |
|---|---|
| Task-ID | TASK-00X |
| 所属 Sprint | Sprint-X |
| 关联 REQ / NFR | REQ-... / NFR-... |
| 关联 TC | TC-... |
| 当前状态 | 待开始 / 进行中 / 已完成 / 条件完成 / 阻塞 |
| 分支 / worktree | 如适用 |

降级 / Mock 边界建议字段：降级原因、当前实现、不等价真实能力、用户可见提示、后续补齐时点、对 09 验收影响。

## 5. 建议新增或细化的 09 规范

### 5.1 `09-verification` 最低要求

建议在文档标准中明确 `09` 是验证计划、证据和验收记录事实源，至少应包含：

| 能力 | 最低字段 / 结构 |
|---|---|
| 文档元信息 | 当前 Phase、交付物形态、覆盖 REQ、验证状态、最后更新、上游依据 |
| 验证策略 | 测试等级矩阵、证据形式、不适用原因 |
| REQ → TC 追溯 | REQ-ID、阶段、交付物形态、TC-ID、验证方式、通过标准、自动化 / 证据位置、状态 |
| TC 用例详情 | 覆盖 REQ、验证类型、前置条件、测试步骤、输入数据、期望结果、异常验证、自动化位置、证据记录 |
| 分阶段验证范围 | Phase、交付物形态、必须通过、不作为当前必过、退出标准 |
| Phase 测试大纲 | Phase、必过 TC、自动化命令、人工验收点、资源 / 环境验证、退出标准 |
| 本机资源与技术环境验证 | 启动、build、端口、Docker / DB / 模型、网络 / 权限、资源指标、后续动作 |
| Mock / 降级 / 外部服务验收口径 | 当前状态、是否等价真实能力、可接受条件、补验时点、风险接受方式 |
| Sprint / Phase 验收记录 | 日期、范围、执行人、结果、命令 / 步骤、验证证据、失败项 / 后续任务 |
| Sprint 验收包 | Sprint / Task、关联 REQ、关联 TC、改动文件、验证命令 / 人工步骤、结果、残留风险 |
| 缺陷与回归记录 | 缺陷 / 变更 ID、来源 / 现象、修复 Task、回归范围、回归命令 / 步骤、回归结果、后续动作 |
| 风险与未验证项 | 编号、未验证项 / 风险、原因、影响、后续计划 |
| 待人工确认项 | 结构化确认项表，不得把 AI 建议写成确认事实 |

### 5.2 TC 状态枚举

建议 `09` 使用以下 TC 状态：

| 状态 | 含义 |
|---|---|
| `待设计` | 仅有 REQ，尚未定义 TC；当前 Phase 不应长期保留 |
| `待执行` | TC 已定义但尚未执行 |
| `自动化已覆盖` | 已有自动化测试覆盖，但尚未完成人工 / 验收确认 |
| `待人工验收` | 自动化或实现已就绪，等待人工 smoke / 验收 |
| `通过` | 已按 09 记录证据并通过 |
| `条件通过` | 通过需附带 Mock / 降级 / 风险接受口径 |
| `失败` | 已执行但不满足通过标准 |
| `阻塞` | 环境、依赖、账号、资源或设计缺口导致无法执行 |
| `后续阶段` | 非当前 Phase 必过项 |

### 5.3 TC 用例详情最低字段

当前 Phase 的每个 TC 应至少包含：

```markdown
### TC-001：<用例名称>

- 覆盖 REQ：REQ-...
- 验证类型：单元 / 集成 / API / 浏览器 smoke / 人工验收 / 资源验证 / 回归
- 前置条件：账号、权限、数据夹具、服务、浏览器、外部服务、Mock 开关
- 测试步骤：可复现步骤，避免只写一句验收目标
- 输入数据：文件、请求体、关键词、问题、示例账号、测试文档
- 期望结果：页面 / API / 日志 / 数据库 / 资源指标的可观察结果
- 失败 / 异常验证：权限拒绝、空结果、外部服务不可用、降级提示等
- 自动化位置：测试文件、命令、CI job；无自动化则写“人工验收”并说明原因
- 证据记录：日志、截图、报告路径或验收记录行
```

## 6. 正式回写与 handoff 边界

建议补充以下规则：

1. `08` 管计划、拆分、进度摘要和完成摘要；`09` 管验证计划、证据、验收记录、缺陷 / 回归和未验证风险。
2. `.ai/session-handoff.md` 只用于本地会话续接，不替代正式 `08/09` 进度和验收记录。
3. Sprint 完成后，如存在验收事实、验证命令、截图 / 日志、残留风险或状态变化，应回填 `09` 或明确暂不落盘原因。
4. Sprint 完成后，如改变 Sprint 状态、下一步、依赖或任务拆分，应回填 `08` 或明确暂不落盘原因。
5. Phase 升级前，应检查 `03`、`08`、`09`、`ai/project-rules.md`、README 和 handoff 是否存在状态冲突。
6. 未经人工确认，AI 不得自动把 Phase 状态改为已完成、不得自动进入下一 Phase。

建议的回写清单：

| 触发 | 应检查 / 回写 |
|---|---|
| Sprint 完成 | `08` 状态 / 下一步、`09` 验收记录 / Sprint 验收包、残留风险 |
| Bug 修复 | `09` 缺陷与回归记录、相关 TC 状态、`08` 当前进度 |
| Mock / 降级验收 | `09` 条件通过 / 风险接受、`08` 后续补齐任务 |
| Phase 验收 | `03` Phase 状态、`08` Phase 完成情况、`09` Phase 验收、`project-rules` 当前阶段 |
| Phase 升级 | `project-rules`、`03/08/09`、README、handoff 状态一致性 |

## 8. 版本影响

建议作为 MINOR 版本处理。

理由：

- 新增 / 恢复 `ai/doc-standards/08-09` 标准文件，影响开发计划、验证计划、任务执行、Sprint 总结和 Phase 升级流程。
- 增强 Sprint 验证包、完成包、TC 详情、验收证据、正式回写和 handoff 边界。
- 会改变 AI 对“任务完成”“验证通过”“Phase 可升级”的默认判断方式。

## 9. 验收口径

1. `ai/doc-standards/08-09` 存在，并分别覆盖 Phase 目标、Sprint 总览、验证包、完成包、TC-ID、用例详情、验收记录、缺陷 / 回归和风险项。
2. `docs/08-09` 模板骨架与标准文件保持一致，且不引入项目专属事实。
3. `tasks/README.md` 包含任务元信息、验证包、降级 / Mock 边界、完成记录和待确认项。
4. `implementation-lifecycle-rules` 明确 Sprint 完成包、正式回写、Phase 升级前状态一致性检查和 handoff 边界。
5. `session-rules` 明确 handoff 不能替代正式 `08/09` 进度和验收记录。
6. `docs-checklist` 编码前能检查当前 Sprint 是否有验证包和相关 TC。
7. `run-dev-task` 执行任务前能引用 Sprint / Task / REQ / TC / 验证包，完成后能提示回写正式文档。
8. `sprint-summary` 能输出完成包，并提示回填 `08` / `09` 或说明暂不落盘原因。
9. `phase-upgrade` 能检查 `03/08/09/project-rules/README/handoff` 状态一致性。
10. 自检脚本能防止关键 08-09 标准、TC 详情、完成包和 handoff 边界规则被误删。
11. 合并后关闭 #108、#109、#115，并在关闭说明中说明三者的合并处理方式。

## 10. 非目标

- 不落地 `00-03` 需求链规范；该范围由 Batch 2（#105、#112）处理。
- 不落地 `04-05` 架构 / 技术方案风险验证；该范围由 Batch 3（#106、#113）处理。
- 不落地 `06-07` DB / API 契约规范；该范围由 Batch 4（#107、#114）处理。
- 不落地 `docs/design/*` 详细设计标准；该范围留给 Batch 6（#110、#116）。
- 不要求每个小改动都产生冗长验收报告。
- 不要求 Lean 小工具必须建立完整测试平台。
- 不把本地 handoff 纳入正式提交。
- 不允许 AI 未经人工确认自动修改 Phase 边界或进入下一阶段。

## 11. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 文档维护成本增加 | 提供最小完成包；复杂项目再扩展证据字段 |
| 08 与 09 职责重复 | 明确 `08` 管计划 / 进度摘要，`09` 管验证证据 / 验收记录 |
| AI 自动改 Phase 状态 | 要求人工确认 Phase 升级和正式文档修订 |
| 与 Batch 3 / 4 风险和契约重复 | Batch 5 只承接为验证包、TC、完成包和回写规则 |
| 本地 handoff 与正式 docs 混淆 | 明确 handoff 只做续接，长期事实写 `08/09` |
| 旧项目补齐成本高 | 允许从当前 Sprint 的最小完成包开始，历史证据不足时标明来源和缺口，不伪造证据 |
