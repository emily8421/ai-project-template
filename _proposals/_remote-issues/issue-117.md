# TEMPLATE-UPGRADE: Mock / 降级 / 权限安全 / 高风险能力横切状态规范

> 来源：zhiyan-digital-cs-platform（emily8421/zhiyan-digital-cs-platform）派生项目回流
> 类型：模板优化提案草稿
> 状态：Batch 6 横切归并审计后起草，待后续模板维护者评估
> 来源 Batch：`docs/research/2026-07-06-template-proposal-audit-batch-6-cross-cutting.md`
> 关联：`ai/document-lifecycle-rules.md`、`ai/global-rules.md`、`ai/project-rules.md`、Batch 2 / 3 / 4 / 5 提案

## 1. 背景与问题

在多阶段派生项目中，Mock、降级、默认关闭、候选、已验证、已启用、禁止等能力状态会贯穿多份文档：

- `03-prd` 定义 Phase 范围和交付物形态。
- `04-architecture` 定义系统边界和外部服务位置。
- `05-tech-spec` 定义技术栈、依赖、配置和降级策略。
- `06-db-design` 与 `07-api-spec` 定义目标结构、当前契约和 Mock / 真实边界。
- `08-dev-plan` 与 `09-verification` 定义实现范围、验证证据和未验证风险。
- `docs/design/*` 细化子系统流程、readiness gate 和实现偏差。

如果没有统一横切状态规范，AI 容易把“候选技术”写成“已用技术”，把“Mock payload”写成“真实通知”，把“降级路径”写成“目标能力已完成”，或在状态变化后漏改下游文档。

## 2. 设计目标

1. **统一术语**：为 Mock、降级、默认关闭、候选、已验证、已启用、禁止等状态提供统一定义。
2. **避免误承诺**：禁止把未验证、Mock、候选或默认关闭能力写成已启用 / 已实现。
3. **强化横切检查**：在审计、评估、Sprint 执行和 Phase 升级时检查横切状态。
4. **触发传播**：状态变化时必须检查 `03/project-rules/04-09/design/README` 的同步影响。
5. **避免重复**：本提案只定义词典和检查表，具体矩阵仍由 Batch 2 / 3 / 4 / 5 提案承担。

## 3. 横切状态词典

建议在模板规则或文档标准中统一以下状态：

| 状态 | 含义 | 可用于 | 禁止误写 |
|---|---|---|---|
| `已启用` | 当前 Phase 默认运行路径实际使用 | 已实现功能、已上线能力 | 未验证或未授权能力不得写已启用 |
| `已验证` | 在指定范围内验证通过 | 技术验证、资源验证、契约验证 | 不等同于生产可用或默认启用 |
| `Mock` | 仅模拟数据、payload、接口或状态流 | Demo、测试、降级演示 | 不得写成真实业务系统接入 |
| `降级` | 替代真实依赖的低成本路径 | 本机 Demo、资源不足、外部服务关闭 | 不得写成目标能力已完成 |
| `默认关闭` | 有设计或代码预留，但当前不开启 | LLM、真实通知、外部 API、付费服务 | 不得在当前 Sprint 默认启用 |
| `候选` | 方向性方案，尚未验证 | 技术路线、未来能力 | 不得进入实现计划，除非先做 Spike / PoC |
| `目标设计` | 未来结构或产品目标 | DB 目标表、API 未来版本、产品愿景 | 不得写成当前实现 |
| `草案` | 契约或方案未冻结 | 早期 API / DB / 设计文档 | 不得对外承诺稳定兼容 |
| `禁止` | 当前阶段不得接入或实现 | 真实数据、真实外部系统、未经授权服务 | 不得通过测试或演示绕过 |

## 4. 横切检查清单

### 4.1 Mock / 真实边界

触发条件：文档中出现订单、项目、通知、CRM、ERP、OA、工单、飞书、客户数据、外部 API。

必查：

- 是否明确 `Mock`、`默认关闭`、`候选`、`已验证` 或 `已启用`。
- 是否禁止真实 token、真实组织数据、真实客户隐私进入 Demo。
- 是否在 API / DB / design / verification 中保持同一口径。
- 是否有失败降级和不编造兜底。

### 4.2 降级路径

触发条件：Docker、数据库、向量检索、Embedding、LLM、外部通知、部署环境不可用或非当前阶段前置。

必查：

- 降级方案是否写清触发条件。
- 降级是否有验证记录。
- 降级是否改变 API / 用户承诺。
- 是否明确降级不等于目标能力已完成。

### 4.3 权限、安全与隐私

触发条件：接口对真实用户开放、员工控制台、外部服务、真实数据、试点部署、生产日志。

必查：

- 权限边界是否由后端接口和服务层执行，而非仅靠前端隐藏。
- 是否有敏感字段、日志脱敏、token / secret 管理和审计说明。
- 是否有角色 × 动作矩阵或明确豁免理由。
- 是否记录真实数据进入系统的合规前置条件。

### 4.4 高风险与不编造

触发条件：投诉、赔付、合同、报价、交期、法律责任、真实订单 / 项目、LLM 自动答复。

必查：

- 是否默认转人工或人工确认。
- 是否禁止无依据承诺。
- 是否记录来源证据、`source_ref` 或缺口。
- 若涉及 LLM，是否先完成证据约束、成本、安全和兜底评估。

### 4.5 候选能力进入 Sprint

触发条件：计划实现 LLM、向量检索、真实飞书、真实 CRM / ERP / OA / 工单、数据库迁移、生产部署、权限体系。

必查：

- 是否已从 `候选` / `默认关闭` 变更为可执行状态。
- 是否有人工确认和 readiness gate。
- 是否有技术验证、权限授权、安全边界和验证用例。
- 是否同步更新 `03/project-rules/05/08/09/design`。

## 5. 状态变化传播规则

当能力状态发生变化时，必须按以下清单评估传播：

| 状态变化 | 必查文档 | 检查内容 |
|---|---|---|
| `候选` → `已验证` | `05`、`09`、相关 design | 验证范围、证据和限制是否记录 |
| `已验证` → `已启用` | `03`、`project-rules`、`05`、`08`、`09`、README | 是否进入当前 Phase 范围，是否有验收计划 |
| `Mock` → `真实接入` | `03`、`04`、`05`、`06`、`07`、`09`、design | 授权、凭据、权限、审计、错误降级是否补齐 |
| `默认关闭` → `默认启用` | `project-rules`、`05`、`08`、`09` | 是否经过人工确认和风险接受 |
| `目标设计` → `当前实现` | `06`、`07`、`08`、`09` | 迁移、契约、验证和回滚是否补齐 |
| `禁止` → `允许` | `project-rules`、`03`、`decisions`、`09` | 是否有明确人工确认、合规边界和验收标准 |

## 7. 不覆盖范围

本提案不直接要求：

- 替代 Batch 2 / 3 / 4 / 5 的具体矩阵。
- 所有项目都接入数据库、权限、LLM 或外部服务。
- AI 未经人工确认改变能力状态。
- 把 Mock、降级、候选或默认关闭能力包装成生产可用。

## 8. 版本影响

建议版本影响：MINOR。

理由：该提案新增横切状态词典和检查流程，会影响规则、文档标准和多个 Prompt；但不破坏已有项目，可通过引用和追加检查渐进落地。

## 9. 影响面

| 影响对象 | 影响 |
|---|---|
| 派生项目文档 | 能用统一状态词典描述 Mock、降级、候选和已启用 |
| 审计 / 评估 Prompt | 更容易发现横切状态冲突和越界实现风险 |
| 开发任务 | 执行前可检查能力是否允许进入当前 Sprint |
| Phase 升级 | 状态变化触发传播检查，避免只改单点文档 |
| 模板维护者 | Batch 2 / 3 / 5 的状态类建议可归并到统一术语 |

## 10. 验收口径

模板维护者可用以下方式验收：

1. 模板有统一状态词典，覆盖已启用、已验证、Mock、降级、默认关闭、候选、目标设计、草案、禁止。
2. 审计 Prompt 能检查 Mock / 真实边界、降级路径、权限安全、高风险不编造和候选能力进入 Sprint。
3. Phase 升级或状态变化时，Prompt 能列出受影响文档和传播清单。
4. 开发任务 Prompt 能阻止 `候选` / `默认关闭` / `禁止` 能力被直接实现。
5. 提案不重复 Batch 2 / 3 / 5 的具体矩阵，只提供横切术语和检查表。

## 11. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 与前序提案重复 | 明确本提案只做术语和横切检查，不展开具体文档矩阵 |
| 术语过多增加理解成本 | 保持状态词典短而稳定，文档只引用必要状态 |
| AI 自动改变能力状态 | 规则明确状态变化需人工确认和验证证据 |
| 小项目负担过重 | 仅在涉及 Mock、降级、外部服务、安全、高风险或候选能力时触发 |
