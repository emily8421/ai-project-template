﻿# TEMPLATE-UPGRADE: 细化 00-03 需求链文档规范

> 来源：LUMEN_demo_T2.1（emily8421/LUMEN-DEMO）派生项目回流
> 依据：`docs/research/2026-07-06-docs-system-template-proposal-audit.md` Batch 0 审计报告

## 1. 背景与问题

派生项目对 `docs/00-09` 与 `ai/doc-standards/*` 做 Batch 0 只读审计时发现：`00-03` 是文档驱动开发的需求链核心，但当前模板规范仍容易让派生项目生成“有需求列表，但追溯和阶段决策不够稳定”的文档。

典型问题包括：

1. `00-scenario` 往往有背景和场景，但缺少场景边界、非目标、上游来源映射和下游影响说明。
2. `01-user-requirements` 容易按阶段列出 U-ID，却缺少用户操作流、用户可观察验收口径、优先级理由和 U-ID → REQ 下游追溯。
3. `02-srs` 容易只列功能 REQ，非功能需求、约束假设、异常 / 边界场景不足，导致后续架构、技术方案和验证计划缺少约束来源。
4. `03-prd` 可能有 Phase 路线图，但缺少统一的 Phase 总览表、进入 / 退出标准、取舍依据和非目标追溯。
5. `00-03` 的状态枚举、阶段标签、来源锚点、变更传播关系不统一，导致后续 `04-09` 和 `tasks/*` 难以稳定判断“当前 Phase 到底批准做什么”。
6. 结构化待人工确认项在规范中已有要求，但需求链文档生成时仍容易退化为一句“无新增确认项”，缺少确认项关闭、回填和影响范围说明。

这些问题不依赖具体业务项目。凡是使用 `Scenario → 用户需求 → SRS → PRD → 设计 → 计划 → 验证 → 代码` 链路的派生项目，都需要更细的 `00-03` 模板规范，确保需求来源、用户目标、系统需求、Phase 路线图和非目标可追溯。

## 2. 目标

细化 `00-03` 文档规范，使模板能稳定生成和审计以下链路：

```text
上游输入 / 愿景 / 人工确认
  → 00 场景 / 边界 / 来源锚点
  → 01 U-ID / 用户操作流 / 用户验收口径
  → 02 REQ-ID / NFR / 约束 / 异常场景
  → 03 Phase / 优先级 / 非目标 / 进入退出标准
  → 04-09 / tasks / tests
```

目标能力：

1. 每个场景能追溯到输入材料或人工确认来源。
2. 每个 U-ID 能追溯到场景，并能映射到一个或多个 REQ-ID。
3. 每个当前 Phase REQ 有可验证口径，远期 REQ 只保留骨架并标明待细化。
4. 每个 Phase 同时声明功能范围和交付物形态，不把 Demo 写成 MVP。
5. 非目标、约束、假设、异常场景能传递到架构、技术方案和验证计划。
6. 待人工确认项不只是问题清单，而能说明 AI 建议、依据、备选方案、影响和是否阻塞。

## 3. 拟修改范围

| 文件 | 建议 |
|---|---|
| `ai/doc-standards/00-scenario.md` | 增强场景边界、来源映射、场景 → U-ID 预期输出和变更影响要求 |
| `ai/doc-standards/01-user-requirements.md` | 增强 U-ID 总览、用户操作流、用户验收口径、优先级建议、非目标和追溯矩阵 |
| `ai/doc-standards/02-srs.md` | 增强功能 REQ、NFR、约束假设、异常 / 边界场景、U→REQ 追溯和验证入口 |
| `ai/doc-standards/03-prd.md` | 增强 Phase 总览、进入 / 退出标准、功能范围 + 交付物形态、优先级取舍和非目标追溯 |
| `ai/document-lifecycle-rules.md` | 可选：补充 `00-03` 需求链最小追溯要求和状态一致性检查 |
| `ai/prompts/docs/00-generate-or-complete-docs.md` | 生成文档体系时强制输出 `00-03` 的来源、U-ID、REQ-ID、Phase、非目标和待确认项结构 |
| `ai/prompts/docs/01-review-inputs.md` | 输入评审时输出可直接驱动 `00` 的来源锚点和缺口清单 |
| `ai/prompts/docs/04-edit-single-doc.md` | 修订 `00-03` 单文档时提醒评估上游 / 下游影响 |
| `ai/prompts/review/19-docs-evaluation.md` | 阶段评估 E1 / E2 时增强需求链断点检查 |

## 4. 建议的 00 文档规范

### 4.1 结构建议

```markdown
# 00 Scenario（项目背景与场景）

## 0. 文档元信息
## 1. 背景与问题
## 2. 目标用户与角色
## 3. 典型场景
## 4. 场景边界与非目标
## 5. 上游来源映射
## 6. 下游影响
## 7. 待人工确认项
```

### 4.2 字段要求

`00` 应避免只写叙事摘要。建议增加以下表格：

| 场景 ID | 场景名称 | 角色 | 触发条件 | 用户目标 | 成功结果 | 来源锚点 | 下游 U-ID |
|---|---|---|---|---|---|---|---|

场景边界建议使用：

| 边界 ID | 不覆盖内容 | 原因 | 后续处理 | 是否影响 01-03 |
|---|---|---|---|---|

来源映射建议使用：

| 来源 ID | 来源类型 | 来源位置 | 支撑场景 | 可信度 / 确认状态 | 备注 |
|---|---|---|---|---|---|

### 4.3 禁止项

- 不得把场景直接写成系统功能清单。
- 不得在 `00` 决定 Phase 范围；Phase 唯一来源应是 `03`。
- 不得把未评审输入或愿景叙事伪装成已确认需求。
- 不得留下无来源场景进入 `01`。

## 5. 建议的 01 文档规范

### 5.1 结构建议

```markdown
# 01 用户需求说明（User Requirements）

## 0. 文档元信息
## 1. 用户需求总览
## 2. 用户操作流
## 3. 用户验收口径
## 4. 优先级与阶段建议
## 5. 非目标与排除需求
## 6. 追溯矩阵
## 7. 待人工确认项
```

### 5.2 U-ID 总览

建议模板要求每条 U-ID 至少包含：

| U-ID | 用户 / 角色 | 用户需求 | 来源场景 | 用户价值 | 初步优先级 | 建议阶段 | 状态 |
|---|---|---|---|---|---|---|---|

说明：

- `建议阶段` 只是需求阶段建议，不是最终 Phase 裁决。
- 最终阶段归属由 `03-prd.md` §3 决定。
- 若文档按 `[P1] / [P2] / [愿景]` 分组展示，仍应保留总览或追溯矩阵，避免跨阶段需求丢失。

### 5.3 用户操作流

建议模板要求关键用户流采用稳定 ID：

```markdown
### Flow-001：<用户流名称>

- 关联角色：
- 关联场景：
- 覆盖 U-ID：
- 前置状态：
- 用户步骤：
- 成功结果：
- 异常 / 不满足示例：
```

### 5.4 用户验收口径

用户验收口径应描述用户可观察结果，不写代码级测试细节。

| U-ID | 用户验收口径 | 不满足示例 | 是否进入当前 Phase | 备注 |
|---|---|---|---|---|

### 5.5 非目标与排除需求

| OUT-ID | 用户说法 / 来源 | 排除类型 | 原因 | 后续处理 | 是否需进入 03 非目标 |
|---|---|---|---|---|---|

排除类型建议枚举：`不做`、`后续阶段`、`待确认`、`愿景保留`、`合规 / 资源禁止`。

## 6. 建议的 02 文档规范

### 6.1 结构建议

```markdown
# 02 需求规格说明（SRS）

## 0. 文档元信息
## 1. 功能需求
## 2. 非功能需求
## 3. 约束与假设
## 4. 边界条件与异常场景
## 5. U-ID → REQ-ID 追溯矩阵
## 6. 下游验证入口
## 7. 待人工确认项
```

### 6.2 功能 REQ

| REQ-ID | 系统需求 | 来源 U-ID | 可验证口径 | 初步阶段 | 状态 |
|---|---|---|---|---|---|

要求：

- 当前 Phase REQ 必须可验证。
- 后续阶段 REQ 可保留粗粒度，但必须标 `待该阶段细化`。
- 不得新增无 U-ID 或无约束来源的 REQ。

### 6.3 非功能需求

建议模板要求 NFR 独立编号，避免散落在 04 / 05 / 09。

| NFR-ID | 类型 | 需求 | 来源 / 依据 | 验证方式 | 状态 |
|---|---|---|---|---|---|

类型建议包括：安全、隐私、性能、可靠性、可维护性、可观测性、资源、兼容性、合规。

### 6.4 约束与假设

| CON-ID | 约束 / 假设 | 来源 | 影响文档 | 若不成立的影响 | 状态 |
|---|---|---|---|---|---|

### 6.5 边界与异常场景

| CASE-ID | 场景 | 关联 REQ / NFR | 预期系统行为 | 验证入口 | 状态 |
|---|---|---|---|---|---|

典型边界包括：无权限、无数据、外部服务不可用、导入失败、资源不足、跨空间访问、库外问题、重复请求。

## 7. 建议的 03 文档规范

### 7.1 结构建议

```markdown
# 03 产品需求文档（PRD）

## 0. 文档元信息
## 1. 产品目标与范围
## 2. 功能清单
## 3. Phase 路线图
## 3.1 Phase 总览表
## 4. 优先级与取舍
## 5. 非目标
## 6. REQ 覆盖矩阵
## 7. 进入 / 退出标准
## 8. 待人工确认项
```

### 7.2 Phase 总览表

建议模板将 Phase 总览表设为强推荐或必填：

| Phase | 功能范围 | 交付物形态 | 状态 | 包含 REQ | 进入标准 | 退出标准 | 明确不做 |
|---|---|---|---|---|---|---|---|
| Phase1 | `[P1]` | Demo | 当前 | REQ-... |  |  |  |

要求：

- 功能范围标签与交付物形态必须同时出现。
- Demo / MVP / 产品不得混用。
- 当前阶段禁止项应同步到 `ai/project-rules.md` 当前阶段指针。
- 后续 Phase 可保留骨架，但不得提前写死细节。

### 7.3 优先级与取舍

| REQ-ID | 优先级 | 所属 Phase | 取舍理由 | 依赖 / 风险 | 是否阻塞当前 Phase |
|---|---|---|---|---|---|

### 7.4 非目标

| OUT-ID | 非目标 | 来源 | 所属阶段 / 永不做 | 理由 | 防蔓延说明 |
|---|---|---|---|---|---|

### 7.5 REQ 覆盖矩阵

| REQ-ID | 来源 U-ID | Phase | 交付物形态 | 下游设计 | 验证入口 | 状态 |
|---|---|---|---|---|---|---|

## 8. 状态与标签建议

建议统一 `00-03` 的状态枚举，避免每个文档自由发挥。

| 状态 | 适用文档 | 含义 |
|---|---|---|
| `草稿` | 00-03 | 尚未完成来源锚定或人工确认 |
| `待人工确认` | 00-03 | 结构完整但存在阻塞确认项 |
| `已确认` | 00-03 | 可作为下游依据 |
| `P1-已设计` | 01 / 02 / 03 | 当前 Phase 需求或 REQ 已完成设计承接 |
| `P1-已实现` | 01 / 02 / 03 | 已有实现与验收记录支撑；不得仅凭代码存在标记 |
| `骨架` | 后续阶段 / 愿景 | 保留方向，不作为当前 Phase 实现依据 |
| `待该阶段细化` | 后续阶段 | 升阶段前必须补充细节 |

## 9. 生成与评估 Prompt 增强建议

### 9.1 `generate-docs`

生成 `00-03` 时应明确输出：

- `00`：场景 ID、来源锚点、边界 / 非目标、下游 U-ID。
- `01`：U-ID、用户流、用户验收口径、优先级建议、非目标、追溯矩阵。
- `02`：REQ-ID、NFR-ID、约束 / 假设、异常场景、U→REQ 映射。
- `03`：Phase 总览、功能范围 + 交付物形态、进入 / 退出标准、REQ 覆盖矩阵、非目标。

### 9.2 `review-inputs`

输入评审应输出可直接转入 `00` 的来源锚点：

| 来源 ID | 原始材料 | 支撑场景 / 用户 | 缺口 | 是否足以进入 00-03 |
|---|---|---|---|---|

### 9.3 `edit-single-doc`

修订 `00-03` 任一文档时，应要求 AI 评估：

- 上游来源是否变化。
- 下游 `01-09` 是否受影响。
- U-ID / REQ-ID / Phase 是否新增、删除或改阶段。
- 是否需要同步 `ai/project-rules.md` 当前 Phase 指针。

### 9.4 `docs-evaluation`

阶段评估 E1 / E2 应增加硬性检查：

- 是否存在无来源场景。
- 是否存在无场景来源的 U-ID。
- 是否存在无 U-ID 的 REQ。
- 是否存在无 REQ 的 Phase 功能。
- 是否存在 Demo / MVP / 产品交付物形态混用。
- 是否存在后续阶段功能被写入当前 Phase。

## 10. 兼容与迁移建议

对已存在派生项目，不建议强制重写 `00-03`。建议渐进迁移：

1. 先补 `03` Phase 总览表和进入 / 退出标准，确保当前阶段边界清楚。
2. 补 `02` 的 NFR、约束和异常场景，供 04 / 05 / 09 使用。
3. 补 `01` 的用户验收口径和 U-ID → REQ 追溯矩阵。
4. 补 `00` 的来源映射和场景边界。
5. 历史来源缺失时标注“由下游摘要反推，待人工确认”，不得伪装为原始输入。

## 11. 验收标准

本提案落地后，模板应满足：

1. 新项目生成 `00-03` 时，每个 U-ID 和 REQ-ID 都能追溯到上游来源。
2. `03` 能一眼看出每个 Phase 的功能范围、交付物形态、进入标准、退出标准和明确不做。
3. `02` 不只包含功能需求，还能承载 NFR、约束、假设和异常场景。
4. `01` 不只包含功能列表，还能承载用户流和用户可观察验收口径。
5. `00` 不只包含背景叙事，还能承载场景边界、来源映射和下游影响。
6. `docs-evaluation` 能稳定发现无来源 U-ID、无 U-ID REQ、无 REQ Phase 功能和阶段越界问题。
7. 旧项目可按最小迁移步骤补齐，不需要一次性重写需求链。

## 12. 版本影响

建议版本：模板方法论 minor 版本升级。

影响面：

- 对新项目：`00-03` 文档会更结构化，初次生成内容略增，但后续设计、实现和验收更稳。
- 对旧项目：同步后不会自动修改项目事实文档；需要通过 `docs-evaluation`、`docs-system-audit` 或 `edit-single-doc` 按需迁移。
- 对 Prompt：生成 / 修订 / 评估 `00-03` 时检查项增加，但不改变主流程。
- 对脚本：原则上无需脚本变更。

## 13. 非目标

- 不要求所有项目从产品愿景开始；仍支持 Inputs-first、SRS-first、PRD-first、Task-first 等入口。
- 不要求 Lean 项目写冗长需求文档；Lean 项目可用轻量表格，但追溯链不能断。
- 不把派生项目的具体业务角色、场景或功能写入模板。
- 不替代 `docs/04-09` 的设计、计划和验证规范。

## 14. 待模板维护者确认

| ID | 待确认项 | AI 建议 | 建议依据 | 备选方案 | 取舍影响 / 阻塞关系 |
|---|---|---|---|---|---|
| C-001 | 是否将 `03` Phase 总览表设为必填 | 建议必填 | Phase 是当前阶段边界唯一来源，缺少总览会影响后续设计和编码 | 保持推荐 | 推荐更灵活，但旧问题容易反复出现 |
| C-002 | 是否要求 `02` 必须包含 NFR / 约束 / 异常场景章节 | 建议必填，Lean 项目可简写 | 架构、技术方案和验证计划都需要这些约束来源 | 仅在项目复杂时触发 | 触发条件不清时容易漏写 |
| C-003 | 是否要求 `01` 同时支持“阶段分组”和“U-ID 总览 / 追溯矩阵” | 建议支持两者，但追溯矩阵必留 | 阶段分组易读，总览 / 矩阵便于审计 | 只保留阶段分组 | 容易跨阶段丢需求或断追溯 |
| C-004 | 是否统一 `00-03` 状态枚举 | 建议统一最小枚举，项目可扩展 | 便于 AI 续接、审计和阶段评估 | 保持自由文本 | 自由文本灵活但难以自动判断 |
