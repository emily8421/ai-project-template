# TEMPLATE-UPGRADE: 文档体系回溯审计提示词（补 review 类缺口）

> 类型：派生项目起草的模板优化提案（去项目化）。
> 状态：已落地于 v1.17.0；二次实证补充已落地于 v1.21.2（2026-06-30）；从 `_proposals/` 归档到 `_archive/proposals/`。
> 关联：补充 `ai/prompts/review/` 审查类提示词；依赖 `ai/document-lifecycle-rules.md`（v1.8.0 落地）。
> 更新：v1.21.0 后规范基线主路径为 `ai/doc-standards/00-09`；`docs/_scaffold/00-09` 仅作为 v1.18.x 旧路径 fallback。

## 1. 动机

v1.8.0 落地 `ai/document-lifecycle-rules.md`（PLM 链路 / 生成矩阵 / 追溯链 / 横切一致性 / 变更传播 / 外部接入）后，派生项目出现一个新且高频的需求：**用新规则回溯审视「已经生成、已跑过若干 Sprint/Phase」的文档体系**是否仍然自洽——尤其阶段间的合理性与可行性。

但现有提示词没有一个正好覆盖此场景：

| 现有提示词 | 定位 | 为什么不够 |
|---|---|---|
| `docs/00-generate-or-complete-docs.md` | 生成 / 补齐文档体系 | 面向「从输入生成」，不是「事后审视」 |
| `review/10-docs-checklist.md` | 编码前验收（含 B0 追溯 / C 一致性） | 定位「生成 03-09 后、编码前」的同步验收，非「项目成型后回溯」 |
| `review/03-project-review.md` | 合规审查（越界/偏离/追溯断点） | 偏「实现是否符合设计」，不做「阶段间合理性 + 可行性」评估 |

缺口：**没有面向「已成型项目、用 document-lifecycle-rules 回溯审视整条 PLM 链路合理性与可行性」的提示词**。

派生项目（数字客服 Demo）实证：v1.9.0 同步后想做「vision→需求是否合理、需求→设计是否可行」的回溯审视，现有提示词都不顺手；最终手动用 document-lifecycle-rules 的 §2/§5/§6/§7/§9 + 可行性维度拼了一次全链路审计，发现该方法可复用、值得固化成提示词。

## 2. 拟改（增量）

新增 `ai/prompts/review/NN-docs-system-audit.md`（编号建议 review 类顺延，如 16），定位：**文档体系全链路回溯审计**——项目成型后，用 document-lifecycle-rules 逐段审视 PLM 链路的追溯完整性、横切一致性、变更传播、外部接入、阶段可行性，产出「链路健康度报告 + 问题清单 + 回梳计划」，不改文件、先出报告。

提示词正文草案（供模板维护者参考，去项目化）：

```text
请基于 ai/index.md 列出的规则文件（尤其 ai/document-lifecycle-rules.md）+ 本项目 docs/ 全部文档，做一次「文档体系全链路回溯审计」。

适用场景：项目已成型（完成若干 Sprint/Phase），用最新方法论回头审视整条 PLM 链路的合理性、可行性与一致性。区别于 00（生成）/ 10（编码前验收）/ 03（合规审查），本提示词是「事后全链路回溯」。

按以下维度逐段核查并输出：
1. 纵向追溯链（§6）：输入→U-ID→REQ-ID→Phase→模块/表/接口/Sprint/用例，是否闭合、有无悬空 ID。
2. 横切一致性（§7）：每个横切事实（平台能力/合规裁决/技术选型禁令/可行性核实）是否有唯一权威源；他处引用而非各自声明。
3. 变更传播（§9）：上游变更（尤其已核实横切事实）是否传播到所有下游；列出残留（措辞过时/未引用权威源）。
4. 外部文档接入（§8）：外部接入文档是否有锚定/定位/追溯/正确分区；有无命名冲突孤岛。
5. 生成矩阵（§5）：每份文档的输入/输出职责/禁止项/追溯锚点/下游影响是否兑现。
6. 可行性维度：需求→设计是否技术可行（受 project-rules §2/§2.5 + docs/env 约束）、设计→计划是否可执行、计划→验证是否可验收；各约束有无降级方案。
7. 交付物形态（global-rules §8.1）：各 Phase 是否声明 Demo/MVP/产品，有无 Demo 误称 MVP/产品。

输出：链路健康度总览表 + 各维度问题清单（定位 文件:行 + 权威源）+ 回梳计划（按横切事实分组）+ 审计新发现（可行性/部署缺口）+ 待人工确认项。
不改文件，先出报告；确认后按 ai/prompts/docs/04-edit-single-doc.md 最小变更回梳。
```

## 3. 版本影响

建议作为 v1.9.x 补丁或 v1.10.0 增量。依赖 v1.8.0 的 `ai/document-lifecycle-rules.md`（本提示词是其应用入口之一）。

## 4. 影响面

- 新增 `ai/prompts/review/NN-docs-system-audit.md`。
- `INIT-PROMPT.md` 场景索引 + `ai/prompts/README.md` + `template-sync.json` 同步清单纳入。
- 与 `03-project-review` / `10-docs-checklist` 在「不适用场景」段互相指向，避免混淆。

## 5. 迁移建议（派生项目实证）

派生项目（数字客服 Demo）作为本提示词的**首个验证案例**：v1.9.0 同步后用它做全链路审计，确认链路主干强健、定位三症状（横切传播残留 + 外部文档孤岛）为唯一实质待修项，并额外发现 3 个可行性/部署小缺口。提示词正文即从该次审计流程抽象而来。

## 6. 验收口径

- 项目成型后，能用本提示词一次性产出全链路健康度报告，而非分别跑 00/10/03 再拼。
- 报告覆盖追溯链、横切一致性、变更传播、外部接入、可行性、交付物形态六个维度。
- 产出的回梳计划能直接驱动 `04-edit-single-doc.md` 最小变更回梳（定位到 文件:行 + 权威源）。
## 7. 二次实证补充（2026-06-29）

v1.18.1 下行同步后，数字客服 Demo 曾按 `16-docs-system-audit` + `docs/_scaffold/00-09` 做了一轮「审计 → 最小回梳 → 自检」闭环；v1.21.0 后同类审计应以 `ai/doc-standards/00-09` 为主路径、`docs/_scaffold/00-09` 仅作旧项目 fallback。该实证补充验证出两个可回流点：

1. **审计报告应显式区分“事实缺口”和“规范基线缺口”**：本轮组B暴露的是 v1.18.1 scaffold 新基线带来的结构缺口（00-09 缺 §0 文档元信息、01/02/03/06 缺矩阵化追溯表），不是业务事实缺失。提示词应要求输出时标明缺口类型，避免把模板新规范误判为需求事实错误。
2. **审计提示词应附带修复后自检清单**：报告之后的最小回梳需要可验证命令，例如检查所有 00-09 是否存在 `## 0. 文档元信息`、关键矩阵是否存在、旧版本引用/已完成待办是否无残留。否则“审计发现”容易修了文档但忘记更新续接状态或遗漏矩阵。
3. **`U-ID` 兼容迁移需要明确策略**：历史项目可能已用 `F-*` 作为用户需求编号。提示词不应强制全文重命名，而应建议新增 `U-ID ↔ 旧编号` 兼容矩阵，先满足追溯闭合，再把是否重命名交给人工决策。
4. **续接便签也应纳入审计闭环**：若项目有本地 `NEXT-STEPS.md` / 类似续接文件，完成回梳后应同步标记状态，避免下次对话重复执行已完成项。该类文件仍不属于模板正式文档，提示词只提醒“若存在则更新本地状态”。

### 建议补入 `16-docs-system-audit` 的输出要求

在“输出”段增加一个小节：

```text
请把问题按以下类型分组：
- 事实 / 追溯断点：影响需求、设计、计划或验证闭环的实质问题。
- 横切传播残留：权威源已变更但下游措辞、引用或阶段标签未同步。
- 规范基线缺口：对照 `ai/doc-standards/00-09`（旧项目 fallback：`docs/_scaffold/00-09`）发现的章节 / 元信息 / 矩阵结构缺失；不代表业务事实错误。
- 可行性 / 部署缺口：技术可行性、资源、调度、运行环境或验证入口未说明。
- 本地续接状态：若存在 NEXT-STEPS.md 等本地便签，列出需同步的状态项。

每个问题给出：类型、严重度、文件:行、权威源、建议修复方式、是否改业务事实。
```

### 建议补入“修复后自检”段

```text
若审计报告被确认并进入回梳，修复后至少做以下聚焦自检：
1. `git diff --check`。
2. 对照问题清单逐项 `rg` 验证旧措辞 / 旧版本号 / 已完成待办无残留。
3. 若使用 `ai/doc-standards/00-09`（旧项目 fallback：`docs/_scaffold/00-09`）：确认 00-09 均有必需章节（如 §0 文档元信息），重点文档的追溯矩阵存在。
4. 确认新增矩阵没有制造新的悬空 ID（U-ID / REQ-ID / Phase / Sprint / 用例）。
5. 若更新了本地续接便签，确认推荐路径不再指向已完成事项。
```

### 验收口径补充

- 能把 scaffold 对照暴露的“结构缺口”与业务事实问题分开输出。
- 对历史 `F-*` / 自定义编号项目，能给出兼容映射方案，而非强制大规模重命名。
- 审计报告能自然导向 `04-edit-single-doc.md` 或小批量文档回梳，并包含修复后自检命令。
- 完成回梳后，项目续接状态不再残留已完成待办。

## 8. 旧派生文档兼容审计补充（2026-06-30）

派生项目的 `docs/00-09` 可能是在旧模板版本下生成的项目事实文档，不应为了贴合新版 `ai/doc-standards/00-09` 示例骨架而整份重写。审计提示词应明确采用“旧派生文档兼容审计模式”：

1. **语义等价优先**：新版规范中的“文档定位 / 上游输入 / 下游输出 / §0 元信息 / 追溯矩阵”等关键约束必须满足；章节标题不必逐字等于规范示例。
2. **标题编号必须基本统一**：同一文档内同级标题不得长期混用“有序号 / 无序号”；若使用数字编号，应从 `## 1` 连续递增，允许 `## 0. 文档元信息` 作为固定前置章节；可使用 `## 2.1` 表示子章节，但不得出现明显跳号、重复号或同级编号风格不一致。
3. **必要版块可增量补齐**：若旧文档缺少对当前项目仍有审计价值的版块（如 REQ→接口矩阵、验收记录、风险与未验证项、运行环境验证），应在保留原语义和历史事实的基础上最小补充，而不是为了保持旧结构不变而放弃关键约束。
4. **禁止机械重写**：不得为了匹配 `ai/doc-standards` 示例结构而删除项目事实、历史验收、决策上下文、旧编号兼容矩阵或已确认的阶段信息。
5. **问题分类输出**：审计报告应区分“必须修复的结构约束”（如标题编号混乱、缺关键元信息、追溯断点）与“可接受的风格差异”（如章节名称未逐字匹配规范示例）。

建议补入 `16-docs-system-audit` 的规范基线检查段：

```text
对旧派生项目执行兼容审计：不要求 docs/00-09 逐字重写成 ai/doc-standards 示例骨架，但必须检查并报告：
- 同一文档内 H2/H3 标题编号风格是否统一、连续、无明显跳号或重复；
- 是否存在必要但缺失的关键版块（元信息、追溯矩阵、接口/表/用例矩阵、验收记录、风险与未验证项、环境验证）；
- 若补充版块，必须保持原项目语义和历史事实，不得机械重写或删除旧内容。
```
