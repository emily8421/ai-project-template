# Document Lifecycle Rules（文档生命周期生成规则）

> Sync notice: This file is maintained by `ai-project-template` and may be overwritten when a derived project syncs template methodology.
> Do not edit it directly in derived projects; propose reusable changes in `_proposals/` and upstream them to the template repository.

本文件定义从产品愿景到需求、总体设计、详细设计、实现计划、测试验证、代码实现的文档生命周期生成规则。AI 生成或修改任何项目事实文档时，必须先识别该文档的上游输入、约束来源、输出职责和下游影响，不得凭空生成。

> **阅读地图**（按你想问的问题读）：
> - **这是什么 / 为什么** → §1 基本原则（文档体系是什么、为什么要这套规则）
> - **怎么做（文档怎么生成）** → §2 PLM 阶段链路 → §3 多入口生成策略 → §4 文档体系剖面 → §5 文档生成矩阵
> - **规范与约束** → §6 追溯链规则 / §7 横切事实与权威源 / §8 外部文档接入规则 / §9 变更传播规则 / §12 禁止项
> - **AI 生成流程** → §10 AI 生成前声明 / §11 AI 生成后自检
> - **图表规范** → §13 设计文档图表规范

## 1. 基本原则

> **文档体系是什么**：一套从输入材料（愿景 / PRD / SRS / brief）到需求、设计、计划、验证、代码的可追溯文档链（`docs/00-09` + 子目录）。**为什么需要这套规则**：避免 AI 凭空生成、上下游脱节、横切事实散落、变更不传播——让每份文档有据可生成、可追溯、可裁剪、可审计。

1. **有据生成**：每个文档必须由明确上游输入生成，不得只凭对话印象或 AI 猜测补齐。
2. **逐级约束**：下游文档不得引入上游未授权的需求、模块、表、接口、依赖或测试目标。
3. **全链追溯**：愿景场景、用户需求、系统需求、阶段路线图、设计、任务、测试、代码之间必须能建立追溯关系。
4. **变更传播**：上游文档变更后，必须评估下游影响；不得只改单点文档。
5. **横切一致**：平台能力、合规裁决、技术可行性、选型禁令等横切事实必须有唯一权威源，其他文档引用权威源而非各自重述。
6. **阶段分离**：愿景功能不等于当前阶段功能；当前阶段范围以 `docs/03-prd.md` §3 与 `ai/project-rules.md` §1 为准。
7. **状态据实**：候选、预留、默认关闭、Mock 或未实现能力不得写成已接入 / 已实现。

## 2. PLM 阶段链路

```text
docs/inputs 原始输入包
  → 输入材料评审 / 愿景就绪评估 / 缺口补齐
  → 产品愿景 / 输入摘要
  → 需求阶段（00 / 01 / 02 / 03）
  → 总体设计阶段（04 / 05）
  → 详细设计阶段（06 / 07 / docs/design/*）
  → 实现计划阶段（08 / tasks/*）
  → 测试验证阶段（09 / tests）
  → 实现阶段（code / scripts / docker / runtime docs）
```

每个阶段必须声明：

- 输入来源。
- 输出职责。
- 禁止引入内容。
- 追溯 ID 或来源锚点。
- 下游影响范围。

关键阶段转换前可使用 `ai/prompts/review/19-docs-evaluation.md` 做文档评估，输出 `Go / Conditional Go / No Go`。评估报告默认只读输出；用户确认落盘后写入 `docs/research/YYYY-MM-DD-docs-evaluation-<scope>.md`，不得写入 `docs/` 根目录，也不得替代 00-09 正式修订。

生成或修订 `docs/05-tech-spec.md` 前，若项目涉及真实运行依赖（如 `backend/`、`frontend/`、`docker/`、数据库、本机模型、外部 API、重型 SDK），应先使用 `ai/prompts/review/20-tech-env-evaluation.md` 做技术路线与环境支撑评估，或在 `ai/project-rules.md` §2.5 / `docs/05-tech-spec.md` 说明跳过理由、风险和补做时点。技术环境评估报告默认只读输出；用户确认落盘后写入 `docs/research/YYYY-MM-DD-tech-env-evaluation-<scope>.md`，不得替代 `docs/env/local-env.md` 或 `docs/05-tech-spec.md`。评估结论应回填或引用到 `05` 的 Risk-ID / readiness gate、`09` 验证项和 `08` Sprint 解锁条件。`collect-env` 只采集事实，不等于评估通过。

| 评估码 | 阶段转换 | 核心问题 |
|---|---|---|
| E1 | 输入 / 愿景 → 需求阶段 | 输入是否足以生成 00-03；是否有来源锚点与需求编号 |
| E2 | 需求阶段 → 总体设计 | 04-05 是否只承接已授权需求；架构和技术基线是否可行 |
| E3 | 总体设计 → 详细设计 | 06/07/design 是否覆盖关键模块、数据、接口、前端交互、UI 原型策略和流程 |
| E4 | 详细设计 → 实现计划 | Sprint 是否可执行、粒度是否合适、禁止事项是否明确 |
| E5 | 实现计划 → 验证 | 09 是否覆盖 REQ 与 Sprint 验收，Mock / 降级边界是否明确 |
| E6 | 实现结果 → 文档回写 | 代码事实是否需要反向同步到 docs |

评估结论含义：`Go` 表示可进入下一阶段；`Conditional Go` 表示满足指定条件或风险接受口径后可进入；`No Go` 表示存在阻塞缺口，不得进入下一阶段。

## 3. 多入口生成策略

文档生命周期规则不假设所有项目都从 `docs/vision/product-vision.md` 开始。默认情况下，用户原始材料先进入 `docs/inputs/`，再通过输入材料评审判断是否足以生成 / 更新 `docs/vision/product-vision.md`。愿景文档是理想的完整产品叙事入口；小系统、内部工具、已有 PRD / SRS 的项目允许从更成熟的输入开始，但仍必须先完成评审、缺口说明和来源锚定。

上游输入包：

```text
上游输入包 = 愿景文档 / docs/inputs 原始输入包 / 00-scenario / 用户需求清单 / SRS / PRD / 任务单 / 现有系统说明 / 接口草案 / 人工确认约束 / 外部接入文档
```

入口选择矩阵：

| 入口模式 | 已有输入 | 允许动作 | 必须补齐 | 停下问人的情况 |
|---|---|---|---|---|
| Inputs-first（默认） | `docs/inputs/` 原始输入包，可能为空或混杂 | 先做输入材料评审与愿景就绪评估；足够时生成 / 更新 `docs/vision/product-vision.md`，不足时输出 `docs/inputs/input-review-report.md` 和最小补充清单 | 目标用户、问题 / 价值、典型场景、核心能力、输入输出、边界 / 非目标、成功标准、阶段倾向、运行 / 技术 / 合规约束 | `docs/inputs/` 为空，或无法判断产品目标、用户、场景、成功标准 |
| Vision-first | 已确认的产品愿景 / 故事叙事，通常为 `docs/vision/product-vision.md` | 生成 `00-09` 完整骨架 | `project-rules` 初值、环境约束、输入评审来源锚点 | 愿景无法抽出角色 / 场景 / 功能点，或无法追溯到输入来源 |
| Scenario-first | `00-scenario.md` | 从 `01` 起补齐需求链与设计 | 来源说明、阶段建议 | 场景无法枚举功能点 |
| URS-first | 用户需求清单 / 访谈结论 | 补轻量 `00`，生成 `02-09` | `00` 工程想定摘要、需求来源 | 需求无法编号或无法验收 |
| SRS-first | 系统需求规格 | 反向补轻量 `00/01` 摘要，生成 `03-09` | U-ID / 来源映射、阶段路线图 | REQ 来源不明或范围冲突 |
| PRD-first | 已有 PRD / 路线图 | 补追溯所需 `00-02` 摘要，生成 `04-09` | REQ 清单、阶段标签唯一来源 | PRD 功能无法追溯到需求 |
| Existing-system | 现有系统 / 代码 / 接口说明 | 先做事实盘点，再补 docs | 现状 / 目标 / 差距 / 风险 | 代码事实与目标需求冲突 |
| Task-first | 明确任务单 | 仅在现有 docs 足够时执行任务 | task → Sprint → REQ 追溯 | 缺少可追溯 REQ 或设计依据 |
| Small-tool brief | 小工具 / 小系统的自由描述，常放 `docs/inputs/initial-brief.md` | 先提炼目标用户、输入、输出、核心操作流、非目标和验收方式，再按 Lean 剖面生成 | 最小输入摘要、运行形态、验证入口 | 无法判断输入 / 输出 / 成功标准 |
| External-input | 外部策略 / 调研 / 决策 / 会议文档，可先放 `docs/inputs/` 输入包 | 先接入锚定，再判断驱动或约束哪些 docs | 映射表、定位声明、影响范围 | 外部结论无法映射到本仓库 docs 或与已确认事实冲突 |

要求：

- 不强制所有项目编写愿景文档。
- 跳过愿景时，必须有可审计的替代输入，并在生成文档中标注来源；尚未归类的替代输入优先放入 `docs/inputs/`。
- `docs/inputs/` 是普通用户投递原始材料的默认入口；`docs/vision/` 是整理后的愿景叙事或兼容旧项目已有愿景，不作为未经评审原始材料的默认入口。
- 生成 `docs/vision/product-vision.md` 前，必须先判断输入是否达到“愿景就绪”：能说明目标用户、问题 / 价值、典型场景、核心能力、输入输出、边界 / 非目标、成功标准、阶段倾向和关键约束。
- 输入不足时，AI 必须先输出评估报告和最小补充清单；用户补齐后重新执行 `ai/prompts/docs/01-review-inputs.md`，不得直接凭猜测生成 product-vision 或 00-09。
- 反向补齐的上游文档必须标注“由下游输入摘要生成，待人工确认”，不得伪装成原始需求来源。
- 简化的是文档厚度和可省略项，不是追溯关系。

## 4. 文档体系剖面

| 剖面 | 适用项目 | 必备文档 | 可裁剪项 | 约束 |
|---|---|---|---|---|
| Full | 产品级系统 / 愿景驱动项目 | `vision` + `00-09` + `docs/design/*` | 无，除非 `project-rules §3` 明确省略 | 完整追溯与阶段骨架；UI 型项目按触发条件补前端交互设计和 UI 原型策略 |
| Standard | 常规 Web / 服务项目 | `00-09` | 非平凡子系统或复杂 UI 才建 `docs/design/*` | `06/07` 按持久化 / 接口裁剪；前端交互设计与 UI 原型策略按 UI 复杂度触发 |
| Lean | 小工具 / 内部脚本 / CLI | 轻量 `00-03` + 极简 `04` + `05` + `08` + `09` | 可省略 `06/07/design`，需在 `project-rules §3` 说明 | 仍需任务、验收与依赖边界；`04` 至少说明运行形态和模块边界 |
| Existing-system | 既有系统改造 | 现状说明 + 轻量 `00-03` + 受影响设计 / 验证 | 不受影响文档可保留骨架 | 必须区分“现状事实”和“目标变更” |


## 5.0 文档标准分层与按 scope 读取规则

AI 生成、精修、审计或评估项目事实文档时，必须区分三层输入：

| 层级 | 文件 | 使用方式 |
|---|---|---|
| 生命周期总控 | `ai/document-lifecycle-rules.md` | 判断入口模式、阶段转换、追溯链、状态传播和变更影响 |
| 细粒度标准 | `ai/doc-standards/<doc>.md` | 判断该文档必须包含的字段、ID、矩阵、状态、审计项和禁止项 |
| 项目事实大纲 | `docs/<doc>.md` | 承载项目实际内容，按 `【撰写提要：……】` 填写，不作为规则源 |

按用户意图读取标准：

| 用户意图 | 必读标准 | 事实输入 |
|---|---|---|
| 生成整个文档体系 | 已存在的 `ai/doc-standards/00-09`、本文件、`ai/project-rules.md` | 输入材料、`docs/00-09` 大纲、`docs/README.md` |
| 生成 / 审计需求阶段 | `ai/doc-standards/00-scenario.md` 至 `03-prd.md`、本文件 §6 | `docs/00-03`、输入材料 |
| 精修单文档 | 对应 `ai/doc-standards/<doc>.md` | 当前 `docs/<doc>.md` 及上下游影响文档 |
| 阶段评估 | 当前阶段涉及的 doc-standards、`ai/prompts/review/19-docs-evaluation.md` | 当前阶段 docs、open items、验证记录 |

`docs/00-09` 的每个 H2 章节应使用 `【撰写提要：……】` 提醒填写方式；复杂规则必须进入 `ai/doc-standards/`，避免项目事实文档变成规则手册。
## 5. 文档生成矩阵

|---|---|---|---|---|---|

### 5.1 docs/design/* 通用详细设计触发与回写规则

`docs/design/*` 是条件性详细设计文档，不新增 `docs/00-09` 固定编号。生成、修订、审计或评估非平凡详细设计时必须对照 `ai/doc-standards/design-doc.md`。满足以下任一条件时，应新增或补齐 `docs/design/<subsystem>.md`；若豁免，必须在 `ai/project-rules.md` §3、`docs/05-tech-spec.md`、评估报告或待确认事项中说明原因：

- 非平凡子系统、跨模块流程、复杂状态机、导入 / 异步任务或外部集成。
- 复杂 UI、多角色权限、权限 / 安全边界、租户 / 空间隔离、敏感字段或审计要求。
- AI / RAG / 外部模型、OCR、真实第三方服务、回调、队列、限流、重试、真实凭据或高风险愿景能力。
- Mock / 降级 / Demo 与真实能力存在差异，或能力处于候选、默认关闭、禁止当前阶段实现、待 Spike / PoC 状态。

详细设计必须保留元信息、职责边界、上游依据、流程 / 状态机、数据 / 接口 / 权限契约引用、失败 / 降级路径、readiness gate、验证与验收追溯、实现偏差 / 设计回写和结构化待确认项。它只能承接上游已批准需求、架构、技术方案和 `06/07` 契约；不得新增 `03` 未批准需求、`06` 未同步表字段、`07` 未同步接口、未批准验收目标或 Phase 外能力。实现后若代码事实与设计不一致，应回写 `docs/design/*` 的实现偏差 / 设计回写区，并按影响同步 `06/07/08/09`。

### 5.2 UI Brief Intake / 前端交互输入补齐规则

UI Brief Intake 用于在输入评审、需求探索原型、正式前端交互设计或前端实现前补齐 UI / UX 输入，不是正式设计或验收文档。若项目涉及 Web、移动端、小程序、桌面端、可点击 Demo 或复杂页面，但输入材料缺少参考产品、演示主线、页面结构、信息密度、首屏目标、设备范围、视觉禁区、状态反馈或权限可见性，应先补 UI brief，再进入需求探索原型、前端交互设计、UI 原型策略或编码。

UI brief 推荐两类路径：用户原始输入补充放 `docs/inputs/ui-brief.md`；AI 与用户共同探索形成的研究记录放 `docs/research/YYYY-MM-DD-ui-brief-intake.md`。模板见 `template-docs/ui-brief-intake-template.md`。UI brief 中的 AI 推断必须标注为待确认，用户确认后才可回填到 `docs/design/frontend-interaction.md`、UI 原型策略、`docs/08-dev-plan.md` 和 `docs/09-verification.md`；未经确认不得直接写成已确认需求、接口、权限、Sprint 必过项或验收目标。

### 5.3 前端交互设计触发规则

前端交互设计是 `docs/design/*` 的页面 / 交互型子类型，不新增 `docs/00-09` 固定编号，推荐路径为 `docs/design/frontend-interaction.md`；多入口项目可拆成 `docs/design/*interaction*.md`。满足以下任一条件时，开发前应补充该文档，或在 `ai/project-rules.md` §3 / `docs/05-tech-spec.md` 说明豁免理由：

- 交付形态包含独立 Web、移动端、小程序、桌面端或其他可点击 UI。
- `ai/project-rules.md` §3 保留 `frontend/`，且项目交付物包含多页面交互。
- `docs/08-dev-plan.md` 的 Sprint 修改范围包含前端页面、组件、编辑器、搜索 / 问答 UI、管理页或桌面端集成。
- 存在多页面、多角色、多入口、跨页面流程、复杂表单、筛选、列表、详情、审批、状态流或异常处理。
- 验收依赖点击路径、页面状态或用户操作。
- 用户需求、PRD 或愿景中出现“页面 / 界面 / 点击 / 扫码 / 表单 / 看板 / 控制台 / 手机 / H5 / Web / App / 小程序”等交互信号。

前端交互设计应承接 `03/04/05/07/08/09`、`ai/doc-standards/design-doc.md` 和 `ai/doc-standards/frontend-interaction.md`，记录页面信息架构、页面 / 路由清单与 REQ 追溯、核心用户流、组件职责、加载 / 空态 / 错误 / 禁用 / 成功 / 风险状态、表单校验、文案与免责声明、权限可见性、接口依赖、响应式与可访问性、UI 原型与可视化证据、验收路径、阶段增量、readiness gate、实现偏差和待确认项。它不得新增上游未授权功能，不得定义未同步到 `docs/07-api-spec.md` 的接口契约，不得把前端隐藏入口、按钮禁用或路由守卫写成唯一权限边界；权限必须由后端接口和服务层执行。

### 5.4 UI 原型策略触发与边界规则

UI 原型策略是 UI 型项目进入前端实现前的可视化门禁，解决“是否需要先让用户看到、点击或评审界面效果”的问题。满足前端交互设计触发条件，且存在以下任一情况时，必须选择并记录 UI 原型策略；若不需要，必须在 `ai/project-rules.md` §2.7 / §3、`docs/05-tech-spec.md` 或 `docs/design/frontend-interaction.md` 写明豁免理由：

- 用户需要在实现前看到界面效果，或交付物形态为 Demo / MVP 且演示体验影响验收判断。
- 页面信息密度较高，包含列表、详情、搜索、问答、管理页、看板、复杂表单、多栏布局或数据密集界面。
- 主流程依赖点击路径验收，或存在加载、空态、错误、禁用、成功、无权限、降级、风险提示等多状态。
- 存在多角色、多空间、多租户、权限可见性或数据隔离。
- Mock / Demo / 降级能力需要在界面上明确用户可见口径，避免被误读为真实生产能力。

UI 原型策略至少记录：是否需要开发前可视化原型、原型形式（Figma / Penpot / Balsamiq / Axure / Storybook / 代码原型 / 截图标注 / 其他）、原型权威位置、覆盖页面 / 主流程 / 状态 / 设备或浏览器范围、与 `docs/design/frontend-interaction.md` / `08` / `09` 的追溯、未覆盖项和豁免理由。生成、精修、审计或评估该策略时必须对照 `ai/doc-standards/ui-prototype-strategy.md`；需要独立记录时可使用 `template-docs/ui-prototype-strategy-template.md`。工程驱动且已有前端框架的项目可优先采用“代码原型 + Mock 数据 + 截图 / smoke 证据”；需要跨角色协作或沉淀设计系统的项目可优先 Figma / Penpot；早期只需确认布局与流程时可使用低保真草图或截图标注；组件库可组合 Storybook 与设计稿。

原型不替代 `00-09`、不替代前端交互设计、不替代 `09` 验收记录，也不新增需求、接口、表字段、权限规则或验收目标。原型发现的新需求、接口、权限或验收变化必须回到 `02/03/06/07/08/09` 和相关 `docs/design/*` 修订流程；若原型与正式文档冲突，以正式文档为准，先修订文档再进入实现。

### 5.4 04-05 总体设计风险验证规则

`04-05` 是需求链进入详细设计、Sprint 和验证前的总体设计门禁。生成、修订或审计 `04-05` 时必须对照 `ai/doc-standards/04-architecture.md`、`ai/doc-standards/05-tech-spec.md`：

- `04` 必须把 `REQ / NFR → Phase → COMP-ID → MOD-ID → Flow-ID` 串起来，并说明上下文边界、外部系统状态、运行拓扑、ADR、异常 / 降级 / 权限拒绝路径。
- `05` 必须把 `04` 的真实依赖、外部服务、数据库、LLM、Docker / 部署、权限安全和资源约束落到技术状态、依赖配置、Risk-ID、验证方式和 readiness gate。
- `05 ↔ 09` 必须双向映射关键技术风险：`05` 记录 Risk-ID、触发条件、影响、验证方式和解锁条件；`09` 记录 TC、证据路径和验证结论；`08` 记录 Sprint 前置条件和禁止事项。
- `候选`、`已验证`、`已启用`、`默认关闭`、`Mock`、`降级`、`禁止` 等状态不得混用；状态变化必须传播到 `04/05/08/09` 和相关详细设计。
- Readiness gate 条件触发：真实外部服务、数据库、LLM、Docker / 部署、重型 SDK、权限安全或真实数据处理进入 Sprint / Phase 前，必须有 Go / Conditional Go / No-Go 结论；No-Go 阻止相关实现，Conditional Go 必须列限制条件和补做时点。

### 5.5 06-07 DB / API 契约状态与升阶段门槛

`06-07` 是详细设计进入实现计划、验证和 Phase 升级前的契约门禁。生成、修订或审计 `06-07` 时必须对照 `ai/doc-standards/06-db-design.md`、`ai/doc-standards/07-api-spec.md`：

- `06` 必须把 `REQ / NFR / Phase → COMP / MOD / Flow → 数据对象 → 表 / 字段 / 约束 → 迁移 / seed / 回滚 → TC / Sprint` 串起来，并说明目标结构、当前实现、Mock / Demo 差异和数据安全留存。
- `07` 必须把 `REQ / NFR / Phase → API-ID → 请求 / 响应 / 错误 / 权限 / 兼容 → DB / Service / Test` 串起来，并为当前 Phase API 提供 endpoint contract matrix。
- `06 ↔ 07 ↔ 09` 必须映射敏感字段、脱敏 / 过滤、权限边界、DB 约束到 API 错误码、API-ID 到 TC-ID；权限隔离不得只依赖前端隐藏 / 禁用。
- `目标设计`、`当前实现`、`Mock`、`已验证`、`草案`、`候选`、`默认关闭`、`禁止` 等契约状态不得混用；状态变化必须传播到 `06/07/08/09` 和相关实现任务。
- Phase 升级或 Mock → 真实能力切换前，当前 Phase 表 / API 必须补齐字段级契约、endpoint contract、迁移 / seed / 回滚、鉴权权限、错误可见性、兼容策略、数据留存和验证项；缺失时不得建议直接升级或开始相关实现。

## 6. 追溯链规则

```text
上游输入锚点（愿景场景 / 需求条目 / SRS 条款 / PRD 条款 / 任务单 / 现有系统事实 / 外部接入文档）
  → U-ID（用户需求）
  → REQ-ID（系统需求）
  → Phase（03 路线图）
  → 架构模块 / 技术决策 / 数据表 / API / 子系统设计
  → Sprint / task
  → 测试用例
  → 代码 / 测试实现
```

要求：

- `01` 中每个 U-ID 必须能回到愿景锚点、人工输入来源或反向摘要来源。
- `02` 中每个 REQ-ID 必须能回到一个或多个 U-ID。
- `03` 必须覆盖全部 REQ-ID，并给出阶段归属。
- `04-07` 中新增的模块、技术、表、接口、子系统必须能追溯到 REQ-ID 或明确的非功能约束。
- `08/09` 必须覆盖当前阶段 REQ-ID；远期 REQ 可保留骨架，不强行细化。
- 代码和测试必须能追溯到 task / Sprint / REQ。

### 6.0.1 00-03 需求链健康度

`00-03` 的最低健康链为 `SC-ID → U-ID → REQ-ID → Phase → AC / TC`。AI 做生成、评估、审计或 Phase 升级时，应能输出或检查该链路的健康度矩阵。

| 链路段 | 必须成立 | 常见断点 |
|---|---|---|
| 输入 / 愿景 → SC-ID | 场景有来源锚点、可信度、边界和非目标 | 只有背景叙述，无可追溯场景 |
| SC-ID → U-ID | 每个用户需求能回到场景或人工输入 | U-ID 无来源或只来自 AI 推断 |
| U-ID → REQ-ID | 每个 REQ 覆盖一个或多个 U-ID，并保留用户 AC | 功能需求无用户价值或无验收口径 |
| REQ-ID → Phase | 每个 REQ 在 `03` 有阶段归属、交付物形态和状态 | 当前 Phase 范围不清或愿景功能误入当前阶段 |
| Phase → AC / TC | 当前 Phase 的 REQ 有可观察验收或测试入口 | 进入 Sprint 前没有验收证据路径 |

健康度矩阵默认输出在 `docs-evaluation` 或 `docs-system-audit` 报告中，不强制所有项目写入正式 `03` 或 `09`。Lean 项目至少要列 P0 / P1 断点、阻塞关系和修复顺序。

兼容补齐规则：旧项目使用 `F-*` 等自定义编号时，不强制全文重命名；优先新增兼容映射表（如 `U-ID ↔ 旧编号`、`REQ-ID ↔ 旧功能编号`）来闭合追溯。

## 6.1 待人工确认项结构

正式文档、任务单和可续接状态文件中的“待人工确认项”不得只是问题列表。除临时对话外，建议使用以下最低结构：

| 字段 | 说明 |
|---|---|
| ID | 稳定编号，如 `C-001`，便于回填和续接。 |
| 待确认项 | 需要用户确认的问题。 |
| AI 建议 | AI 推荐的默认口径或优先选项；必须标明是建议，不是事实。 |
| 建议依据 | 来自上游文档、阶段目标、资源约束、风险边界、验证结果或通用经验的理由。 |
| 备选方案 | 至少列出可选路径；没有可行备选时说明原因。 |
| 取舍影响 / 阻塞关系 | 说明成本、风险、后续影响，以及是否阻塞当前 Sprint / Phase；不阻塞时写可延后时点。 |

约束：
- AI 可以给“临时建议 + 需补充的信息”，但不得把临时建议写成用户已确认事实。
- 高风险、合规、成本、账号权限、数据安全、生产部署等事项即使有 AI 建议，也必须保留人工确认。
- 用户确认后，必须把确认结果回填到权威文档或续接文件，并将原待确认项标为“已确认”或移出待确认区。

## 6.2 待确认事项总览

当一次生成、审计、评估或回梳涉及多份核心文档时，AI 必须输出“待确认事项总览”，避免待确认项散落在 00-09、`docs/design/*`、任务单和续接文件中无法闭环。

总览至少包含以下字段：

| 字段 | 说明 |
|---|---|
| ID | 稳定编号，建议带来源前缀，如 `DOC-C-001`、`B1-C1`。 |
| 提出时间 | 使用绝对日期；无法确认时写“待确认”，不得猜测。 |
| 来源文档 / 位置 | 待确认项来自哪个文件、章节或审计发现。 |
| 待确认项 | 需要人工确认的问题。 |
| AI 建议 | 推荐口径；不得写成已确认事实。 |
| 建议依据 | 上游输入、权威源、风险、验证证据或状态词典依据。 |
| 备选方案 | 可选处理路径。 |
| 取舍影响 | 成本、复杂度、阶段范围、验证、合规、用户体验或维护影响。 |
| 需确认节点 | 如生成 03 前、进入 04 前、生成 08 前、编码前、Phase 升级前、发布前。 |
| 阻塞关系 | 阻塞 / 条件阻塞 / 不阻塞。 |
| 回填位置 | 确认后应回填的 `docs/`、`docs/design/`、`tasks/`、`ai/project-rules.md` 或 README。 |
| 当前状态 | 待确认 / 已确认 / 暂缓 / 关闭 / 转任务。 |
| 关闭依据 | 人工确认记录、文档回填位置、PR / commit / 会议记录。 |

门禁语义：

- P0 / 阻塞项不得被埋在单文档尾部，必须进入总览并说明停止原因。
- P1 项必须说明应在进入哪个阶段、Sprint 或 PR 前解决。
- P2 项可延后，但必须给出建议回看时点。
- 进入设计阶段前，必须检查是否存在阻塞 `00-03` 的待确认项。
- 进入详细设计 / 计划阶段前，必须检查技术选型、接口 / DB 契约、交互方案、验证策略是否仍有阻塞项。
- 编码前和 Phase 升级前必须检查 open items；阻塞项未关闭、未转任务、未回填权威文档或未被明确风险接受时，不得开始对应 Sprint 或直接升级。
- 用户确认后，应回填权威文档或续接文件，并从总览中移除或标记已确认。
- 默认建议落盘到 `docs/research/YYYY-MM-DD-docs-open-items.md`；若项目选择长期固定入口 `docs/open-items.md`，必须在 `docs/README.md` 说明其定位。open items 只做总览，不替代事实文档。

## 7. 横切事实与权威源

横切事实是指会影响多个不相邻文档的事实、结论或裁决，例如平台能力结论、合规裁决、技术选型禁令、可行性核实结果、资源红线、对外交付口径。

规则：

- 横切事实必须有唯一权威定义点，通常放在 `docs/decisions/` 的决策记录，或 `ai/project-rules.md` 的允许 / 禁止清单。
- 其他文档引用权威源（如“见 ADR-X / DEC-X / `project-rules` §Y”），不得各自重新声明事实本身。
- 权威源变更时，必须检索所有引用处并同步引用、状态和影响描述。
- 若同一事实在多个文档中出现不同表述，先建立或修正权威源，再合并引用口径。

## 7.1 横切状态词典

以下状态词用于 00-09、`docs/design/*`、审计、评估、任务计划和续接文件。AI 不得混用状态，不得把较低确定性的状态写成较高确定性事实。

| 状态 | 含义 | 写作要求 |
|---|---|---|
| 目标设计 | 计划达到的设计目标，尚不代表已经实现或验证 | 标明来源需求、适用阶段和验证入口。 |
| 草案 | 临时起草内容，尚未完成审查 | 标明待审查项，不得作为开发唯一依据。 |
| 候选 | 可能采用的方案、依赖、接口或需求 | 列选择条件、备选方案和决策触发点。 |
| 待人工确认 | AI 无法替用户决定的事实、范围、成本、权限或风险 | 使用 §6.1 结构，AI 建议不得写成已确认事实。 |
| 待技术验证 | 技术可行性、性能、资源或集成方式尚未实证 | 标明验证方式和进入下一阶段的门槛。 |
| Mock | 为演示、测试或未接入依赖而模拟的数据 / 服务 / 行为 | 标明替换为真实实现的触发条件，不得声称已接入真实系统。 |
| 降级 | 主路径不可用时的备用方案或功能收缩 | 标明触发条件、用户影响和恢复条件。 |
| 默认关闭 | 能力存在或预留，但默认不启用 | 标明启用条件、配置入口和未启用时行为。 |
| 预留 / 未启用 | 设计或代码留有扩展点，但当前阶段不使用 | 不得计入当前阶段验收成果。 |
| 已验证 | 已通过指定验证方式得到证据 | 写明证据位置、验证时间或测试用例。 |
| 已启用 | 已在当前交付物默认启用或生产 / 演示路径可用 | 写明启用范围、配置和回退方式。 |
| 禁止 | 项目或模板明确不允许的做法 | 标明权威来源和替代方案。 |

状态传播规则：

- 状态变化时，必须检查 `docs/03-prd.md`、`ai/project-rules.md`、`docs/04-09`、`docs/design/*`、`docs/README.md`、任务单和验证文档的下游影响。
- `候选 / 草案 / 待确认 / 待技术验证 / Mock / 降级 / 默认关闭 / 预留` 不得在下游被改写成 `已验证 / 已启用`，除非有验证证据和人工确认。
- `已验证` 不等于 `已启用`；验证证据只证明通过某项检查，是否启用仍需看阶段、配置和交付范围。
- `Mock` 或 `降级` 参与演示时，验收文档必须说明真实能力缺口和后续替换条件。

## 8. 外部文档接入规则

运行中接入外部文档（非从本仓库 docs 生成的策略、调研、决策、会议、接口或客户输入文档）时，必须先完成锚定，不得让外部文档成为无引用孤岛。

接入要求：

1. **顶部锚定说明**：写明外部文档命名与本仓库 docs 的映射关系。
2. **定位声明**：说明文档类型（策略 / 调研 / 决策 / 会议 / 接口 / 需求输入）以及它驱动或约束哪些 docs。
3. **纳入追溯链**：其结论若影响 `00-09`、`docs/design/*`、`tasks/*` 或 `ai/project-rules.md`，必须触发影响分析。
4. **分区归属**：按 `docs/README.md` 放入对应子目录；策略 / 决策类进入 `docs/decisions/`，调研 / 实验类进入 `docs/research/`，会议或访谈进入 `docs/meetings/`，不要误放 `docs/vision/`。

## 9. 变更传播规则

变更分为两类，传播义务不同：

| 变更类型 | 定义 | 传播义务 |
|---|---|---|
| 局部变更 | 单一需求、契约、字段、文案或章节的增删改，影响范围有限 | 先声明影响范围和同步计划；若需要连带改其他文档，等待人工确认后处理 |
| 横切事实 / 约束变更 | 一个事实、结论或裁决变化，可能影响多个不相邻文档 | 主动列出受影响文档清单、引用同步计划和一致性验证项；不得只改单点文档 |

上游变更影响矩阵：

| 变更对象 | 必须评审 / 影响分析 |
|---|---|
| `docs/vision/*` | `00-03`，再评估 `04-09` 与 `docs/design/*` 是否受影响 |
| 其他上游输入包 | 先判断入口模式，再评审其下游文档链路是否需要补齐或重生成 |
| 外部接入文档 | 锚定映射、权威源、被驱动 / 约束的 docs、引用同步范围 |
| 横切事实权威源 | 全部引用处、相关 `project-rules` 约束、`03-09` 影响和验证口径 |
| `docs/00-scenario.md` | `01-03`，以及当前阶段演示场景 / 验证夹具 |
| `docs/01-user-requirements.md` | `02/03`，新增或删除 U-ID 时必须评估 REQ 编号和阶段路线图 |
| `docs/02-srs.md` | `03-09`，新增或删除 REQ 时必须更新追溯链 |
| `docs/03-prd.md` | `project-rules §1`、`04-09`、`tasks/*`、Phase 状态和验收引用 |
| `ai/project-rules.md` | `03-09`、代码目录结构、运行环境文档 |
| `docs/04-architecture.md` | `05-09`、`docs/design/*`、代码结构 |
| `docs/05-tech-spec.md` | `06-09`、依赖文件、运行脚本、部署配置 |
| `docs/06-db-design.md` | API、service/model、迁移、测试数据 |
| `docs/07-api-spec.md` | API 层、前端调用、集成测试 |
| `docs/design/*` | `08/09`、相关实现任务 |
| `docs/08-dev-plan.md` / `tasks/*` | 当前实现范围和测试计划 |
| code / tests | 若代码事实偏离文档，必须走文档反向同步；若代码超出已批准需求，优先标记越界风险，不得直接把越界事实写入需求文档 |

横切事实 / 约束变更完成后，必须做一次聚焦的跨文档一致性检查，输出受影响文档清单、各项一致性核对结果和残留待确认项。

## 10. AI 生成前声明

AI 生成或修改任何项目事实文档前，先输出简短声明：

```text
本次输入文档：...
入口模式与文档剖面：...
本次约束文档：...
本次变更类型：局部变更 / 横切事实或约束变更 / 新增文档体系
本次允许修改范围：...
本次不得新增内容：...
预计下游影响：...
```

若上游输入不足，AI 必须停下并列出待确认项，不得用猜测补齐。

### 10.1 文档体系生成总控最低规则

当用户说“生成整个文档体系”“补齐全部文档”“铺完整 docs”或类似需求时，AI 不得直接写 00-09。必须先输出阶段路线和生成模式选择：

阶段路线：

1. 输入材料评审。
2. 需求确认。
3. 需求分析。
4. 总体设计。
5. 技术选型 / 技术路线评估。
6. 详细设计。
7. 验证用例设计。
8. 执行计划 / Sprint 规划。
9. 完整文档体系评估与审计。
10. 待确认事项总览与编码前门禁。

生成模式：

- 分阶段确认模式：默认推荐；每阶段确认后再生成下一阶段。
- 输入充分后批量生成模式：适合快速起草；必须先做输入充分性评估，生成后必须做完整评估 / 审计。

无论选择哪种模式，都必须说明入口模式、文档剖面、预计写入文件、待确认事项总览位置和后续审计方式。

### 10.2 需求探索原型边界

需求探索原型用于在正式 `00-03` 定稿、架构和技术路线选择前，用可视化方式帮助用户确认系统边界、页面结构、主流程、信息密度、文案方向和关键状态。它适合触发于“我想先看原型 / 先做页面原型确认需求 / 先别定技术栈，先把界面流程画出来 / Demo 前先确认交互”等场景。

需求探索原型默认输出为探索报告或原型记录，推荐路径为 `docs/research/YYYY-MM-DD-ui-prototype-exploration.md`，结构可参考 `template-docs/ui-prototype-exploration-template.md`。它不是 `docs/00-09` 的替代品，也不是 `docs/design/frontend-interaction.md` 或 UI 原型策略的替代品；确认前不得写成已确认需求、架构、接口、数据表、技术栈、任务或验收通过证据。

约束：

- 原型状态必须标为“探索 / 待确认 / 候选”，并列出待确认假设、用户反馈和建议回填位置。
- 原型可使用低保真草图、Figma、Penpot、截图标注、HTML 静态页、代码静态 Mock 或其他轻量形式；不得因工具选择提前锁死架构、技术栈、接口或数据库。
- 用户确认后的内容必须回填到 `00-03`，再进入 `04-05` 架构与技术路线、`docs/design/*` 详细设计、`08` 计划和 `09` 验证；未确认内容只能进入 open items 或继续探索。
- 若需求探索原型发现新需求、角色、流程、权限、接口或验收目标，必须先更新需求链或待确认事项，不得直接进入编码。

### 10.3 专题方案讨论边界

需求层人机交互、总体设计 / 技术选型、交互设计方案等专题讨论，不等于正式文档事实。AI 必须先输出多方案、依据、适用条件、风险、取舍影响、AI 推荐、待确认项和建议回填位置。

约束：

- 人工确认前，专题讨论结果只能标为“候选 / 待人工确认 / 待技术验证”，不得写成已确认需求、架构、接口、交互或任务。
- 人工确认后，必须通过 `edit-single-doc` 或对应文档流程回填权威文档，并更新或关闭相关 open item。
- 交互设计讨论不得新增 PRD 未授权需求、API 未定义接口或 09 未覆盖验收目标；前端可见性控制不得替代后端权限边界。
- 技术选型讨论若依赖真实运行能力，必须给出 Spike / PoC / 技术环境评估入口和进入下一阶段的门槛。

## 12. 禁止项

- 不得凭空新增需求、模块、表、接口、依赖或测试目标。
- 不得强制所有项目从愿景文档起步；但跳过愿景时必须声明替代输入来源与可信度。
- 不得因当前 Phase 裁剪掉愿景中的功能；应保留为后续阶段或愿景骨架。
- 不得把愿景功能直接塞进当前阶段，除非 `03-prd.md` 和 `project-rules §1` 已明确批准。
- 不得留下悬空 ID：无来源的 U-ID、无 U-ID 的 REQ、无 REQ 的模块 / 表 / 接口 / Sprint / 测试用例。
- 不得把候选、预留、默认关闭、Mock 或未实现能力写成已接入 / 已实现。
- 不得只修改上游文档而不评估下游影响。
- 不得把横切事实散落到多个文档中各自定义；必须先建立唯一权威源，再由其他文档引用。
- 不得让外部接入文档成为无映射、无定位、无追溯关系的孤岛。
- 不得把由下游文档反向摘要得到的 `00/01` 内容伪装成原始用户输入。

## 13. 设计文档图表规范

设计类文档应包含规定图表，让架构、数据、接口、流程可视化、可对照；格式统一便于阅读和渲染。

**适用文档与应含图表**：

| 文档 | 应含图表 |
|---|---|
| `docs/04-architecture.md` | 系统架构图、运行拓扑 / 部署图 |
| `docs/05-tech-spec.md` | 技术栈分层、部署拓扑 |
| `docs/06-db-design.md` | ER 图 |
| `docs/07-api-spec.md` | 接口交互 / 时序图 |
| `docs/design/*` | 流程图、状态机、交互图 |

**格式**：默认 `mermaid`（GitHub 原生渲染），可选 `plantuml`；项目可在 `ai/project-rules.md §2.6` 覆盖默认。

**性质**：「建议 + 默认」而非强制——图表服务于表达，不要求每类文档必须凑齐所有图；但涉及架构 / 数据 / 接口 / 关键流程的设计文档应有对应图表。生成或精修设计文档时（见 `template-docs/scenario-guides.md` A7），AI 应按本节出图，并提示用户确认格式偏好。
