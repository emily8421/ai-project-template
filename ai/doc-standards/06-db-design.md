本文件是 `docs/06-db-design.md` 的细粒度标准，用于 AI 生成、修订、审计和评估数据库设计文档。它不是项目事实文档，派生项目的实际数据设计事实仍写入 `docs/06-db-design.md`。

## 1. 定位与边界

`06` 仅在项目涉及持久化数据库时保留；浏览器端 `localStorage` / `IndexedDB`、Mock / fixture、内存仓储或无数据库项目可按 `ai/project-rules.md` §3 省略或轻量说明替代数据结构。

- `06` 承接 `02/03` 的 REQ / NFR / Phase，`04` 的组件、模块、Flow 和数据边界，`05` 的数据库、权限、安全、Mock / 降级和 readiness gate 约束。
- `06` 输出数据对象、概念模型、表字段、索引约束、迁移 / seed / 回滚、数据安全留存和 REQ → 表 / 字段追溯。
- 不得创建无 REQ、NFR、架构约束或人工确认来源的表、字段、索引或迁移。
- 不得把目标结构、远期表、Mock / Demo 数据、内存仓储或未迁移数据库写成当前真实实现。

## 3. 契约状态与实现依据

`06` 必须引用 `ai/document-lifecycle-rules.md` §7.1 的横切状态词典，并在 DB 场景补充是否可作为实现依据：

| 状态 | DB 场景含义 | 是否可作为实现依据 |
|---|---|---|
| 目标设计 | 未来方向或完整目标结构 | 不可直接视为当前已实现 |
| 当前实现 | 当前 Phase 实际使用的数据存储或表结构 | 可以作为当前范围实现依据 |
| Mock | 仅模拟数据、fixture、payload 或内存仓储 | 只能作为 Demo / 测试依据 |
| 已验证 | 已通过迁移、seed、查询、集成或验收验证 | 可按验证范围使用 |
| 草案 | 字段、约束、迁移或留存策略尚未冻结 | 需要补齐后再作为稳定契约 |
| 候选 | 方向性存储方案或字段方案，未验证 | 不得进入实现，除非先做 Spike / PoC |
| 默认关闭 | 数据能力存在或预留，当前不开启 | 不得在当前阶段默认启用 |
| 禁止 | 当前阶段不得接入的存储、字段或数据流 | 实现或测试不得绕过 |

最低规则：

- Demo 阶段允许“目标存储”和“当前实现”不同，但必须在目标结构与当前实现对照矩阵中明示。
- 若当前实现不是数据库，不应把数据库表写成已迁移或已启用。
- 若进入 MVP / 试点，当前 Phase 表必须补迁移状态、seed / fixture 来源和验证项。
- 即使无真实数据库，Mock / fixture 数据来源也应可追溯。

## 5. DB / API 交叉追溯

`06` 应为 `07` 提供字段来源、权限、错误和测试映射，至少支持以下交叉检查：

| API-ID | 请求 / 响应字段 | 数据来源 | 表 / 字段 | 计算逻辑 | 敏感性 | 脱敏 / 过滤 |
|---|---|---|---|---|---|---|

| DB / 业务约束 | 触发接口 | 错误码 | 用户可见信息 | 是否可重试 | 日志 / 审计 |
|---|---|---|---|---|---|

最低规则：

- API 输出敏感字段时，必须能追溯到数据来源、脱敏 / 过滤策略和权限规则。
- DB 唯一约束、外键约束、业务约束或迁移失败应映射到 API 错误或运维处理口径。
- 权限隔离不得只依赖前端隐藏 / 禁用，必须在 API / Service / DB 查询边界体现。

## 6. 审计检查

- 是否每张表、字段、索引、约束和迁移都有 REQ / NFR / 架构约束或人工确认来源。
- 是否区分目标设计、当前实现、Mock / Demo 差异、已验证和已启用范围。
- 是否为当前 Phase 或即将进入 Phase 的表补齐字段级契约、迁移 / seed / 回滚和验证入口。
- 是否补齐敏感字段、访问控制、脱敏 / 加密、留存 / 删除和外部传输限制。
- 是否与 `07`、`08`、`09` 建立 API-ID、错误码、TC-ID 和 Sprint 映射。

## 7. 下游影响

- 给 `07`：请求 / 响应字段来源、权限过滤、错误映射、状态机和敏感字段策略。
- 给实现：模型、迁移、seed / fixture、查询逻辑、回滚和数据修复边界。
- 给 `08` / `09`：DB 相关 Sprint 前置条件、TC、验证入口和 Phase 升级门槛。
