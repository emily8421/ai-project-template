﻿# TEMPLATE-UPGRADE: 新增 UI 原型策略与可视化验收门禁

> 来源：LUMEN_demo_T2.1（emily8421/LUMEN-DEMO）派生项目回流

## 1. 背景与问题

模板已经有前端交互设计触发规则，要求 UI 型项目在开发前补充 `docs/design/frontend-interaction.md` 或同类交互设计文档。该规则解决了“页面流、状态、文案、接口依赖和验收路径写在哪里”的问题，但仍缺少一个更早、更直观的门禁：项目是否需要先产出一个可被用户看到、点击或评审的原型。

在大量派生项目里，前端 UI / UX 的风险并不只来自需求缺失，也来自以下情况：

1. 交互设计文档写得完整，但用户直到实现后才第一次看到界面效果。
2. 文字描述中的页面结构、状态反馈和信息密度，在真实屏幕中才暴露问题。
3. Demo / Mock / 降级能力没有在界面原型中明确表达，容易误导用户以为真实能力已接入。
4. 设计稿、截图、代码原型、Storybook、Figma / Penpot 等工具之间没有模板级选择口径，导致项目临时决定，AI 也难以稳定评审。
5. UI 型 Sprint 的验收依赖点击路径，但缺少“原型链接 / 截图 / 可运行 mock”作为前置证据。

这些问题不依赖具体业务。凡是包含独立 Web、移动端、小程序、桌面端、多页面管理台、搜索 / 问答 UI、复杂表单、审批流、看板或数据密集界面的项目，都需要在正式实现前明确原型策略。

## 2. 目标

新增模板级“UI 原型策略”规则，使 UI 型项目在进入前端实现前至少完成以下判断：

1. 是否需要可视化原型。
2. 使用哪种原型形式：Figma、Penpot、Balsamiq、Axure、Storybook、代码原型、截图标注或其他工具。
3. 原型覆盖哪些主流程、页面状态和设备 / 浏览器范围。
4. 原型与 `docs/design/frontend-interaction.md`、`docs/08-dev-plan.md`、`docs/09-verification.md` 的追溯关系。
5. 原型不得绕过文档驱动开发，不得新增未授权需求、接口、权限行为或验收目标。

建议原则：模板应强制“选择并记录原型策略”，但不强制所有项目使用 Figma。

## 4. 建议规则

### 4.1 UI 原型策略触发条件

当项目满足前端交互设计触发条件，且存在以下任一情况时，应选择并记录 UI 原型策略：

- 用户需要在实现前看到界面效果。
- 页面信息密度较高，包含列表、详情、搜索、问答、管理页、看板、表单或多栏布局。
- 主流程依赖点击路径验收。
- 存在多状态：加载、空态、错误、禁用、成功、无权限、降级、风险提示。
- 存在多角色、多空间、多租户、权限可见性或数据隔离。
- 交付物形态为 Demo / MVP，且演示体验本身会影响验收判断。

### 4.2 项目级策略字段

建议在 `ai/project-rules.md` 模板源中加入类似字段：

```markdown
## 2.7 UI 原型策略（如适用）

- 是否涉及可点击 UI：是 / 否
- 是否需要开发前可视化原型：需要 / 不需要 / 豁免
- 原型形式：Figma / Penpot / Balsamiq / Axure / Storybook / 代码原型 / 截图标注 / 其他
- 原型权威位置：链接或仓库路径
- 原型覆盖范围：主流程 / 页面状态 / 响应式范围 / 权限与降级状态
- 原型与文档关系：承接 `docs/design/frontend-interaction.md`，不得新增未授权需求、接口或验收目标
- 豁免理由：仅当不需要原型时填写
```

### 4.3 前端交互设计中的原型章节

建议 `docs/design/frontend-interaction.md` 或 `docs/design/*interaction*.md` 增加如下章节：

```markdown
## X. UI 原型与可视化证据

| 项 | 内容 |
|---|---|
| 原型形式 | Figma / Penpot / Balsamiq / Axure / Storybook / 代码原型 / 截图 |
| 原型位置 | URL / 仓库路径 / 截图目录 |
| 覆盖页面 |  |
| 覆盖主流程 |  |
| 覆盖状态 | 加载 / 空态 / 错误 / 禁用 / 成功 / 权限 / 降级 |
| 设备 / 浏览器范围 |  |
| 与 REQ / Sprint / TC 追溯 |  |
| 未覆盖项 |  |
```

约束：

- 原型只表达已授权需求的界面与交互，不作为新增需求来源。
- 原型发现的新需求、接口、表字段、权限规则或验收目标，应回到 02 / 03 / 06 / 07 / 08 / 09 修订流程。
- 前端隐藏、按钮禁用、路由守卫在原型中只能作为可见性表现，不能替代后端权限边界。

## 5. 原型形式选择建议

模板不应绑定某个工具。建议给出工具选择矩阵：

| 原型形式 | 适用场景 | 产出 | 注意事项 |
|---|---|---|---|
| Figma | 高保真 UI、团队协作、设计交付、设计系统、AI 读取设计上下文 | 设计稿、可点击原型、Dev Mode 交付 | 需要账号、文件管理和一定学习成本 |
| Penpot | 偏开源、自托管、工程团队协作、设计系统 | 设计稿、原型、设计资产 | 适合希望降低闭源工具绑定的团队 |
| Balsamiq | 早期低保真线框、快速确认页面结构和流程 | Wireframe / 低保真原型 | 不适合精细视觉规范 |
| Axure | 复杂交互、条件逻辑、动态面板、企业后台流程 | 高交互原型 | 学习和维护成本较高 |
| Storybook | 已有前端组件，需展示组件状态和交互变体 | 组件文档、状态样例、可视化回归入口 | 偏组件级，不替代整页用户流 |
| 代码原型 | 工程驱动项目、希望直接运行 mock UI | 可运行页面、mock 数据、截图 / smoke 证据 | 需明确 mock 与真实能力边界 |
| 截图标注 | 已有系统改造或轻量 UI | 截图 + 批注 + 决策记录 | 适合低成本复核，不适合复杂新流程 |

### 5.1 默认推荐

建议模板默认推荐：

- 工程驱动、已有前端框架的项目：优先“代码原型 + mock 数据 + 截图 / smoke 证据”。
- 需要跨角色协作或沉淀设计系统的项目：优先 Figma 或 Penpot。
- 只需快速确认布局和流程的早期阶段：Balsamiq / 低保真草图即可。
- 组件库或设计系统建设：Storybook 与 Figma / Penpot 可组合使用。

## 6. 与现有文档链路的关系

```text
02/03 已授权 REQ / Phase
  → 04/05 前端架构与技术边界
  → docs/design/frontend-interaction.md 页面流 / 状态 / 文案 / API
  → UI 原型策略：Figma / Penpot / 代码原型 / Storybook / 截图
  → 08 前端 Sprint / Task
  → 09 浏览器 smoke / 视觉状态 / 验收记录
  → code / tests
```

关键边界：

1. 原型不替代 00-09，不新增需求。
2. 原型不替代 `docs/design/frontend-interaction.md`，但可以作为其可视化证据。
3. 原型不替代 09 验收；实现完成后仍需浏览器 smoke、截图、日志或测试记录。
4. 若原型与文档冲突，以文档为准；若原型发现文档不足，先修订文档。

## 7. Prompt 增强建议

### 7.1 `generate-docs`

当项目触发 UI 条件时，生成或提示：

- 是否需要 `docs/design/frontend-interaction.md`。
- 是否需要 UI 原型策略。
- 推荐原型形式和理由。
- 原型覆盖范围与豁免理由。

### 7.2 `edit-single-doc`

补前端交互设计时检查：

- 是否记录原型形式和位置。
- 是否覆盖主流程、关键状态、权限 / 降级状态。
- 是否有未覆盖项和待确认项。
- 原型是否越过上游需求或接口边界。

### 7.3 `docs-checklist` / `docs-system-audit`

增加检查项：

- UI 型项目是否已选择原型策略或写明豁免。
- 原型证据是否可访问、可复核。
- 原型是否与 03 / 07 / 08 / 09 追溯一致。
- 是否存在“只看到了最终实现，开发前无原型证据”的风险。

### 7.4 `docs-evaluation`

E3 / E4 评估建议增加：

> 对于 UI 型项目，是否已有足以支持前端 Sprint 的可视化原型或明确豁免？如果没有，是否会导致用户体验决策在编码阶段临场发生？

## 8. 兼容与迁移建议

对已存在派生项目，不建议强制补完整 Figma 或设计系统。建议渐进迁移：

1. 先在 `ai/project-rules.md` 或 `docs/05-tech-spec.md` 记录原型策略。
2. 当前正在开发的 UI Sprint，至少补一份代码原型、截图标注或 Figma / Penpot 链接。
3. 在 `docs/design/frontend-interaction.md` 中补“UI 原型与可视化证据”章节。
4. 在 `docs/09-verification.md` 中把原型覆盖的主流程转成浏览器 smoke 或人工验收 TC。
5. 对不需要原型的 Lean / CLI / 纯后端项目，写明豁免理由即可。

## 9. 验证方式与验收标准

建议模板维护者落地后用以下方式验证：

1. 用一个带独立 Web / 桌面端 UI 的样例项目运行 `generate-docs`，应能提示选择 UI 原型策略。
2. 用一个 UI 型项目运行 `docs-checklist` 或 `docs-system-audit`，若缺少原型策略且无豁免，应能稳定报告缺口。
3. 用一个仅 CLI / 纯后端样例项目运行同类检查，应允许记录“不适用 / 豁免”，不强制创建原型。
4. 用一个已有前端交互设计文档的样例项目修订 `docs/design/frontend-interaction.md`，应能补出原型形式、位置、覆盖状态和未覆盖项。
5. 检查所有新增规则都明确：原型不新增需求，不替代 00-09，不替代后端权限边界。

本提案落地后，模板应满足：

1. UI 型项目在文档生成或前端 Sprint 前，会被提示选择 UI 原型策略。
2. 模板不会强制绑定 Figma；可选择 Figma、Penpot、Balsamiq、Axure、Storybook、代码原型、截图标注或豁免。
3. 前端交互设计文档能记录原型链接 / 路径、覆盖页面、覆盖状态和未覆盖项。
4. 审计和文档评估能发现 UI 项目缺少可视化原型证据的问题。
5. 原型发现的新需求或接口变更会被引导回 00-09 / design 文档链路，而不是直接进入实现。
6. Demo / Mock / 降级能力在原型中有明确用户可见口径，不会被误读为真实生产能力。

## 10. 版本影响

建议版本：模板方法论 minor 版本升级。

影响面：

- 对新项目：UI 型项目的前期工作稍重，但能减少实现后返工。
- 对旧项目：同步后不强制补原型；可在下一次 UI Sprint 或 Phase 升级时按需补齐。
- 对 Prompt：生成、修订、审计和评估时增加原型策略检查项。
- 对工具：不新增强依赖；Figma / Penpot / Storybook 等仅作为可选策略。

## 11. 非目标

- 不要求所有项目使用 Figma。
- 不要求所有 UI 项目做高保真视觉设计。
- 不要求 Lean / CLI / 纯后端项目产出原型。
- 不把原型作为需求权威源。
- 不引入具体派生项目的页面、文案、数据、账号、路径或业务流程。
