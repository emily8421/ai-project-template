﻿# TEMPLATE-UPGRADE: 新增 `docs/design/*` 通用详细设计文档规范

> 来源：LUMEN_demo_T2.1（emily8421/LUMEN-DEMO）派生项目回流
> 依据：`docs/research/2026-07-06-template-proposal-audit-batch-5-design-docs.md` Batch 5 审计报告

## 1. 背景与问题

派生项目对 `docs/design/*` 与当前 `ai/doc-standards/README.md`、`ai/document-lifecycle-rules.md` 做 Batch 5 只读审计时发现：模板已经定义了前端交互设计基线，也要求详细设计承接 `04/05/06/07/08/09`，但缺少一个通用的 `docs/design/*` 详细设计文档标准。

当前派生项目中的详细设计文档已覆盖前端交互、内容导入、权限、RAG 检索、术语管理和愿景情报分析等多个子系统，但结构差异较大。典型问题包括：

1. 多数 `docs/design/*` 缺少统一 `0. 文档元信息`，输入来源、覆盖 REQ、关联 04-09、当前状态、最后更新和下游影响不稳定。
2. 职责、边界、不做事项的结构不统一，详细设计中容易越过 PRD / API / Phase 边界。
3. 流程、状态机、失败 / 降级路径的要求不一致，外部服务、AI、OCR、异步任务、权限拒绝等场景容易遗漏。
4. `docs/design/*` 与 `06` 表字段、`07` API-ID、`09` TC-ID、`08` Sprint / Task 的追溯表不统一。
5. UI、权限、安全、AI/RAG、导入 / 异步任务、高风险愿景能力等不同类型的详细设计需要不同 checklist，但模板目前只有前端交互基线。
6. Mock / 降级 / Demo 与真实能力之间的差异缺少固定记录位，容易在实现后产生文档漂移。
7. 待人工确认项容易退化为自由文本，缺少 AI 建议、依据、备选方案和阻塞关系。

这些问题不依赖具体业务项目。凡是存在非平凡子系统、复杂 UI、权限边界、AI / 外部服务、导入 / 异步任务或高风险愿景能力的派生项目，都需要通用 `docs/design/*` 模板规范。

## 2. 目标

新增或强化 `docs/design/*` 通用详细设计规范，使模板能稳定支撑以下链路：

```text
02 REQ / NFR / 边界场景
  → 03 Phase / 非目标 / 交付物形态
  → 04 模块 / Flow
  → 05 技术状态 / 风险 / Mock / 降级
  → 06 DB / 07 API 契约
  → docs/design/* 子系统详细流程 / 状态 / 失败 / 权限 / UI / AI
  → 08 Sprint / Task
  → 09 TC / 验收 / 风险
```

目标能力：

1. 所有 `docs/design/*` 都有统一元信息、职责、边界、阶段、追溯和待确认项结构。
2. 不同类型设计可在同一通用模板下扩展：UI、权限 / 安全、AI/RAG、导入 / 异步任务、数据 / 状态机、高风险愿景。
3. 详细设计明确不新增未授权需求、接口、表、验收目标或 Phase 外能力。
4. 详细设计能承接 06 / 07 契约并输出到 08 / 09。
5. Mock / 降级 / 实现偏差有固定记录位置，避免 Demo 能力被误读为真实能力。

## 4. 建议新增 `ai/doc-standards/design-doc.md`

### 4.1 文档定位

建议新增如下定位：

```markdown
# docs/design/* 详细设计文档标准

> **文档定位**：当某个模块 / 子系统 / 前端入口 / 权限边界 / AI 流程 / 外部集成复杂到无法只靠 04-07 表达时，使用 `docs/design/<subsystem>.md` 记录内部流程、状态、失败处理、权限边界和验证追溯。
>
> **上游输入**：`docs/02-srs.md`、`docs/03-prd.md`、`docs/04-architecture.md`、`docs/05-tech-spec.md`、`docs/06-db-design.md`、`docs/07-api-spec.md`、相关 `docs/env/*` 与技术评估报告。
>
> **下游输出**：约束 `docs/08-dev-plan.md`、`docs/09-verification.md`、代码实现、测试和验收。
```

### 4.2 通用结构建议

```markdown
# 详细设计：<子系统名>（<slug>）

## 0. 文档元信息
## 1. 职责与边界
## 2. 上游依据与追溯
## 3. 核心流程 / 状态机
## 4. 数据、接口与权限契约
## 5. 失败、异常与降级路径
## 6. 阶段增量与实现状态
## 7. 验证与验收追溯
## 8. 与其他子系统交互
## 9. 待人工确认项
```

## 5. 分类 Checklist 建议

### 5.1 UI / 前端交互设计

适用：独立 Web、移动端、小程序、桌面端、复杂表单、多页面流程。

必须补充：

- 页面 / 路由清单。
- 核心用户流。
- 组件职责。
- 加载态、空态、错误态、禁用态、成功反馈。
- 表单校验和文案。
- 权限可见性与后端权限边界。
- 响应式 / 可访问性。
- 浏览器 / 设备范围。
- 验收路径和 TC-ID。

### 5.2 权限 / 安全设计

适用：空间、租户、角色、文档权限、敏感数据、外部共享。

必须补充：

- 角色 × 动作矩阵。
- 资源边界与查询过滤点。
- 越权失败策略。
- 不泄露资源存在性的错误口径。
- 日志 / 审计。
- 负向测试用例。

### 5.3 AI / RAG / 外部模型设计

适用：LLM、Embedding、RAG、OCR、分类、推荐、抽取、推理。

必须补充：

- 输入 / 输出合同。
- Prompt / 检索 / 来源引用合同。
- 模型、Mock、降级状态。
- 安全边界：敏感数据、库外问题、不编造、权限过滤。
- 质量评估指标或样例集。
- 失败 / 超时 / 外部服务不可用策略。

### 5.4 导入 / 异步任务 / 外部集成设计

适用：文件上传、导入、导出、OCR、同步、webhook、长任务。

必须补充：

- 任务状态机。
- 文件 / 输入校验。
- 重试 / 幂等。
- 部分失败与恢复。
- 资源限制。
- 进度显示或轮询策略。
- 失败原因与用户可见提示。

### 5.5 高风险愿景设计

适用：远期 AI 推理、复杂图谱、自动决策、合规高风险能力。

必须补充：

- 明确 `[愿景]` 或后续阶段，不进入当前 Phase。
- Readiness gate：进入 Phase 前置条件。
- Spike / PoC 验证项。
- 技术风险和误判风险。
- 不作为当前验收目标。

## 6. Prompt 增强建议

### 6.1 `generate-docs`

生成文档体系时应：

- 根据 04 模块复杂度和 UI 触发规则判断是否需要 `docs/design/*`。
- 若需要，按通用 design 标准生成骨架。
- 对 UI / 权限 / AI / 导入 / 高风险愿景选择相应 checklist。

### 6.2 `edit-single-doc`

修订 design 文档时应检查：

- 是否有元信息、上游依据、下游影响。
- 是否新增了未授权需求、接口、表、验收目标。
- 是否需要同步 06 / 07 / 08 / 09。
- 是否明确 Mock / 降级和实现偏差。
- 待确认项是否结构化。

### 6.3 `docs-system-audit`

审计 design 文档时应输出：

- 缺失的 design 文档。
- 孤立 design 文档。
- design 与 04 / 06 / 07 / 08 / 09 的断链。
- design 中的越界功能或接口。
- Mock / 降级未标注风险。

### 6.4 `docs-evaluation`

E3 详细设计评估应检查：

- 非平凡子系统是否有 design。
- design 是否足以支撑 Sprint / Task。
- design 是否覆盖失败、权限、状态、数据 / API 契约和验收追溯。

## 7. 兼容与迁移建议

对已存在派生项目，不建议强制重写全部 `docs/design/*`。建议渐进迁移：

1. 先为每个 design 文档补 `0. 文档元信息`。
2. 补职责与边界，明确不新增需求 / 接口 / 表。
3. 为当前 Sprint 相关 design 补流程、失败 / 降级、DB / API / TC 追溯。
4. 对 UI / 权限 / AI / 导入 / 高风险愿景按分类 checklist 补关键缺口。
5. 历史设计与实现不一致时，标“实现偏差 / 待同步”，不得直接写成已实现事实。

## 8. 验收标准

本提案落地后，模板应满足：

1. 新项目需要 `docs/design/*` 时，有通用设计标准可用。
2. `docs/design/*` 默认包含元信息、职责边界、流程 / 状态机、失败 / 降级、数据 / 接口 / 权限契约、阶段增量、验收追溯和待确认项。
3. UI、权限、安全、AI/RAG、导入 / 异步任务、高风险愿景有分类 checklist。
4. design 文档不再孤立定义未同步到 06 / 07 的字段或接口。
5. Mock / 降级 / 实现偏差不会被写成真实能力已完成。
6. 旧项目可按最小迁移步骤补齐当前 Sprint 相关 design，不需要一次性重写全部详细设计。

## 9. 版本影响

建议版本：模板方法论 minor 版本升级。

影响面：

- 对新项目：详细设计生成更一致，后续 Sprint 输入更稳定。
- 对旧项目：同步后不会自动修改项目事实文档；可通过 `docs-system-audit` 或 `edit-single-doc` 按需迁移。
- 对 Prompt：生成、修订、审计和评估 design 文档的检查项增加。
- 对同步清单：如新增 `ai/doc-standards/design-doc.md`，需更新 `template-sync.json`。

## 10. 非目标

- 不要求所有项目都创建 `docs/design/*`；仅在非平凡子系统或触发 UI / 复杂流程时使用。
- 不替代 `docs/04-architecture.md` 的总体架构，也不替代 `docs/06` / `docs/07` 的数据和接口契约。
- 不要求 Lean 项目写冗长 design；Lean 项目可用轻量结构，但不能断追溯。
- 不把派生项目的具体页面、表、接口或业务流程写入模板默认事实。
