# TEMPLATE-UPGRADE: 待人工确认项应包含 AI 建议与依据

> 来源：GitHub issue #83（https://github.com/emily8421/ai-project-template/issues/83）。
> 类型：派生项目回流的模板优化提案。
> 状态：已实施并归档；已从 issue 收件箱转为本地执行稿。
> 建议实施批次：PR-B，建议在文档评估机制之前落地。

## 1. 背景与问题

当前模板鼓励在 `docs/00-09`、`docs/design/*`、项目规则和开发计划中保留“待人工确认项”，以避免 AI 越权替用户做业务、技术、资源或合规决策。

该机制必要，但常见输出偏向纯问题清单，例如只列出“Phase1 存储采用哪种方式”“是否允许真实发送通知”。这能暴露不确定性，却没有充分发挥 AI 辅助分析价值。用户仍需自行判断，后续容易反复确认和返工。

模板需要明确：待人工确认项不应只是问题列表，还应包含 AI 的建议、依据、备选方案和影响分析。AI 不替用户最终拍板，但应提供可审查的推荐口径。

## 2. 设计目标

- 保留“AI 不越权决策”的红线。
- 提升待确认项的可操作性。
- 帮助非专业用户理解选择依据、成本、风险和阻塞关系。
- 让后续 AI 会话可根据“建议 + 依据 + 用户确认”稳定续接。
- 让确认结果更容易回填到 PRD、技术方案、开发计划、任务单和续接文件。

## 3. 建议规则

### 3.1 待确认项最低结构

正式文档和可续接状态文件中的“待人工确认项”应尽量采用结构化表格，而不是纯问题列表。

最低字段：

| 字段 | 说明 |
|---|---|
| 待确认项 | 需要人确认的问题。 |
| AI 建议 | AI 推荐的默认口径或优先选项。 |
| 建议依据 | 来自上游文档、阶段目标、资源约束、风险边界或通用经验的理由。 |
| 备选方案 | 可选的其他路径。 |
| 取舍影响 | 推荐方案与备选方案的成本、风险、阻塞或后续影响。 |
| 阻塞关系 | 是否阻塞当前 Sprint / Phase；不阻塞时说明可延后到哪个阶段。 |

### 3.2 推荐格式

```markdown
## 待人工确认

| ID | 待确认项 | AI 建议 | 建议依据 | 备选方案 | 取舍影响 / 阻塞关系 |
|---|---|---|---|---|---|
| C-001 | Phase1 存储采用哪种方式？ | Sprint-1 先用 JSON / 内存。 | Sprint-1 目标是 API 骨架；当前数据库不是前置条件。 | SQLite、PostgreSQL。 | JSON / 内存最快但非生产；SQLite 更接近持久化；PostgreSQL 最接近目标但引入环境阻塞。 |
```

### 3.3 AI 行为约束

- AI 应给出建议，但必须标明“建议 / 推荐 / 默认口径”，不得伪装成用户已确认事实。
- AI 的建议必须附依据，不得只写“建议选 A”。
- AI 应说明备选方案和取舍影响，避免单一路径诱导。
- 高风险事项必须保留人工确认，不得因给出建议而视为自动确认。
- 若信息不足，AI 可以给“临时建议 + 需补充的信息”，并说明缺失信息如何影响判断。
- 用户确认后，AI 应把确认结果回填到相关文档或续接文件，并移出“待确认”或标记为“已确认”。

## 4. 适用范围

- `docs/00-09` 的待人工确认章节。
- `docs/design/*` 的设计决策与待确认章节。
- `docs/decisions/*` 中 ADR 的待确认后续动作。
- `docs/08-dev-plan.md` 的 Sprint 前置条件与待确认项。
- `tasks/task-*.md` 的阻塞项、禁止事项和验收口径。
- `.ai/session-handoff.md` 中的待确认问题清单。

临时对话不强制使用完整表格；一旦写入正式文档或可续接状态文件，应尽量保留建议与依据。

## 5. 拟改范围

- `ai/global-rules.md`：补充 AI 可提供建议但不得替人确认的边界。
- `ai/document-lifecycle-rules.md`：补充待确认项应包含建议、依据和影响分析。
- `ai/session-rules.md`：续接文件中的待确认项建议包含推荐口径与阻塞关系。
- `ai/doc-standards/`：统一 `00-09` 标准中的“待人工确认”结构，按文档类型提供轻量变体。
- `ai/prompts/docs/00-generate-or-complete-docs.md`：生成 docs 时输出建议、依据、备选方案和影响。
- `ai/prompts/docs/04-edit-single-doc.md`：新增待确认项时补齐建议和依据。
- `ai/prompts/docs/07-sync-docs-from-code.md`：发现未确认决策时给出建议和影响。
- `ai/prompts/review/10-docs-checklist.md`：检查待确认项是否只有问题、缺少建议 / 依据 / 影响。
- `ai/commands/README.md`：补充相关命令对待确认项结构的要求。
- `VERSION` / `CHANGELOG.md`：按实际落地范围更新。

## 6. 版本影响

建议作为 minor 版本落地。该变更会影响文档生成质量、审查标准和 AI 与用户的确认协作方式；不改变目录结构。

## 7. 验收口径

- 新生成的 `docs/00-09` 中，待确认项不是纯问题列表，而包含 AI 建议和依据。
- 文档审查能发现“只列问题但未给建议 / 依据 / 影响”的情况。
- 用户确认某项后，AI 会把该项从待确认迁移到已确认口径 / 当前决策 / 文档事实中。
- 高风险或合规事项即使有 AI 建议，也仍需人工确认。
- 续接文件能记录未确认项的推荐口径、阻塞关系和下一步确认动作。

## 8. 风险与缓解

- **AI 建议被误认为最终决策**：用字段名明确“AI 建议 / 待人工确认”，用户确认后才能回填为事实。
- **文档变长**：允许按重要性裁剪，关键业务、技术、资源和合规项保持完整。
- **建议依据不足**：要求标明依据来源；不足时写“临时建议”和“缺失信息”。
- **过度诱导用户**：必须列备选方案和取舍影响。
- **确认结果散落**：确认后同步更新正式文档或续接文件。

## 10. 实施摘要

- 规则层：补充 AI 可给建议但不得替人确认的边界；待确认项最低结构纳入文档生命周期与续接规则。
- 模板层：`docs/00-09` 的待人工确认项统一为结构化表格。
- Prompt / Command 层：生成、修订、反向同步和审查均要求待确认项包含 AI 建议、依据、备选方案和影响 / 阻塞关系。
- 自检层：增加待确认项字段与规则断言，防止回退为纯问题列表。
