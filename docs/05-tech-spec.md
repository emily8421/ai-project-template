# 05 技术方案

> AI生成初稿，人工确认。明确技术栈版本、关键技术决策、Phase边界对应的技术约束。
> 写法：技术栈 + 关键决策；版本可标“待确认”，不虚构。按 global-rules.md §8 积累式演进。

## 1. 技术栈与版本
（前端 / 后端 / 数据库 / AI 模型 / 关键依赖，均带版本号）

## 2. 关键技术决策
（影响实现方式的取舍，如同步 vs 异步、分层方式、鉴权方式等）

## 3. Phase 技术约束
（当前 Phase 允许 / 禁止的技术，与 ai/project-rules.md §1、§2 一致）

## 4. 运行环境与资源评估
（读取 `docs/env/local-env.md`；说明本机 Demo 可行性、资源瓶颈、降级 / Mock 策略，以及本机不足时需要的公司服务器资源）

## 5. 编码约定
（详见 ai/project-rules.md §5，本文不重复；若初始化阶段该节尚未补齐，则先标注“待 03-09 审核后回填”，不要虚构约定）

## 最小示例（填写粒度参考）

```markdown
## 1. 技术栈与版本
- 运行环境：Node.js 22 LTS（待确认）
- 语言：TypeScript 5.x（待确认）
- 测试：Vitest（待确认）

## 2. 关键技术决策
- Phase1 使用本地同步处理流程，不引入后台任务系统。
- 配置通过环境变量读取，不在代码中硬编码密钥。

## 3. Phase 技术约束
- 允许：标准库、已确认依赖、文件内轻量状态。
- 禁止：新增数据库、消息队列、登录系统。

## 4. 运行环境与资源评估
- 本机 Demo：读取 `docs/env/local-env.md` 后确认可运行最小闭环。
- 资源瓶颈：若模型、数据库或多服务编排超出本机资源，Phase1 使用 Mock / 小数据集 / 远程 API 降级。
- 服务器预案：若完整功能需公司服务器，列出 CPU、内存、GPU/显存、磁盘、端口和权限要求。

## 5. 编码约定
- 待 03-09 审核后回填 `ai/project-rules.md` §5；当前不虚构命名与分层规则。
```
