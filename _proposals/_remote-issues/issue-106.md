﻿# TEMPLATE-UPGRADE: 细化 04-05 架构与技术方案文档规范

> 来源：LUMEN_demo_T2.1（emily8421/LUMEN-DEMO）派生项目回流
> 依据：`docs/research/2026-07-06-template-proposal-audit-batch-2-04-05-architecture-tech.md` Batch 2 审计报告

## 1. 背景与问题

派生项目对 `docs/04-architecture.md`、`docs/05-tech-spec.md` 与 `ai/doc-standards/04-architecture.md`、`ai/doc-standards/05-tech-spec.md` 做 Batch 2 只读审计时发现：当前模板已要求架构目标、组件视图、模块划分、技术栈、资源评估等内容，但规范仍偏骨架，容易出现“有架构图和技术栈，但架构边界、技术状态、风险验证和安全约束不可审计”的问题。

典型问题包括：

1. `04` 容易用一张总图同时表达系统上下文、组件关系和部署拓扑，缺少分层视图。
2. `04` 缺少架构目标、Phase 约束、运行环境、项目形态裁剪和禁止项的集中声明。
3. `04` 模块缺少稳定组件 / 模块 / 流程 ID，导致 `05-09` 难以稳定引用。
4. `04` 关键流程常只写成功路径，失败、降级、权限拒绝、外部服务不可用等路径未被传递到技术方案和验证计划。
5. `04` 缺少 ADR 式架构决策与取舍记录，横切事实容易散落在 `04`、`05`、`ai/project-rules.md` 和 `docs/decisions/*`。
6. `05` 容易列出技术栈和版本，但缺少依赖文件、配置文件、环境变量、密钥 / 凭据管理方式。
7. `05` 对技术环境评估报告、已验证项、未验证项和补做时点的承接不足，容易把环境采集误当成技术可行性验证。
8. `05` 安全、隐私、合规、真实数据、外部 AI、日志和数据留存等内容缺少独立结构。
9. `05` 技术风险、Mock / 降级、资源瓶颈未强制映射到 `docs/09-verification.md` 或 Spike / PoC / Sprint。
10. 技术状态缺少统一枚举，`已用`、`候选`、`预留`、`Mock`、`降级`、`禁止`容易混写。

这些问题不依赖具体业务项目。凡是包含真实运行依赖、数据库、前端、外部服务、AI / OCR / 模型或容器化部署的派生项目，都需要更细的 `04-05` 模板规范。

## 2. 目标

细化 `04-05` 文档规范，使模板能稳定生成和审计以下链路：

```text
03 Phase / REQ / NFR / 约束
  → 04 架构目标 / 组件 / 模块 / 流程 / 部署 / ADR
  → 05 技术栈 / 依赖配置 / 运行资源 / 安全隐私 / 技术风险
  → 06 DB / 07 API / docs/design/* / 08 Sprint / 09 验证
```

目标能力：

1. 架构目标、约束、上下文、组件、模块、流程、部署拓扑可分层阅读。
2. 每个关键模块和流程能追溯到 REQ、Phase 和详细设计文档。
3. 架构取舍有 ADR / 决策表记录，横切事实有权威源。
4. 技术栈有状态、用途、约束来源和版本漂移处理口径。
5. 依赖、配置、环境变量、密钥管理和运行资源有独立章节。
6. 技术环境评估结论能进入 05，并影响 09 验证计划。
7. 安全、隐私、合规、真实数据、外部服务和日志边界可审计。
8. 技术风险、Mock / 降级和资源瓶颈能映射到验证用例、Spike、PoC 或 Sprint。

## 3. 拟修改范围

| `ai/doc-standards/05-tech-spec.md` | 强化依赖配置、技术环境评估、前端交互边界、安全隐私合规、技术风险、状态枚举和版本漂移规则 |

| `ai/prompts/docs/04-edit-single-doc.md` | 修订 04 / 05 时提示同步下游 DB、API、design、08、09 和依赖文件 |

| `ai/prompts/review/20-tech-env-evaluation.md` | 输出项更明确映射到 05 的 `技术环境评估结论` 和 09 的验证计划 |

## 4. 建议的 04 文档规范

### 4.1 结构建议

```markdown
# 04 系统架构（Architecture）

## 0. 文档元信息
## 1. 架构目标与约束
## 2. 系统上下文图
## 3. 容器 / 进程 / 组件视图
## 4. 模块划分
## 5. 关键流程与数据流
## 6. 部署 / 运行拓扑约束
## 7. 架构决策与取舍
## 8. REQ / 功能 → 模块追溯矩阵
## 9. 待人工确认项
```

### 4.2 架构目标与约束

建议模板要求 04 显式承接：

| 约束项 | 内容 | 权威源 | 影响 |
|---|---|---|---|
| 当前 Phase |  | `docs/03-prd.md` / `ai/project-rules.md` | 模块范围与禁止事项 |
| 交付物形态 | Demo / MVP / 产品 | `docs/03-prd.md` | 设计深度与降级口径 |
| 运行环境 |  | `docs/env/local-env.md` | 部署和资源 |
| 项目形态裁剪 |  | `ai/project-rules.md` §3 | 是否需要 DB / API / frontend |
| 禁止项 |  | `ai/project-rules.md` | 防止越界设计 |

### 4.3 系统上下文图

系统上下文图应表达“系统与外界的关系”，不承担内部组件细节。

建议字段：

| 外部参与者 / 系统 | 交互 | 输入 | 输出 | 信任边界 / 风险 |
|---|---|---|---|---|

### 4.4 容器 / 进程 / 组件视图

建议使用稳定组件 ID：

| COMP-ID | 组件 / 进程 | 职责 | 部署位置 | 通信方式 | 阶段 | 状态 | 关联 REQ / NFR |
|---|---|---|---|---|---|---|---|

状态建议：`骨架`、`P1-已设计`、`P1-已实现`、`后续阶段`、`候选·未启用`。

### 4.5 模块划分

模块划分应明确“负责 / 不负责”，并指向详细设计。

| MOD-ID | 模块 | 职责 | 输入 | 输出 | 边界 / 不负责 | 关联设计 | 关联组件 |
|---|---|---|---|---|---|---|---|

要求：

- 非平凡子系统必须指向 `docs/design/<subsystem>.md`。
- UI 型项目必须指向 `docs/design/frontend-interaction.md` 或豁免理由。
- 不得在 04 中新增 `03` 未批准的模块。

### 4.6 关键流程与数据流

建议关键流程使用 `Flow-ID`：

```markdown
### Flow-001：<流程名称>

- 触发：
- 关联 REQ / NFR：
- 涉及组件 / 模块：
- 主要步骤：
- 成功结果：
- 失败 / 降级路径：
- 权限 / 数据边界：
- 下游详细设计：
```

要求：

- 当前 Phase 主流程必须至少覆盖成功路径和关键失败路径。
- 外部服务、AI、OCR、支付、消息、文件解析等不稳定能力必须写降级路径或阻塞口径。
- 权限拒绝、无数据、资源不足、超时等不应只留到实现期再处理。

### 4.7 部署 / 运行拓扑约束

建议表格：

| 环境 | 拓扑 | 端口 / 进程 | 资源要求 | 持久化 | 外部依赖 | Mock / 降级 | 备注 |
|---|---|---|---|---|---|---|---|
| 本机 Demo |  |  |  |  |  |  |  |
| CI / 测试 |  |  |  |  |  |  |  |
| 服务器 / 远程 |  |  |  |  |  |  |  |

### 4.8 架构决策与取舍

建议引入 `ADR-ID` 或 `ARCH-DEC-ID`：

| 决策 ID | 决策 | 理由 | 替代方案 | 影响范围 | 权威源 | 状态 |
|---|---|---|---|---|---|---|

要求：

- 横切事实应引用权威源，如 `ai/project-rules.md`、`docs/decisions/*`、技术评估报告。
- 不应在多个文档中各自重复定义同一事实。
- 决策变更应触发 05-09 和代码影响分析。

### 4.9 REQ / 功能 → 模块追溯矩阵

| REQ / NFR | Phase | COMP-ID | MOD-ID | Flow-ID | 详细设计 | 覆盖状态 |
|---|---|---|---|---|---|---|

## 5. 建议的 05 文档规范

### 5.1 结构建议

```markdown
# 05 技术方案（Technical Specification）

## 0. 文档元信息
## 1. 技术栈与版本
## 2. 关键技术决策
## 3. 依赖与配置
## 4. 运行环境与资源评估
## 4.1 技术环境评估结论（如触发）
## 5. Phase 技术约束
## 5.1 前端交互设计边界（如适用）
## 6. 编码约定
## 7. 安全、隐私与合规考虑
## 8. 技术风险与验证计划
## 9. 待人工确认项
```

### 5.2 技术栈与状态枚举

建议技术栈表增加状态和约束来源：

| 类别 | 技术 / 版本 | 状态 | 用途 | 约束来源 | 替代品禁令 / 备注 |
|---|---|---|---|---|---|

建议状态枚举：

| 状态 | 含义 |
|---|---|
| `已用` | 当前实现或当前 Phase 明确使用 |
| `待接入` | 当前 Phase 计划接入，但尚未实现 |
| `Mock` | 当前以 Mock 方式替代真实能力 |
| `降级` | 当前使用降级路径，不等价真实能力通过 |
| `候选` | 尚未确认，不得作为实现事实 |
| `预留·未启用` | 设计预留或后续阶段能力 |
| `禁止` | 当前 Phase 不允许使用 |

要求：

- 候选、预留、Mock、降级不得写成已用。
- 技术版本与依赖文件不一致时，必须同步修订 05 或依赖文件，并记录验证结果。

### 5.3 关键技术决策

建议使用 `TDR-ID`：

| 决策 ID | 决策 | 理由 | 替代方案 | 影响范围 | 权威源 | 验证状态 |
|---|---|---|---|---|---|---|

### 5.4 依赖与配置

建议表格：

| 项 | 路径 / 名称 | 用途 | 当前状态 | 是否含敏感信息 | 备注 |
|---|---|---|---|---|---|
| 依赖文件 |  |  |  | 否 |  |
| 配置文件 |  |  |  |  |  |
| 环境变量 |  |  |  |  |  |
| 密钥 / 凭据 | 不落文档 |  |  | 是 | 只写管理方式，不写值 |

要求：

- 不得把真实 token、密钥、账号密码写入文档。
- 新增依赖必须进入依赖文件，并说明用途。
- 外部服务配置必须说明 Mock / 降级或缺失时的行为。

### 5.5 运行环境与资源评估

建议表格：

| 运行项 | 本机要求 | 当前本机是否满足 | 验证证据 | 降级 / Mock | 服务器预案 |
|---|---|---|---|---|---|

要求：

- 必须引用 `docs/env/local-env.md` 的事实。
- 涉及真实运行依赖时，必须引用技术环境评估报告，或说明跳过理由、风险和补做时点。
- `collect-env` 只采集事实，不等于技术评估通过。

### 5.6 技术环境评估结论

建议模板要求：

```markdown
### 4.1 技术环境评估结论（如触发）

- 评估报告：`docs/research/YYYY-MM-DD-tech-env-evaluation-<scope>.md` / 暂无（原因：）
- 评估结论：Go / Conditional Go / No-Go / 待评估
- 已验证项：依赖安装 / 导入 / 最小运行 / build / dev server / Docker / 模型加载
- 未验证项与补做时点：
- 对 09 验证计划和依赖文件的影响：
```

### 5.7 Phase 技术约束

| Phase | 允许 | 禁止 | Mock / 降级 | 技术状态说明 | 权威源 |
|---|---|---|---|---|---|

要求：必须与 `ai/project-rules.md` §1、§2、§3 一致。

### 5.8 前端交互设计边界

当项目触发 UI 条件时，05 应明确：

- 是否需要前端交互设计。
- 交互设计文档路径。
- 前端技术栈、运行方式、状态管理、接口调用方式。
- 权限边界：前端隐藏 / 禁用 / 路由守卫只作可见性控制；权限必须由后端接口和服务层执行。
- 05 不替代页面流、空态 / 错误态、文案和验收路径。

### 5.9 安全、隐私与合规考虑

建议表格：

| 项 | 要求 | 阶段 | 状态 | 权威源 | 验证入口 |
|---|---|---|---|---|---|
| 敏感数据 |  |  |  |  |  |
| 外部 AI / 第三方服务 |  |  |  |  |  |
| 密钥 / 凭据 |  |  |  |  |  |
| 日志 / 审计 |  |  |  |  |  |
| 数据留存 / 删除 |  |  |  |  |  |

无已确认要求时，也应写“当前阶段无已确认要求”，不得留空。

### 5.10 技术风险与验证计划

建议使用 `RISK-ID`：

| 风险 ID | 风险 | 影响 | 验证方式 | 对应用例 / 任务 | 当前结论 | 后续动作 |
|---|---|---|---|---|---|---|

要求：

- 高风险技术不得只写在 05，必须映射到 09、Spike、PoC 或 Sprint。
- Mock / 降级必须说明“不等价真实能力通过”的条件。
- No-Go 风险不得进入相关实现 Sprint，除非用户明确接受跳过风险并记录影响。

## 6. Prompt 增强建议

### 6.1 `generate-docs`

生成 04-05 时应要求：

- 04 输出上下文图、组件视图、模块表、Flow-ID、部署拓扑、ADR 表、REQ → 模块追溯。
- 05 输出技术状态、依赖配置、技术环境评估、前端交互边界、安全隐私、风险验证映射。
- 对 Mock / 降级 / 外部服务 / 模型 / Docker / 数据库等真实依赖写明 Go / Conditional Go / No-Go 状态。

### 6.2 `edit-single-doc`

修订 04 或 05 时应提示：

- 是否改变模块、组件、技术栈、依赖或运行拓扑。
- 是否影响 `06`、`07`、`docs/design/*`、`08`、`09`、依赖文件或代码目录。
- 是否触发技术环境评估或安全 / 隐私确认。
- 是否需要同步 `ai/project-rules.md` 的允许 / 禁止项。

### 6.3 `docs-evaluation`

E2 / E3 阶段评估应增加检查：

- 04 是否有目标约束、上下文、组件、模块、流程、部署和 ADR。
- 05 是否区分已用、候选、Mock、降级、预留和禁止技术。
- 真实运行依赖是否已有技术环境评估或跳过记录。
- 安全、隐私、外部服务和真实数据是否有独立口径。
- 技术风险是否映射到 09 或 Spike / PoC。

### 6.4 `tech-env-evaluation`

技术环境评估输出应明确可回填到 05 的字段：

- 评估报告路径。
- Go / Conditional Go / No-Go 结论。
- 已验证项与未验证项。
- 对依赖文件、05、09、首个相关 Sprint 的影响。
- 若用户跳过评估，记录跳过原因、风险、影响范围和补做时点。

## 7. 兼容与迁移建议

对已存在派生项目，不建议强制重写 04-05。建议渐进迁移：

1. 先补 04 的架构目标与约束、部署拓扑和 REQ → 模块追溯。
2. 再补 05 的依赖配置、技术环境评估结论和技术状态枚举。
3. 涉及真实数据、外部 AI、密钥、日志的项目优先补安全隐私合规章节。
4. 将现有风险、Mock / 降级和资源瓶颈映射到 09 验证计划。
5. 若历史实现已偏离 05，先走 `sync-docs-from-code` 或最小文档修订，不得直接把越界实现写成已批准需求。

## 8. 验收标准

本提案落地后，模板应满足：

1. 新项目生成 04 时，能区分上下文图、组件视图、模块划分、关键流程、部署拓扑和 ADR。
2. 04 中每个主要模块 / 流程可追溯到 REQ / NFR，并能指向详细设计。
3. 新项目生成 05 时，能区分已用、候选、预留、Mock、降级和禁止技术。
4. 05 能列出依赖文件、配置文件、环境变量和密钥管理方式，但不泄露真实密钥。
5. 涉及真实运行依赖时，05 能引用技术环境评估结论或记录跳过理由和风险。
6. 安全、隐私、合规、真实数据、外部服务和日志边界不会被遗漏。
7. 技术风险、Mock / 降级、资源瓶颈能映射到 09 验证、Spike、PoC 或 Sprint。
8. 旧项目可按最小迁移步骤补齐，不需要一次性重写全部架构与技术方案。

## 9. 版本影响

建议版本：模板方法论 minor 版本升级。

影响面：

- 对新项目：04-05 文档会更结构化，设计阶段更早暴露环境、依赖、安全和风险问题。
- 对旧项目：同步后不会自动修改项目事实文档；需要通过 `docs-evaluation`、`docs-system-audit` 或 `edit-single-doc` 按需迁移。
