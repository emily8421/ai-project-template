# 04 Architecture Standard（系统架构规范镜像）

> Sync notice: This file is maintained by `ai-project-template` and may be overwritten when a derived project syncs template methodology.
> Do not edit it directly in derived projects; propose reusable changes in `_proposals/` and upstream them to the template repository.

本文件是 `docs/04-architecture.md` 的规范镜像，用于 AI 生成、修订、审计和评估项目架构文档。它不是项目事实文档，派生项目的实际架构事实仍写入 `docs/04-architecture.md`。

## 1. 定位与边界

`04` 承接 `02` / `03` 的 REQ、NFR、Phase、交付物形态、非目标和约束，输出系统边界、架构视图、组件 / 模块职责、关键流程、运行拓扑和架构决策。

- 只说明“系统如何组织”，不写具体依赖版本、表字段、接口字段或 Sprint 实施步骤。
- 不得引入 `00-03` 未授权的需求、角色、外部服务、数据存储或验收目标。
- 必须区分真实接入、已验证、候选、默认关闭、Mock、降级和禁止状态。
- 前端隐藏 / 禁用 / 路由守卫不是权限边界；权限必须由后端接口和服务层执行。

| 系统上下文图 | 外部参与者 / 系统、交互、输入、输出、信任边界 / 风险 |

| 模块划分 | MOD-ID、模块、职责、输入、输出、边界 / 不负责、关联设计、关联组件 |
| 关键流程与数据流 | Flow-ID、成功路径、异常路径、降级路径、权限拒绝、外部服务不可用、关联 API / 数据 / TC |
| 部署 / 运行拓扑约束 | 环境、拓扑、端口 / 进程、资源要求、持久化、外部依赖、Mock / 降级、备注 |
| 架构决策与取舍 | ADR、决策、状态、适用 Phase、备选方案、取舍影响、验证方式 / 证据 |
| REQ / 功能 → 模块追溯 | REQ / NFR、Phase、COMP-ID、MOD-ID、Flow-ID、详细设计、覆盖状态 |

## 3. 架构视图检查表

| 视图 | 必查项 | 通过标准 | 缺失修复 |
|---|---|---|---|
| 系统上下文 | 用户、外部系统、核心服务、边界 | 明确哪些外部系统是真实、Mock、候选或默认关闭 | 补上下文图或边界说明 |
| 组件 / 容器 | 前端、后端、数据、适配层、测试入口 | 每个组件有职责、部署位置、通信方式、阶段、状态和覆盖 REQ | 补组件表或模块图 |
| 关键流程 | 主流程、异常流程、降级流程、权限拒绝 | 流程能关联 API、服务、数据对象、TC 或验收口径 | 补流程图和映射 |
| 运行拓扑 | 本机、服务器、Docker、外部服务 | 明确端口、资源、持久化、可选依赖和降级路径 | 补拓扑和资源约束 |
| ADR / 决策 | 关键技术 / 交付 / 安全决策 | 有状态、理由、备选、取舍影响和验证方式 | 补 ADR 字段 |
| Web App Structure Profile | 复杂 Web / 全栈交互项目的 App Shell、页面 / feature 边界、前后端组件关系 | 说明是否触发 `template-docs/web-fullstack-profile.md`，以及 WSG-001 / WSG-003 如何由架构承接 | 补 App Shell / vertical slice 架构边界或写明豁免 |

## 4. ID 与追溯规则

- 组件使用 `COMP-001`，模块使用 `MOD-001`，关键流程使用 `Flow-001`，架构决策使用 `ADR-001`。
- `REQ / NFR → Phase → COMP-ID → MOD-ID → Flow-ID → design / API / DB / TC` 应可追溯。
- `04` 中出现的外部系统、真实依赖、数据库、AI / OCR / LLM、部署能力或安全边界，必须在 `05` 技术状态、依赖配置或风险矩阵中承接。
- `04` 中标为候选、默认关闭、Mock、降级或禁止的能力，不得在 `08` Sprint 中默认解锁。

## 5. ADR 最低字段

| 字段 | 要求 |
|---|---|
| 决策 ID | 使用 `ADR-001` 等稳定编号 |
| 决策 | 明确选择什么，不写模糊方向 |
| 状态 | 候选 / 已确认 / 默认关闭 / 废弃 / 禁止 |
| 适用 Phase | 指明当前 Phase 或愿景阶段 |
| 理由 | 引用 REQ、NFR、约束、环境评估或人工确认 |
| 替代方案 | 至少说明主要备选或“不适用”理由 |
| 取舍影响 | 对安全、成本、复杂度、验证、扩展性的影响 |
| 验证方式 / 证据 | 命令、Spike、PoC、TC、报告或待补验证 |

## 6. 输出与下游影响

- 给 `05`：技术栈、运行环境、依赖配置、安全隐私和风险验证输入。
- 给 `06` / `07`：数据边界、接口边界、组件与流程约束。
- 给 `docs/design/*`：模块职责、前端交互触发条件、详细设计入口。
- 给 `08` / `09`：Sprint 拆分边界、验收路径、异常 / 降级 / 权限拒绝验证入口。
- 给 Web App Structure Profile：App Shell、组件边界、vertical slice 和前后端协作拓扑输入。
