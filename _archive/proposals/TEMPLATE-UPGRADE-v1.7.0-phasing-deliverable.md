## 1. 动机（两点）

### 1.1 阶段只按「功能范围」划分，缺「交付物形态」维度
当前 `global-rules §8` / `03-prd §3` 阶段划分只按功能（Phase1/Phase2/愿景 = 做哪些 REQ），没声明「做到什么程度」（Demo 可演示 / MVP 可上线 / 产品完整）。导致：
- 「Phase1 完成」含糊（是 Demo 还是产品？）
- 派生项目把 Demo 叫「MVP」，误读为产品可用
- 执行完所有阶段才发现「还有功能/要素未完成」（缺的要素没显式声明，如真实通道/LLM/飞书）

### 1.2 vision → 文档体系生成缺约束，AI 易自由发挥
`INIT-PROMPT.md` 从 vision 生成 00-09 时，缺硬约束（交付物声明、REQ 全覆盖自检、不编造红线、声称据实），AI 可能：
- 漏标交付物形态（Phase 完成是 Demo 还是产品含糊）
- REQ 覆盖不全（某 REQ 无阶段归属，悬空）
- 把 Demo 阶段声称成 MVP/产品
- 过度声称未实现的技术（如 LLM/飞书写「已用」实际代码未调用）

## 2. 拟改（去项目化，落模板通用层）

### 2.1 `global-rules §8` 阶段划分：加「交付物形态」维度
阶段含两个**正交维度**：
- **功能范围**（Phase1/Phase2/愿景）：做哪些 REQ
- **交付物形态**（Demo/MVP/产品）：做到什么程度

每 Phase 必须同时标两维度。如「Phase1 功能完成 · 交付物＝Demo」。

`vision/product-vision.md` 必须声明**最终交付物＝产品**（完整图景），分 **Demo → MVP → 产品** 三阶递进；各阶段交付物定义：
- **Demo**：核心价值可演示（模拟器/原型通道、最简实现保红线）
- **MVP**：可真实上线（真实通道 + 关键生产要素 + 可选增强启用）
- **产品**：全功能生产化（愿景全落地 + 运营级）

### 2.2 `INIT-PROMPT.md`：vision → 文档生成**硬约束**（AI 不得自由发挥）
基于 vision 生成 00-09 时，**必须**：
1. **交付物声明**：03 §3 每 Phase 标交付物形态（Demo/MVP/产品）；00 / vision 头部声明「最终交付物＝产品」+ 当前所处阶段。
2. **REQ 全覆盖**：03 §1 列全 REQ（源自 vision/01），每 REQ 在 §3 有阶段归属；生成后自检「无悬空 REQ」。
3. **不编造红线**（产品红线·永久）：作答必须有知识依据，不编造事实（参数/退换/进度）；Demo 作答＝检索原文（最简保红线），未来可 RAG 限定内 LLM。
4. **阶段双维度**：功能范围 + 交付物形态 同时声明（不得只标功能）。
5. **声称据实**：技术栈/实现状态区分「已用 / 预留·未启用 / 默认关闭」，不得把候选/预留写成已用（核实代码）。

### 2.3 自检机制（文档生成后跑）
- **REQ 覆盖自检**：03 §1 每 REQ → §3 阶段 → 08 Sprint → 09 验收，追溯链完整无悬空。
- **交付物声明自检**：每 Phase 有交付物形态；vision 有最终产品声明。
- **声称 vs 实现自检**：技术栈「已用」项核实代码调用（防 LLM/飞书类过度声称）。

- **现有派生项目**：主动同步 v1.7.0 后，回补 00/03 交付物声明 + 跑 REQ 覆盖自检 + 核实技术栈声称。
- **向后兼容**：不改变既有 docs 骨架编号/结构，只在阶段声明 + 生成约束加维度/自检。
