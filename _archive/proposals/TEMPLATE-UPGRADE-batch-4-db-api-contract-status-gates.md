## 1. 背景与问题

`06-db-design` 与 `07-api-spec` 承接 `02` REQ / NFR、`04` 模块 / Flow、`05` 技术约束和安全边界，是模型、迁移、接口层、前端调用、集成测试和验收用例的契约基线。现有模板已经具备表结构、索引、接口清单、请求响应、错误码、权限和追溯矩阵等基础骨架，但从派生项目回流 issue 看，`06-07` 仍存在以下通用缺口：

1. `06` 有表结构，但缺少数据需求概览、数据对象生命周期、敏感性和数据留存策略。
2. `06` 容易物理表字段先行，概念模型 / ER 图不够明确，实体关系和需求来源不易审计。
3. `06` 字段表缺少必填、默认值、约束、来源、敏感性、留存 / 删除策略等字段级契约。
4. `06` 索引、唯一约束、外键、级联、查询模式、向量 / 全文索引和性能影响未形成统一结构。
5. `06` 对迁移、初始化、seed 数据、回滚、数据修复和环境差异的要求不足。
6. `07` 有接口清单和少量示例，但缺少稳定 API-ID、逐接口请求契约、响应契约、错误契约、权限契约和兼容策略。
7. `07` 对文件上传、异步任务、导入 / OCR、RAG / LLM 等接口缺少状态机、降级 / Mock、外部服务失败和幂等口径。
8. `06` 与 `07` 之间缺少 DB 字段、API 字段、权限、错误码、测试用例的双向追溯。
9. Demo / Mock / 内存仓储与真实数据库、真实接口能力之间的差异缺少记录位，容易被误写成生产能力已具备。
10. 升阶段时缺少 DB / API 契约门槛，目标表结构、当前实现、Mock payload、候选外部接口和稳定对外契约容易混用。

这些问题不依赖具体业务项目。凡是包含持久化数据库、REST API、CLI / SDK / 事件接口、文件上传、异步任务、权限隔离或 AI / 外部服务调用的派生项目，都需要更细的 `06-07` 契约规范。

## 2. 设计目标

Batch 4 目标是把 #107 的“06-07 DB / API 契约文档规范细化”和 #114 的“DB / API 契约状态与升阶段规范”合并为一组可实施的模板优化，确保模板能稳定支撑以下链路：

```text
02 REQ / NFR / 边界场景
  → 04 模块 / 组件 / Flow
  → 05 技术栈 / 鉴权 / 错误码 / 安全 / Mock
  → 06 数据对象 / 概念模型 / 表字段 / 约束 / 迁移 / 留存
  → 07 API-ID / 请求 / 响应 / 错误 / 权限 / 兼容
  → 08 Task / 09 TC / tests / implementation
```

具体目标：

1. 每张表、每个字段、每个接口都能追溯到 REQ / NFR / 约束或明确的架构需要。
2. DB 文档能说明数据对象、生命周期、敏感性、概念模型、迁移、seed、回滚和留存策略。
3. API 文档能按 API-ID 定义当前 Phase 的请求、响应、错误、权限、兼容和测试追溯。
4. DB 与 API 的字段、权限、错误、状态机和测试用例能互相映射。
5. Mock / 降级 / Demo 仓储与真实 DB / API 能力不混写。
6. 当前 Phase 或即将进入 Phase 的表 / API 具备实现所需最低契约；远期目标只保留骨架和状态。
7. DB / API 契约状态能影响 `08` 任务执行、`09` 验证和 Phase 升级判断。

## 4. 建议新增或细化的 06-07 规范

### 4.1 `06-db-design` 最低要求

建议在文档标准中明确 `06` 不只是表结构清单，至少应包含：

| 能力 | 最低字段 / 结构 |
|---|---|
| 文档元信息 | 保留 / 省略决策、决策来源、覆盖 REQ / 模块、当前状态、最后更新 |
| 数据需求概览 | 数据对象、来源 REQ / NFR、用途、生命周期、敏感性、访问边界、下游接口 / 模块 |
| 概念模型 / ER | 实体、描述、关联 REQ / NFR、关系、阶段、备注；必要时补 ER 图 |
| 表清单 | 表名、阶段、状态、用途、来源 REQ / NFR / 约束、数据对象、备注 |
| 表结构字段契约 | 字段、类型、必填、默认值、约束、说明、来源、敏感性、留存 / 删除 |
| 索引、约束与关系 | 表、索引 / 约束、字段、类型、用途、查询 / 写入影响、阶段 |
| 数据迁移与初始化 | 迁移路径、seed / fixture 来源、回滚策略、环境差异、验证入口 |
| 数据安全与留存 | 数据 / 表 / 字段、安全要求、访问控制、脱敏 / 加密、留存 / 删除、外部传输限制、验证入口 |
| 实现状态与 Mock / Demo 差异 | 数据能力、目标设计、当前实现状态、Mock / Demo 差异、是否等价真实能力、补齐时点 |
| REQ → 表 / 字段追溯 | REQ / NFR / 约束、数据对象、表、字段 / 索引 / 约束、覆盖说明、状态 |
| 待人工确认项 | 结构化确认项表，不得只写问题列表 |

### 4.2 DB 目标结构与当前实现对照矩阵

建议在 `06` 标准和评估 Prompt 中增加以下矩阵：

| 数据对象 / 表 | 来源 REQ | 目标存储 | 当前实现 | 契约状态 | 迁移状态 | Seed 状态 | 验证项 |
|---|---|---|---|---|---|---|---|
| conversations | REQ-001 | PostgreSQL 表 | 内存 / JSON / SQLite | Mock / 当前实现 / 目标设计 | 未迁移 / 已迁移 | 有 / 无 | TC-001 / TC-015 |

最低规则：

1. Demo 阶段允许“目标存储”和“当前实现”不同，但必须明示。
2. 若当前实现不是数据库，不应把数据库表写成已迁移。
3. 若进入 MVP / 试点，必须补迁移状态、seed / fixture 来源和验证项。
4. 即使无真实数据库，Mock / fixture 数据来源也应可追溯。

### 4.3 `07-api-spec` 最低要求

建议在文档标准中明确 `07` 不只是接口列表，至少应包含：

| 能力 | 最低字段 / 结构 |
|---|---|
| 文档元信息 | 保留 / 省略决策、接口形态、覆盖 REQ / 模块、当前状态、最后更新 |
| 统一约定 | 认证方式、版本策略、分页 / 排序 / 过滤、错误格式、幂等、时区、编码 |
| 接口 / 命令清单 | API-ID、方法 / 命令、路径 / 名称、用途、阶段、状态、来源 REQ / NFR、关联模块 |
| 请求 / 输入契约 | API-ID、字段 / 参数、位置、类型、必填、校验、示例、说明、来源 |
| 响应 / 输出契约 | API-ID、字段、类型、必填、示例、说明、数据来源 / 表字段、敏感性 |
| 错误码与异常处理 | 错误码 / 退出码、HTTP 状态 / 退出码、触发条件、用户可见信息、客户端处理、日志 / 审计、关联 API-ID |
| 权限、安全与限流 | API-ID、鉴权、空间 / 租户边界、资源权限、敏感字段、限流、审计日志、越权失败策略 |
| 异步任务 / 状态机 | 状态、含义、进入条件、退出条件、用户可见信息、可重试、终态 |
| 兼容性与版本演进 | 当前稳定性、破坏性变更规则、弃用策略、客户端影响、迁移说明 |
| API ↔ DB / Service / Test 追溯 | API-ID、Service / Command、数据来源 / 表、权限规则、错误码、关联测试 / TC、状态 |
| REQ → 接口追溯 | REQ / NFR、API-ID、覆盖说明、阶段、状态 |
| 待人工确认项 | 结构化确认项表，不得把 AI 建议写成确认事实 |

### 4.4 Endpoint contract matrix

建议在当前 Phase 或即将进入 Phase 的 API 中使用 endpoint 级契约矩阵：

| API-ID | 契约状态 | 请求契约 | 响应契约 | 错误契约 | 权限契约 | 兼容性 | 验证项 | 是否可实现 |
|---|---|---|---|---|---|---|---|---|
| API-001 | 草案 / 当前实现 / 已验证 / Mock | 完整 / 缺失 | 完整 / 缺失 | 完整 / 缺失 | 完整 / 缺失 | 内部 / 对外稳定 | TC-... | 是 / 否 / 待确认 |

最低规则：

- 当前 Phase 的 API 必须有稳定 API-ID。
- `草案` / `候选` / `默认关闭` API 不得直接作为开发任务依据，除非已获得人工确认或先做 Spike / PoC。
- 对外接口、跨团队接口、前后端并行接口、处理敏感数据接口必须补齐请求、响应、错误、权限和兼容性契约。
- Mock 接口必须说明与真实外部系统的差异和补齐时点。

## 5. DB / API 交叉契约

建议在 `06`、`07`、`docs-evaluation` 和开发任务 Prompt 中增加交叉检查：

### 5.1 字段映射

| API-ID | 请求 / 响应字段 | 数据来源 | 表 / 字段 | 计算逻辑 | 敏感性 | 脱敏 / 过滤 |
|---|---|---|---|---|---|---|

### 5.2 权限交叉检查

| 权限场景 | DB 过滤 / 约束 | API 校验 | 错误码 | 测试 / TC |
|---|---|---|---|---|

### 5.3 错误与约束映射

| DB / 业务约束 | 触发接口 | 错误码 | 用户可见信息 | 是否可重试 | 日志 / 审计 |
|---|---|---|---|---|---|

最低规则：

- API 输出敏感字段时，必须能追溯到数据来源、脱敏 / 过滤策略和权限规则。
- DB 唯一约束、外键约束、业务约束或迁移失败应映射到 API 错误或运维处理口径。
- 权限隔离不得只依赖前端隐藏 / 禁用，必须在 API / Service / DB 查询边界体现。

## 6. DB / API 契约状态与升阶段门槛

建议在 `06/07` 使用统一状态口径，并引用 Batch 1 横切状态词典：

| 状态 | 含义 | 是否可作为实现依据 |
|---|---|---|
| `目标设计` | 未来方向或完整目标结构 | 不可直接视为当前已实现 |
| `当前实现` | 当前 Phase 实际使用 | 可以作为当前范围实现依据 |
| `Mock` | 仅模拟数据、payload 或接口行为 | 只能作为 Demo / 测试依据 |
| `已验证` | 已经过本机 / 集成 / 契约验证 | 可按验证范围使用 |
| `草案` | 字段 / 请求响应尚未完全冻结 | 需要补齐后再对外稳定 |
| `候选` | 方向性方案，未验证 | 不得进入实现，除非先做 Spike / PoC |
| `默认关闭` | 代码或设计可预留，当前不开启 | 不得在当前阶段默认启用 |
| `禁止` | 当前阶段不得接入 | 实现或测试不得绕过 |

升阶段门槛：

| 触发场景 | 必须补充内容 |
|---|---|
| Demo → MVP | 当前 Phase 表 / API 的状态、字段级契约、endpoint contract、迁移 / seed / 验证项 |
| MVP → 试点 / 对外 | 鉴权、权限、审计、限流、错误可见性、兼容策略、数据留存和回滚 |
| Mock → 真实调用 | 幂等、重试、超时、限流、失败降级、凭据管理、外部服务失败处理 |
| 内部 API → 外部 API | 版本策略、破坏性变更规则、弃用策略、客户端影响和测试证据 |
| 处理真实隐私 / 业务数据 | 敏感字段、脱敏、访问边界、留存 / 删除、审计日志和合规约束 |

## 8. 版本影响

建议作为 MINOR 版本处理。

理由：

- 新增 / 恢复 `ai/doc-standards/06-07` 标准文件，影响 DB / API 文档生成、审计、评估和开发任务门禁。
- 增强字段级契约、endpoint contract matrix、DB / API 交叉契约、契约状态和升阶段门槛。
- 会改变 AI 对 DB / API 草案、Mock、目标设计、当前实现和稳定契约的默认判断方式。

## 9. 验收口径

1. `ai/doc-standards/06-07` 存在，并分别覆盖数据对象、概念模型、字段级契约、迁移 / seed / 回滚、API-ID、请求 / 响应 / 错误 / 权限 / 兼容和追溯矩阵。
2. `docs/06-07` 模板骨架与标准文件保持一致，且不引入项目专属事实。
3. `06` 标准支持目标结构与当前实现对照，避免把未来表写成已实现。
4. `07` 标准支持 endpoint 级 API contract matrix 和状态字段。
5. 文档生命周期规则明确 DB / API 契约如何承接 REQ、NFR、架构模块、技术约束和安全边界。
6. `docs-system-audit` 和 `docs-evaluation` 能发现 Demo 草案、Mock 接口和 MVP 稳定契约混用的问题。
7. `phase-upgrade` 能在升阶段前检查当前 Phase 表 / API 是否满足契约门槛。
8. `run-dev-task` 在 DB / API 相关任务中能检查契约状态、表字段、API-ID、错误码和 TC-ID。
9. 自检脚本能防止关键 06-07 标准、契约状态、endpoint matrix 和 DB / API / TC 追溯规则被误删。
10. 合并后关闭 #107、#114，并在关闭说明中说明 #107 与 #114 的合并处理方式。
