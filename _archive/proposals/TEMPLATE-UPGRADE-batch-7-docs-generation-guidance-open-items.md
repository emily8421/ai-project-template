# TEMPLATE-UPGRADE: 文档体系生成引导、专题讨论场景与待确认事项总览

> Proposal status: Draft for review
> Proposal batch: Batch 7
> Related issues: follow-up from maintainer review of Batch 1-6 proposal coverage
> Source context: user-requested coverage review after Batch 1-6 proposal drafts

## 1. 背景与问题

Batch 1 已提出“文档体系生成总控最低规则”和“待确认事项总览机制”，Batch 2-6 分别细化 `00-09` 与 `docs/design/*` 的文档标准。但在维护者复核 Batch 1-6 覆盖度时发现，仍缺少一组面向使用者的“场景入口 / 引导流程 / 方案讨论 / 总控门禁”规则。

典型缺口包括：

1. 用户说“生成整个文档体系”时，AI 虽应先说明阶段路线和生成模式，但缺少更完整的引导话术、阶段门禁和选择机制。
2. 文档体系生成过程中产生的信息缺失、技术选型待定、交互方案待定、外部依赖待验证等事项，缺少一个自动汇总的总览文档。
3. 待确认事项需要记录提出时间、来源、需在哪个阶段 / 节点前明确、是否阻塞当前阶段、明确后的回填位置；现有提案只定义了最小字段和 `docs-open-items` 候选。
4. 需求层的人机交互讨论、总体设计 / 技术选型讨论、交互设计讨论，目前分散在 `00-05` / `docs/design/*` 标准中，缺少自然语言场景入口。
5. AI 在讨论方案时容易把建议写成已确认事实，或从不足的输入中过度推理；需要强制“多方案 + 依据 + 取舍影响 + 待确认项 + 不落事实”的引导规则。
6. 文档体系是逐步打磨过程，AI 应引导用户按阶段确认、讨论修改、生成文档、评估审计，而不是一次性直接写完并假定正确。

这些问题属于模板级通用能力，不依赖具体业务项目。它们不替代 Batch 2-6 的文档标准，而是为“生成、评估、审计、定稿文档体系”和“专题方案讨论”提供场景入口和总控流程。

## 2. 设计目标

Batch 7 目标是新增或强化面向使用者的文档体系生成与专题讨论场景，使 AI 能像总体架构师助手一样引导需求分析、设计、验证和计划，同时用模板规则约束推理边界。

具体目标：

1. 用户说“生成整个文档体系”时，AI 先说明完整阶段路线，不直接写文件。
2. AI 提供两种模式供用户选择：分阶段确认模式、输入充分后批量生成模式。
3. 每个阶段都有输入、输出、人工确认点、待确认事项门禁和下一阶段进入条件。
4. 自动汇总文档体系生成过程中的未完整事项，形成待确认事项总览。
5. 待确认事项记录提出时间、来源、需确认节点、阻塞关系、AI 建议、依据、备选方案、取舍影响、关闭状态和回填位置。
6. 新增需求层人机交互、总体设计 / 技术选型、交互设计等专题讨论场景。
7. 方案讨论必须给出可选方案、建议方案、依据、适用条件、风险、取舍影响和需要人工确认的问题。
8. AI 建议不得直接写成项目事实；只有人工确认后才能进入 `docs/00-09`、`docs/design/*` 或任务计划。
9. 文档生成完成后必须做完整文档体系评估 / 审计，并输出评估报告或落盘建议。

## 4. 新增场景建议

### 4.1 场景 A：生成整个文档体系

触发语：

- “生成整个文档体系”
- “帮我补齐 00-09”
- “从需求到开发计划都帮我规划”
- “帮我从想法生成完整项目文档”

AI 首次响应必须先说明阶段路线，并让用户选择模式：

```text
生成完整文档体系通常会经历：
1. 输入材料评审
2. 需求确认
3. 需求分析（00-03）
4. 总体设计与技术路线（04-05）
5. 详细设计（06/07/docs/design，按项目形态触发）
6. 验证用例设计（09）
7. 执行计划 / Sprint 规划（08/tasks）
8. 完整文档体系评估与审计
9. 待确认事项总览与编码前门禁

可选模式：
A. 分阶段确认模式（推荐）：每阶段先讨论 / 确认，再生成下一阶段文档。
B. 输入充分后批量生成模式：先评估输入是否足够；信息齐全后批量起草，再做完整评估 / 审计。
```

最低规则：

1. 默认推荐分阶段确认模式。
2. 用户选择批量生成模式前，必须先做输入充分性评估。
3. 输入不足时，AI 应让用户补充，或提供可选方案及影响，不得自行补成事实。
4. 每阶段结束前必须更新待确认事项总览。
5. 完整文档体系生成后，必须执行文档体系评估 / 审计，输出 Go / Conditional Go / No Go 或等效结论。

### 4.2 场景 B：生成 / 更新待确认事项总览

触发语：

- “汇总待确认事项”
- “生成 open items”
- “还有哪些没确认”
- “编码前自检未决项”
- “阶段任务前检查待确认项”

建议路径：

- 默认建议落盘到 `docs/research/YYYY-MM-DD-docs-open-items.md`。
- 若项目希望长期维护固定入口，可选 `docs/open-items.md`，但需在 `docs/README.md` 说明其定位。
- 如果用户暂不允许落盘，则输出会话内总览，并明确“尚未成为正式项目事实”。

建议字段：

| 字段 | 说明 |
|---|---|
| ID | Open Item ID，如 OI-001 |
| 提出时间 | 使用绝对日期 / 时间 |
| 来源 | 输入材料、用户问题、AI 评估、文档章节、任务执行、验证失败等 |
| 待确认事项 | 需要人工判断的问题 |
| AI 建议 | 推荐选项，不得写成已确认事实 |
| 建议依据 | 来源材料、规则、约束、风险、经验判断 |
| 备选方案 | 2-3 个可选方案 |
| 取舍影响 | 成本、复杂度、阶段范围、验证、风险影响 |
| 需确认节点 | 例如生成 03 前、进入 04 前、生成 08 前、编码前、Phase 升级前 |
| 阻塞关系 | 阻塞 / 条件阻塞 / 不阻塞 |
| 回填位置 | 确认后应回填的 `docs/`、`tasks/`、`ai/project-rules.md` 或 README |
| 当前状态 | 待确认 / 已确认 / 暂缓 / 关闭 / 转任务 |
| 关闭依据 | 人工确认记录、文档回填位置、PR / commit / 会议记录 |

门禁规则：

1. 进入设计阶段前，应检查是否存在阻塞 `00-03` 的待确认项。
2. 进入详细设计 / 计划阶段前，应检查技术选型、接口 / DB 契约、交互方案、验证策略是否仍有阻塞项。
3. 编码前必须检查待确认事项总览；阻塞项未关闭或未被明确风险接受，不得开始对应 Sprint。
4. 待确认项确认后，必须回填权威文档；open items 只做总览，不替代事实文档。

### 4.3 场景 C：需求层人机交互讨论

触发语：

- “讨论系统的人机交互”
- “从需求层面给交互建议”
- “用户应该怎么使用这个系统”
- “帮我设计用户操作流程，但先别写正式文档”

适用阶段：`00-03` 生成 / 修订前，或需求仍未确认时。

输出要求：

1. 明确输入依据：用户角色、目标、场景、痛点、约束、非目标。
2. 给出 2-3 个交互方向或用户操作流方案。
3. 每个方案说明适用条件、依据、优点、风险、成本和对需求 / 验收的影响。
4. 给出 AI 推荐方案，但明确“需人工确认后才能写入 01 / 03”。
5. 输出待确认事项，标明需在生成 `01-user-requirements` 或 `03-prd` 前确认。
6. 不得新增未授权角色、业务流程或功能目标；若需要新增，必须列为待确认。

建议输出结构：

```markdown
## 输入依据
## 交互目标与约束
## 方案对比
| 方案 | 描述 | 依据 | 优点 | 风险 / 成本 | 对需求 / 验收影响 |
## AI 建议
## 需要人工确认的问题
## 建议回填位置
```

### 4.4 场景 D：总体设计 / 技术选型讨论

触发语：

- “讨论总体设计”
- “讨论技术选型”
- “给我几个架构方案”
- “这个系统用什么技术路线更好”

适用阶段：`03` 已有初步 Phase / REQ，`04-05` 生成 / 修订前。

输出要求：

1. 明确上游依据：REQ、NFR、Phase、交付物形态、运行环境、资源、团队约束、禁止项。
2. 给出 2-3 个架构 / 技术路线方案。
3. 每个方案说明适用条件、依赖、成本、风险、可验证性、扩展性、维护成本和降级路径。
4. 明确哪些内容需要技术环境评估、Spike / PoC 或 readiness gate。
5. 给出 AI 推荐方案和备选方案，但不得直接写成 `04/05` 的已确认事实。
6. 输出待确认事项，标明需在生成 `04-architecture` / `05-tech-spec` 或进入相关 Sprint 前确认。

建议输出结构：

```markdown
## 上游依据与约束
## 方案对比
| 方案 | 架构 / 技术路线 | 依据 | 优点 | 风险 | 验证方式 | 适用阶段 |
## AI 推荐与理由
## 需要 Spike / PoC / 技术环境评估的事项
## 待确认事项与回填位置
```

### 4.5 场景 E：交互设计方案讨论

触发语：

- “讨论交互设计”
- “给我几个 UI / 页面交互方案”
- “这个页面流程怎么设计更好”
- “先讨论前端交互，不要直接写文档”

适用阶段：`03/04/05/07` 已有基本边界，`docs/design/frontend-interaction.md` 或其他 `docs/design/*interaction*.md` 生成 / 修订前。

输出要求：

1. 明确上游依据：角色、场景、REQ、Phase、API-ID、权限、安全、验收目标。
2. 给出 2-3 个交互方案，说明页面 / 路由 / 用户流 / 状态模型 / 表单 / 空态 / 错误态 / 权限可见性差异。
3. 每个方案说明适用条件、用户体验、实现复杂度、接口依赖、验证成本和风险。
4. 明确前端隐藏 / 禁用 / 路由守卫不是权限边界，权限必须由后端接口和服务层执行。
5. 给出 AI 推荐方案，但需人工确认后才能写入 `docs/design/*`。
6. 输出待确认事项，标明需在生成交互设计或进入前端 Sprint 前确认。

建议输出结构：

```markdown
## 上游依据与约束
## 方案对比
| 方案 | 用户流 / 页面结构 | 依据 | 优点 | 风险 / 成本 | 接口 / 权限依赖 | 验证方式 |
## AI 推荐与理由
## 不得越界事项
## 待确认事项与回填位置
```

## 5. 文档体系生成阶段路线

建议把完整文档体系生成明确为以下阶段：

| 阶段 | 主要任务 | 产出 | 进入下一阶段门禁 |
|---|---|---|---|
| S0 输入材料评审 | 评估输入完整性、来源可信度、缺口 | 输入评审报告 / 缺口清单 | 阻塞输入缺口已补充或风险接受 |
| S1 需求确认 | 明确场景、角色、目标、非目标、待确认问题 | `00-scenario` 草稿 / 确认项 | 关键场景和边界已确认 |
| S2 需求分析 | 形成 U-ID、REQ-ID、NFR、Phase、优先级 | `01/02/03` | 当前 Phase 范围和非目标明确 |
| S3 总体设计 | 形成架构、技术路线、风险和 readiness gate | `04/05` | 技术选型阻塞项已确认或有 Spike / PoC |
| S4 详细设计 | 按触发条件形成 DB / API / design 文档 | `06/07/docs/design/*` | 当前 Sprint 所需契约和设计足够实现 |
| S5 验证设计 | 形成 TC、验收标准、资源 / 环境验证 | `09` | 当前 Phase REQ 有 TC 覆盖 |
| S6 执行计划 | 拆 Sprint / Task，绑定验证包 | `08/tasks` | 编码前待确认阻塞项关闭 |
| S7 完整评估 | 对文档体系做评估 / 审计 | 评估报告 | Go / Conditional Go / No Go 明确 |
| S8 定稿 / 编码前门禁 | 回填确认项、锁定当前阶段 | 更新 docs / open items | 阻塞项关闭或风险接受 |

## 6. 两种生成模式

### 6.1 分阶段确认模式（推荐）

流程：

1. AI 先说明阶段路线和当前阶段目标。
2. 每阶段先讨论方案 / 缺口 / 待确认项。
3. 用户确认后，AI 生成或修订对应文档。
4. 当前阶段完成后，AI 更新待确认事项总览。
5. 当前阶段无阻塞项或已风险接受后，进入下一阶段。
6. 全部文档完成后，AI 做完整文档体系评估 / 审计。

适用：新项目、需求不稳定、技术路线未确认、业务复杂、需要高质量文档体系。

### 6.2 输入充分后批量生成模式

流程：

1. AI 先做输入充分性评估。
2. 若缺信息，AI 输出缺口、建议补充项和可选方案。
3. 用户确认信息足够或接受风险后，AI 按文档规范逐个生成草稿。
4. AI 汇总未完整事项，生成待确认事项总览。
5. AI 对生成的完整文档体系做评估 / 审计。
6. 根据评估报告修订文档，直至达到 Go / Conditional Go。

适用：已有较完整输入、希望快速起草全套文档、允许后续集中修订。

默认建议：向用户同时提供两个场景，但将“分阶段确认模式”标为推荐。

## 8. 版本影响

建议作为 MINOR 版本处理。

理由：

- 新增文档体系生成的场景引导和模式选择，影响 `generate-docs` 默认行为。
- 新增待确认事项总览门禁，会影响阶段转换、编码前检查和文档定稿判断。
- 新增专题讨论场景，会影响 AI 如何从建议过渡到正式文档事实。

## 9. 验收口径

1. 用户说“生成整个文档体系”时，AI 先说明阶段路线和两种模式，不直接写文件。
2. 分阶段确认模式与输入充分后批量生成模式均有清晰流程、适用条件和门禁。
3. 待确认事项总览字段包含提出时间、来源、需确认节点、阻塞关系、AI 建议、依据、备选方案、取舍影响、状态和回填位置。
4. 阶段任务前、编码前、Phase 升级前能检查待确认事项总览中的阻塞项。
5. 新增需求层人机交互、总体设计 / 技术选型、交互设计方案讨论场景。
6. 专题讨论输出必须包含多方案、依据、取舍影响、AI 推荐、待确认项和回填位置。
7. AI 建议在人工确认前不得写成已确认事实。
8. 文档体系生成完成后必须触发完整文档体系评估 / 审计，并输出评估报告或落盘建议。
9. 自检脚本能防止阶段路线、两种生成模式、open items 门禁和专题讨论场景被误删。

## 10. 非目标

- 不重复 Batch 2-6 的具体文档字段标准。
- 不要求所有项目都落盘固定 `docs/open-items.md`；可先使用 `docs/research/YYYY-MM-DD-docs-open-items.md`。
- 不允许 AI 在输入不足时自动补全为事实。
- 不允许专题讨论结果跳过人工确认直接写入正式需求、设计或任务。
- 不把 open items 总览替代 `docs/00-09`、`docs/design/*` 或 `tasks/*` 的事实记录。

## 11. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 场景入口过多导致用户困惑 | 在 `scenario-guides` 中按“生成 / 讨论 / 评估 / 门禁”分类展示 |
| open items 变成新的事实源 | 明确 open items 只做总览，确认后必须回填权威文档 |
| 分阶段确认流程变慢 | 同时提供输入充分后批量生成模式，但要求生成后评估 / 审计 |
| AI 方案建议被误当事实 | 所有专题讨论输出均标“AI 建议，待人工确认”，并列回填位置 |
| 与 Batch 1 重叠 | Batch 1 保留最低规则；Batch 7 补完整场景、字段和门禁 |
