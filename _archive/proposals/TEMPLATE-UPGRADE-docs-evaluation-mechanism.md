# TEMPLATE-UPGRADE: 文档评估机制

> 来源：GitHub issue #85（https://github.com/emily8421/ai-project-template/issues/85）。
> 类型：派生项目回流的模板优化提案。
> 状态：已实施并归档；已从 issue 收件箱转为本地执行稿。
> 建议实施批次：PR-C，建议在“待人工确认项建议与依据”之后落地。

## 1. 背景与问题

模板已有文档生成、文档检查、项目审查和文档体系全链路审计能力，但缺少“正式评估机制”：

- 审计偏发现问题，默认只读输出，不定义评估报告如何沉淀为项目事实留痕。
- 编码前检查偏当前开发就绪，不覆盖愿景、需求、设计之间的阶段转换质量。
- 单文档修改有 Prompt，但缺少单个文档在输入、职责、追溯、下游影响上的独立评估入口。
- 派生项目在文档体系成型后，需要留下“为什么认为当前文档可以进入下一阶段”的评估证据。

因此建议新增“文档评估机制”，用于在关键阶段转换前后生成可落盘、可追踪、可复评的评估报告。

## 2. 设计目标

- 支持整体评估、阶段评估、单文档评估三种粒度。
- 明确评估规则：完整性、合理性、可行性、一致性、追溯性、阶段边界、风险与待确认项。
- 明确报告生成机制：默认先只读评估，用户确认后落盘到标准目录。
- 与 `docs-system-audit` 区分：审计偏全链路回溯发现问题，评估偏阶段判断与正式留痕。
- 补充命令路由、Prompt 和场景指引，使用户能用自然语言触发。

## 3. 评估类型

### 3.1 整体评估

适用于项目文档体系已基本成型，需要判断当前文档体系是否能支撑下一阶段。

评估内容：

- 文档链是否完整。
- 追溯链是否完整。
- 阶段边界是否清楚。
- 横切事实是否一致。
- 风险是否显式。
- 是否可进入下一阶段，并给出 Go / Conditional Go / No Go 结论。

### 3.2 阶段评估

建议内置阶段：

| 评估码 | 阶段转换 | 核心问题 |
|---|---|---|
| E1 | 输入 / 愿景 → 需求阶段 | 输入是否足以生成 00-03；是否有来源锚点与需求编号 |
| E2 | 需求阶段 → 总体设计 | 04-05 是否只承接已授权需求；架构和技术基线是否可行 |
| E3 | 总体设计 → 详细设计 | 06/07/design 是否覆盖关键模块、数据、接口和流程 |
| E4 | 详细设计 → 实现计划 | Sprint 是否可执行、粒度是否合适、禁止事项是否明确 |
| E5 | 实现计划 → 验证 | 09 是否覆盖 REQ 与 Sprint 验收，Mock / 降级边界是否明确 |
| E6 | 实现结果 → 文档回写 | 代码事实是否需要反向同步到 docs |

阶段评估输出：转换结论、合规项、问题项、风险项、修复建议、是否阻塞下一阶段。

### 3.3 单文档评估

适用于修改或评审某一份核心文档前后。

评估内容：

- 输入来源是否明确。
- 输出职责是否符合文档定位。
- 是否引入上游未授权内容。
- 追溯 ID 是否完整且无悬空。
- 阶段标签、交付物形态、状态是否一致。
- 待人工确认项是否真实且包含建议与依据。
- 下游影响是否列明。

## 4. 评估规则

固定评估维度：

| 维度 | 判断问题 | 输出要求 |
|---|---|---|
| 完整性 | 必需文档、章节、矩阵、图表是否存在 | 列缺失项和影响 |
| 合理性 | 需求拆分、阶段归属、优先级是否符合项目目标 | 说明是否需要重分层 |
| 可行性 | 技术、资源、数据、Mock / 降级是否支撑目标 | 给出风险和验证建议 |
| 一致性 | 横切事实是否冲突或重复定义 | 指定权威源和残留位置 |
| 追溯性 | 是否能从输入追到需求、设计、计划、验证 | 列断点和悬空 ID |
| 阶段边界 | 当前阶段是否混入后续功能 | 标出越界内容 |
| 可验证性 | 需求和设计是否有可验收口径 | 标出不可测条目 |
| 维护性 | 文档是否便于后续增量演进 | 标出结构化改进项 |

评估结论：`Go`、`Conditional Go`、`No Go`。

问题优先级：`P0` 阻塞、`P1` 应在相关 Sprint / Phase 前处理、`P2` 后续优化。

## 5. 报告生成机制

默认流程：

1. 读取 `ai/index.md` 及其列出的规则文件。
2. 根据用户意图判断评估粒度：整体、阶段、单文档。
3. 只读扫描相关 docs、doc-standards、project-rules、env、design 文档。
4. 先在对话中输出评估报告草稿。
5. 用户确认需要记录后，将报告写入 `docs/research/`。
6. 若评估发现模板通用缺口，可同步起草 `_proposals/TEMPLATE-UPGRADE-*.md`。

推荐路径：

```text
docs/research/YYYY-MM-DD-docs-evaluation-<scope>.md
```

推荐结构：

```markdown
# 文档评估报告：<scope>

## 1. 评估摘要
## 2. 评估范围与依据
## 3. 整体评估 / 阶段评估 / 单文档评估
## 4. 合规项
## 5. 问题项
## 6. 风险项
## 7. 修复建议
## 8. 是否阻塞下一阶段
## 9. 待人工确认项
```

## 8. 验收口径

- 用户输入“评估愿景到需求阶段文档”，AI 能路由到 `docs-evaluation`。
- 用户输入“评估需求到总体设计”，AI 能输出阶段评估并给 Go / Conditional Go / No Go。
- 用户输入“评估单个 03-prd”，AI 能只读检查该文档的输入、职责、追溯、阶段边界和下游影响。
- 用户要求“记录下来”时，AI 能写入 `docs/research/YYYY-MM-DD-docs-evaluation-<scope>.md`。
- 评估报告不进入 docs 根目录，不覆盖 00-09，不替代正式修订。
- 若发现模板通用问题，AI 能建议起草 `_proposals/TEMPLATE-UPGRADE-*.md`。

## 9. 风险与缓解

- **与 docs-system-audit 重叠**：明确审计偏找问题，评估偏阶段判断和正式留痕。
- **报告泛化过度**：通过三种粒度和 E1-E6 限定评估范围。
- **写入过早**：默认只读输出草稿，用户确认后才落盘。
- **评估替代修订**：评估报告只给判断和建议，修复仍走 `edit-single-doc` 或对应流程。
