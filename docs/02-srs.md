# 02 需求规格说明（SRS）

> **文档定位**：把用户需求转化为系统必须满足的可验证需求。本文回答“系统必须做什么 / 达到什么约束”，不写产品优先级、技术实现、数据库或 API 细节。
>
> **上游输入**：`docs/01-user-requirements.md`、`docs/00-scenario.md`、人工确认的非功能约束。
>
> **下游输出**：为 `docs/03-prd.md`、`docs/04-architecture.md`、`docs/05-tech-spec.md`、`docs/09-verification.md` 提供 REQ-ID。

## 1. 功能需求

**撰写提要**：每条 REQ 必须可验证，并映射到一个或多个 U-ID。当前阶段候选需求要写清可验证口径；远期需求可保留骨架，标“待该阶段细化”。

| REQ-ID | 系统需求 | 来源 U-ID | 可验证口径 | 初步阶段 | 状态 |

| REQ-001 | 系统必须… | U-001 | 给定…当…则… | P1/P2/愿景 | 草稿 / 已确认 |
