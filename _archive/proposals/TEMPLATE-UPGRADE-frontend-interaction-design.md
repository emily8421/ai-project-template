# TEMPLATE-UPGRADE: 前端交互设计文档规则

> 来源：GitHub issue #81（https://github.com/emily8421/ai-project-template/issues/81）与 #86（https://github.com/emily8421/ai-project-template/issues/86）。
> 类型：派生项目回流的模板优化提案。
> 状态：已实施并归档；已将两个高度重叠 issue 合并为本地执行稿。
> 建议实施批次：PR-D，建议在 #82、#83、#85 之后落地。

## 1. 合并说明

#81 与 #86 均围绕“UI 型项目何时需要前端交互详细设计、文档放哪里、如何追溯到 00-09”。

- #81 更完整地描述触发条件、标准结构、页面流 / 状态 / 文案 / 接口依赖 / 验收路径。
- #86 补充了权限可见性、前端隐藏不等于权限边界、UI 文档不得新增需求 / 接口 / 验收目标等约束。

因此不应拆成两个 PR 独立落地；应合并为一个“前端交互设计文档规则”主题，避免重复改同一批规则、Prompt 和审查清单。

## 2. 背景与问题

模板当前已规定：独立 Web / 移动端演示类项目应保留 `frontend/`，并在 `docs/04-architecture.md`、`docs/05-tech-spec.md` 中体现前端架构与技术方案。

但对于真实页面交互项目，模板尚未明确：

- 什么时候必须或建议补充前端交互设计文档。
- 前端交互设计应放在哪个目录，是否占用 `docs/00-09` 编号。
- 页面流、组件、状态、空态、错误态、权限 / 风险文案、接口依赖和验收路径如何记录。
- 审查时如何判断“前端技术方案已写”与“前端交互足以指导开发”的差异。
- 如何防止前端实现把“隐藏入口 / 按钮禁用 / 路由守卫”误写成唯一权限边界。

结果是派生项目容易只写页面名称、技术栈和 API 调用，缺少可指导前端实现和验收的交互细节，导致 AI 编码阶段临场补交互，违反“先设计后实现”和“AI 不可自主扩展需求”。

## 3. 设计目标

- 不增加所有项目的固定文档负担，保持 `docs/00-09` 核心编号稳定。
- 对独立 Web、移动端、小程序、桌面端等 UI 型项目，提供明确触发规则。
- 让前端开发前具备足够设计输入。
- 统一页面结构、用户路径、状态模型、边界文案、接口依赖和验收口径。
- 明确 UI 文档只细化既有需求的界面呈现，不新增需求、接口或验收目标。
- 明确权限必须由后端接口和服务层执行，前端可见性控制不是安全边界。

## 4. 文档定位

前端交互设计是条件性详细设计文档，不新增固定编号，不进入 `docs/` 根目录。

推荐路径：

```text
docs/design/frontend-interaction.md
```

可接受按入口拆分：

```text
docs/design/customer-app-interaction.md
docs/design/admin-console-interaction.md
docs/design/mobile-app-interaction.md
```

不建议路径：

```text
docs/10-ui-design.md
docs/ui-design.md
docs/design-ui.md
```

## 5. 触发条件

满足以下任一条件时，应在开发前补充前端交互设计，或在项目规则 / 技术方案中说明豁免理由：

- 交付形态包含独立 Web、移动端、小程序、桌面端或其他可点击 UI。
- `ai/project-rules.md` §3 保留 `frontend/`，且项目交付物包含多页面交互。
- `docs/08-dev-plan.md` 的 Sprint 修改范围包含前端页面、组件、编辑器、搜索 / 问答 UI、管理页或桌面端集成。
- 存在多页面、多角色、多入口或跨页面流程。
- 存在复杂表单、筛选、列表、详情、审批、状态流或异常处理。
- 验收依赖点击路径、页面状态或用户操作。
- 用户需求、PRD 或愿景中出现“页面、界面、点击、扫码、表单、看板、控制台、手机、H5、Web、App、小程序”等交互信号。

CLI、脚本、后端 API、批处理、纯消息通道内机器人，且无可视化操作界面时，可以不创建该文档。

## 6. 推荐文档结构

```markdown
# 详细设计：前端交互与界面（frontend-interaction）

> 本文承接 03/04/05/07/08/09，只定义既有需求的界面呈现，不新增需求、接口或验收目标。

## 0. 文档元信息
## 1. 职责与边界
## 2. 页面信息架构
## 3. 页面 / 路由清单与 REQ 追溯
## 4. 核心用户流
## 5. 页面结构与组件职责
## 6. 状态、空态与错误态
## 7. 表单、校验与提交反馈
## 8. 文案、风险提示与免责声明
## 9. 权限与可见性原则
## 10. 接口依赖与失败处理
## 11. 响应式、适配范围与可访问性
## 12. 与 API / Sprint / 验证追溯
## 13. 阶段增量
## 14. 待人工确认项
```

## 7. 内容边界

文档应写：

- 页面 / 视图 / 导航结构。
- 关键用户流和交互步骤。
- 加载态、空态、错误态、禁用态、成功态、风险态。
- 表单字段、默认值、校验规则、提交反馈和失败恢复。
- 权限可见性与安全边界说明。
- 页面到 API 的映射、请求时机、关键字段、失败处理、降级策略。
- 与 Sprint、验收用例的追溯矩阵。
- 当前阶段与后续阶段差异。

文档不应写：

- 上游未授权的新功能。
- 未定义接口的请求 / 响应契约。
- 过早锁定的视觉品牌、精细 UI 稿或具体组件库，除非技术方案已批准。
- 把前端隐藏、按钮禁用或路由守卫写成唯一权限边界。

## 8. 与 00-09 的关系

| 关系 | 规则 |
|---|---|
| 与 `03-prd.md` | 只能承接已批准 REQ 与阶段范围 |
| 与 `04-architecture.md` | 承接前端架构、子系统划分和安全边界 |
| 与 `05-tech-spec.md` | 承接前端技术栈、运行环境和依赖边界 |
| 与 `07-api-spec.md` | 只能调用已定义接口；新增接口需先同步 API 文档 |
| 与 `08-dev-plan.md` | 支撑前端相关 Sprint 拆分和修改范围 |
| 与 `09-verification.md` | 将界面操作映射到验收用例 |

若 UI 文档发现 00-09 缺口，应先列为待确认项，不得直接在 UI 文档中补写事实。

## 9. 拟改范围

- `ai/global-rules.md`：在 `frontend/` 启用说明中补充 UI 型项目应按触发条件补充 `docs/design/*interaction*.md`。
- `ai/document-lifecycle-rules.md`：补充详细设计层可包含条件性前端交互设计。
- `ai/project-rules.md`：在项目形态裁剪中说明是否需要前端交互设计。
- `docs/README.md`：说明 UI 交互设计应归入 `docs/design/`，不占用根目录编号。
- `ai/doc-standards/04-architecture.md`：页面型系统应指向交互设计文档。
- `ai/doc-standards/05-tech-spec.md`：区分技术方案与交互设计。
- `ai/prompts/docs/00-generate-or-complete-docs.md`：根据演示形态判断是否需要前端交互设计。
- `ai/prompts/docs/04-edit-single-doc.md`：允许按标准结构新增或补齐 `docs/design/frontend-interaction.md`。
- `ai/prompts/review/10-docs-checklist.md` / `ai/prompts/review/16-docs-system-audit.md` / `ai/prompts/review/03-project-review.md`：增加 UI 型项目检查项。
- `template-docs/scenario-guides.md`：补充触发语和前端 Sprint 前整理页面设计场景。
- `scripts/check-template.sh`：增加关键断言，防止规则 / Prompt / sync 清单遗漏。
- `template-sync.json`：纳入新增或修改的同步文件。
- `VERSION` / `CHANGELOG.md`：按实际落地范围更新。

## 10. 版本影响

建议作为 minor 版本落地。该变更新增条件性建议和文档模板，不破坏既有 `00-09` 编号体系；已有项目无需迁移。

## 11. 验收口径

- 用户输入“补前端交互设计”时，AI 能建议 `docs/design/frontend-interaction.md` 路径。
- 有独立 Web / 移动端 / 桌面端交互的项目，AI 能判断是否需要 UI 详细设计。
- 生成的 UI 文档能追溯到 `03/04/05/07/08/09`，不新增未授权功能。
- `docs-checklist` / `docs-system-audit` 能检查前端交互设计是否缺失或越界。
- 模板仍保持 `docs/` 根目录只放 `README.md` 与 `00-09`。
- 文档明确前端隐藏入口不是权限边界，权限必须由后端接口和服务层执行。

## 12. 风险与缓解

- **文档负担增加**：定义为条件性文档，仅在 UI 复杂度达到触发条件时要求。
- **与 PRD / 技术方案重复**：明确 PRD 写目标和范围，技术方案写实现策略，交互设计写操作路径和状态。
- **过早追求高保真设计**：只要求结构化交互设计，不要求视觉稿或像素级 UI。
- **项目差异过大**：允许统一 `frontend-interaction.md` 或按入口拆分。
- **AI 自行补交互**：要求前端编码前有交互设计或明确豁免理由。
