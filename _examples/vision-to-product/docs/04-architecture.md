# 04 系统架构

> 完整框架按愿景铺开，Phase1 写细；后续阶段保留骨架。

## 1. 整体架构图

```text
用户
  ↓ CLI 命令
cli.py（参数解析与输出）
  ↓
service.py（业务编排：新增 / 搜索 / 详情）
  ├─ summarizer.py（摘要策略：规则摘要 / mock 摘要）
  └─ repository.py（知识条目读写）
        ↓
      SQLite 本地数据库
```

## 2. 模块划分

| 模块 | 阶段 | 状态 | 职责 | 对应功能 |
|---|---|---|---|---|
| CLI 入口 | [P1] | P1-已设计 | 解析 add/search/show 命令，输出用户可读结果 | F-006 |
| 业务服务层 | [P1] | P1-已设计 | 编排新增、搜索、详情查看流程 | F-001/F-003/F-004 |
| 摘要模块 | [P1] | P1-已设计 | 生成简短摘要；Phase1 默认规则摘要或 mock 摘要 | F-002 |
| 数据访问层 | [P1] | P1-已设计 | 封装 SQLite 增查逻辑 | F-005 |
| 标签模块 | [P2] | 骨架 | 为条目添加标签并过滤 | F-101 |
| 语义检索模块 | [愿景] | 骨架 | 支持向量化和相似度召回 | F-201 |

## 3. 技术选型理由

- 选择 CLI：Phase1 聚焦本地单人闭环，CLI 最小、可验证、避免前端和服务部署复杂度。
- 选择 SQLite：满足本地持久化，零服务依赖，便于重启验证。
- 摘要能力先抽象为模块：避免 Phase1 被在线模型、密钥、费用和网络稳定性绑死。
- 暂不引入 ORM：表结构简单，标准库 `sqlite3` 足够，减少依赖与抽象成本。
