# 07 API Spec Standard（接口设计规范镜像）

> Sync notice: This file is maintained by `ai-project-template` and may be overwritten when a derived project syncs template methodology.
> Do not edit it directly in derived projects; propose reusable changes in `_proposals/` and upstream them to the template repository.

本文件是 `docs/07-api-spec.md` 的细粒度标准，用于 AI 生成、修订、审计和评估接口设计文档。它不是项目事实文档，派生项目的实际接口事实仍写入 `docs/07-api-spec.md`。

## 1. 定位与边界

`07` 仅在项目对外暴露 REST API、CLI、SDK、事件、Webhook、文件导入导出、异步任务或跨系统集成契约时保留；无对外接口项目可按 `ai/project-rules.md` §3 省略。

- `07` 承接 `02/03` 的 REQ / NFR / Phase，`04` 的组件、模块、Flow 和信任边界，`05` 的技术栈、鉴权、错误、安全、Mock / 降级和 readiness gate，必要时承接 `06` 的表字段与约束。
- `07` 输出 API-ID、请求 / 响应 / 错误 / 权限 / 兼容契约、异步状态机、API ↔ DB / Service / Test 追溯和 REQ → 接口追溯。
- 不得创建无 REQ、NFR、架构约束或人工确认来源的接口、参数、错误码或事件。
- 不得把草案、候选、默认关闭、Mock 或内部接口写成已验证、已启用或对外稳定契约。

## 3. API-ID 与契约状态

- 当前 Phase 的接口必须有稳定 `API-001` 或项目约定的等价 API-ID；CLI / SDK / 事件接口也应有稳定 ID。
- `草案` / `候选` / `默认关闭` API 不得直接作为开发任务依据，除非已获得人工确认或先做 Spike / PoC。
- 对外接口、跨团队接口、前后端并行接口、处理敏感数据接口必须补齐请求、响应、错误、权限和兼容性契约。
- Mock 接口必须说明与真实外部系统的差异和补齐时点。

| 状态 | API 场景含义 | 是否可作为实现依据 |
|---|---|---|
| 目标设计 | 未来方向或完整目标接口 | 不可直接视为当前已实现 |
| 当前实现 | 当前 Phase 实际可调用接口 / 命令 | 可以作为当前范围实现依据 |
| Mock | 仅模拟 payload、外部服务或接口行为 | 只能作为 Demo / 测试依据 |
| 已验证 | 已通过契约、集成、E2E 或验收验证 | 可按验证范围使用 |
| 草案 | 请求 / 响应 / 错误 / 权限尚未冻结 | 需要补齐后再对外稳定 |
| 候选 | 方向性接口或外部 API，未验证 | 不得进入实现，除非先做 Spike / PoC |
| 默认关闭 | 接口存在或预留，当前不开启 | 不得在当前阶段默认启用 |
| 禁止 | 当前阶段不得接入或暴露的接口 | 实现或测试不得绕过 |

## 5. DB / Service / Test 交叉追溯

| API-ID | Service / Command | 数据来源 / 表 | 权限规则 | 错误码 | 关联测试 / TC | 状态 |
|---|---|---|---|---|---|---|

| 权限场景 | DB 过滤 / 约束 | API 校验 | 错误码 | 测试 / TC |
|---|---|---|---|---|

最低规则：

- API 输出敏感字段时，必须能追溯到数据来源、脱敏 / 过滤策略和权限规则。
- DB 唯一约束、外键约束、业务约束或迁移失败应映射到 API 错误或运维处理口径。
- 权限隔离不得只依赖前端隐藏 / 禁用，必须在 API / Service / DB 查询边界体现。

## 6. 升阶段触发条件

| 触发场景 | 必须补充内容 |
|---|---|
| Demo → MVP | 当前 Phase API 的契约状态、请求 / 响应 / 错误 / 权限契约、验证项 |
| MVP → 试点 / 对外 | 鉴权、权限、审计、限流、错误可见性、兼容策略、数据留存和回滚 |
| Mock → 真实调用 | 幂等、重试、超时、限流、失败降级、凭据管理、外部服务失败处理 |
| 内部 API → 外部 API | 版本策略、破坏性变更规则、弃用策略、客户端影响和测试证据 |
| 处理真实隐私 / 业务数据 | 敏感字段、脱敏、访问边界、留存 / 删除、审计日志和合规约束 |

## 7. 审计检查

- 当前 Phase API 是否都有稳定 API-ID、契约状态和 REQ / NFR / 模块来源。
- 对外、跨团队、前后端并行或敏感数据接口是否补齐 endpoint contract matrix。
- 请求、响应、错误、权限、兼容性和异步状态机是否足以进入实现。
- Mock、草案、候选、默认关闭接口是否被误写成当前实现、已验证或对外稳定。
- API 是否与 `06`、Service、`08`、`09` 建立字段、错误码、权限、TC-ID 和 Sprint 映射。

## 8. 下游影响

- 给实现：API / CLI / SDK / 事件处理层、Service、权限检查、错误处理和客户端调用边界。
- 给 `06`：字段来源、敏感字段、约束错误、权限过滤和数据访问路径。
- 给 `08` / `09`：API 相关 Sprint 前置条件、契约测试、集成测试、验收测试和 Phase 升级门槛。
